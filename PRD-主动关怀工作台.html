致 学工领导

这个系统不是"预测谁会出事"，是把全校散落在各系统里的学生异常信号集中起来，告诉辅导员"今天该主动联系哪几个学生、聊什么、怎么记录"。所有判断基于明确规则、可解释、可追溯。AI 不做诊断、不下结论、不替学校决定。学校能拿到的是：每周多少次主动关怀、多少次形成闭环、哪些规则有效、哪些误报多。

致 辅导员

每天上班打开它，看到不超过 5 个人的"今日关怀清单"。每个人下面写清楚：为什么进入清单、建议聊什么、不建议碰什么。聊完一键归档，下次不再骚扰你。它不会推 50 条 alert 让你自己挑，也不会给学生打风险分让你尴尬。

致 产品 / 研发

核心动作是把传统"异常预警"升级为"主动关怀任务系统"：alert 不是通知，是带建议动作、建议话术、建议时限的任务；AI 只做解释和提纲、不参与判定；通知按严重度分层扇出避免骚扰；全流程留痕、可审计、可申诉、可调优。

★ 北极星指标
每位辅导员每周完成 ≥ 5 次系统推荐的主动关怀，且其中 ≥ 80% 形成闭环记录。

一、为什么要做这件事

1.1 学工现场的三个真实痛点

痛点 A · 信息散、看不全

学生的异常信号散落在请假、考勤、违纪、成绩、欠费、宿舍、心理咨询等 7-10 个系统里。辅导员要主动跨系统查，绝大部分人不会查、没时间查、也查不全。结果是：出事之前总有信号，出事之后才被串联。

痛点 B · 系统会推，但推得没人看

很多学校上过"预警系统"，最后都死在两件事上：

  • 一上线就推 80 条 alert，辅导员不知道哪条要紧
  • 推送之后，辅导员不知道下一步做什么——电话？家长？心理？

结果：alert 变成另一个"未读红点"。

痛点 C · 责任不清、不敢用

如果系统说"该生高风险"，辅导员就要担责：

  • 我去找了、学生有想法、出事算谁的？
  • 我没去找、系统报过、出事算谁的？

模糊定性让辅导员拒绝使用。

1.2 同类产品的失败模式 & 我们的对策

失败模式表现我们怎么避免
黑箱评分"该生 87 分高危"，没人知道为什么不做总分，每条线索单独存在、单独可解释
大杂烩规则200 条规则混在一个池子，运维瘫痪规则按 风险类型 + 严重度 两层归类
AI 自动定性模型说"疑似抑郁"直接推AI 只做解释和提纲，不参与判定
推一堆没人管推送数 ≠ 关怀数北极星指标是 完成关怀次数，不是触发数
给学生贴标签"重点关注学生"档案永久标记状态附在 alert 上，不附在学生上

二、产品定位

2.1 一句话

主动关怀工作台是一套"学生异常线索发现 + 主动关怀建议 + 闭环跟进留痕"的辅导员工具。

2.2 我们做什么、不做什么

✓ 做

  • 全校系统行为标准化、汇总到事件中心
  • 可解释的规则识别异常组合，生成"关怀任务"
  • 给每个任务配 AI 解释 + 沟通提纲 + 建议时限
  • 全程留痕：谁看、谁接、聊什么、何时关闭
  • 给学校管理者一份"主动关怀工作汇总"

✗ 不做

  • 不做学生风险总分排名
  • 不做学生"高风险/低风险"标签
  • 不让 AI 替学校做决定（联系家长、转介、上报）
  • 不直接告诉学生"你被预警了"
  • 不做心理诊断（路由到心理中心）

三、五条核心原则（不可妥协）

原则 1 · 一切预警必须可解释

每条任务必须能回答四问：

  • 这是哪个学生？
  • 为什么进入清单？（哪些事件、什么时间、什么阈值）
  • 建议怎么办？（动作、话题、时限）
  • 谁负责跟？（指派人 + SLA）

任何不可解释的输出禁止上线。

原则 2 · 不做评分排名、不贴标签

  • 不出现"风险分 87"这种数字
  • 不出现"重点关注学生"这种永久标签
  • 学生档案上 只展示当前未关闭的关怀线索，关闭后不留分级痕迹（仅保留事件本身）

原则 3 · AI 不参与判定

AI 可以

  • 解释一条规则为什么触发
  • 基于学生画像给沟通提纲
  • 整理谈话录音 / 文字成结构化记录
  • 摘要群体趋势给领导

AI 不能

  • 下结论说"该生有风险"
  • 自动外发任何通知给学生 / 家长
  • 参与任务的"接 / 拒 / 改期"决策

原则 4 · 一切预警必须可闭环

不接受"提示完就结束"。每条任务必须有终点：

  • 已完成（聊了、记了）
  • 已拒绝（说明原因）
  • 已改期（再给一次机会）
  • 已转介（说明转给谁）

未闭环的任务进入"超期池"，自动升级到院系领导视图。

原则 5 · 敏感数据最小化

数据类型用法
学籍、成绩、考勤、请假、违纪可用于规则判断、可喂给 AI
资助、欠费可用于规则判断，不喂给 AI
心理咨询、医疗、家庭信息不参与规则、不喂给 AI，仅心理中心专用模块使用
自由文本（谈话内容、留言）命中关键词时仅路由到心理中心，不进辅导员视图

六、核心功能模块

6.1 风险事件中心

职责：把分散在各业务系统的学生行为，标准化为"事件流"，作为整个系统的数据飞轮起点。

6.1.1 数据飞轮总览

┌──────────────────────────────────────────────────────────────────┐ │ 数据飞轮闭环 │ │ │ │ 业务模块行为 │ │ ↓ (Publisher 写入，失败吞掉不阻塞业务) │ │ student_event_log ←──────────────────┐ │ │ ↓ │ │ │ 规则引擎评估 │ 反馈回写 │ │ ↓ │ (调阈值 / 标失效) │ │ care_task（带 AI 解释 + SLA） │ │ │ ↓ │ │ │ 辅导员处理 + care_task_feedback ──────┘ │ │ ↓ │ │ 规则运维：误报率 / 拒绝原因聚合 → 自动建议调整 │ └──────────────────────────────────────────────────────────────────┘
飞轮的关键设计点
  • 业务无感知：事件写入失败不影响业务事务（log warn 后吞掉），事件流是观测层不是业务层
  • append-only：student_event_log 永不更新永不删除，是审计与回溯的真相源
  • 反馈双向：care_task_feedback 把辅导员的处置结果回写，规则运维侧据此调阈值
  • 脱敏存储：event_data jsonb 只存结构化字段，不存姓名 / 身份证 / 自由文本

6.1.3 事件目录 + 字段字典

  • 命名规范：<domain>_<verb>[_<modifier>]，全小写、下划线分词
  • 字段稳定性：已发布事件的字段名 / 类型禁止修改，新增字段须向后兼容
  • 禁止直接写未注册字段（规则引擎依赖字段字典做表达式校验）

6.1.4 事件存储

  • 主表：student_event_log（tenant_id / student_id / event_type / event_source / event_data jsonb / severity / occurred_at / created_at）
  • 4 个核心索引：(tenant_id, student_id) · (event_type) · (occurred_at DESC) · (student_id, occurred_at DESC)
  • 严重度部分索引：(severity DESC, occurred_at DESC) WHERE severity >= 5 加速预警引擎扫描
  • 保留期：在校期间全量保留；毕业 / 离校后归档到只读区

产品方（朝夕团队）编辑阈值 / 改 prompt / 加新规则 / shadow 灰度 / 季度发版—
学校（学工部 / 信息中心）每条规则开 / 关 + 全局严重度偏移 + 看 read-only 报表不能改阈值、SLA、冷却、scope 等任何参数
辅导员接 / 改期 / 拒绝任务 + 反馈"误报"原因不能修改规则
为什么这样改
  • 阈值是产品决策，不是配置数据——学校乱调比不调更糟
  • 每所学校半年后没人维护是同类产品的全部死法（评审老陈警告）
  • 规则集统一升级，跨校反馈反哺产品迭代，飞轮才真正转起来

类型用法例子

6.2.5 反馈飞轮回流机制（v1.2 新增）

v1.1 把规则收回产品方，但没说反馈怎么从学校 SaaS 回到产品方——这是评审老陈警告的"飞轮断了一半"。v1.2 补这个机制。

当前架构难点

  • 每个 tenant 的 PG schema 物理隔离，care_task_feedback 数据存在学校自己的 schema 里
  • 产品方默认看不到学校数据（合规设计）
  • 没有汇集机制，"跨校反馈反哺规则迭代"是空话

回流设计

环节设计
匿名化字段导出前删除：student_id / counselor_id / tenant_id（用哈希 + per-batch salt 不可逆）/ 任何自由文本（note / reason_detail）
保留字段rule_id / rule_version / severity / feedback_type / reason_code / time_to_close_hours / 匿名 tenant 哈希
学校 opt-in 开关部署时默认开（合同条款约定），学校可在管理后台关闭
回流方式每周一 02:00 由 tenant 侧定时任务批量推送到产品方 feedback_warehouse API（端到端 TLS + 服务端再哈希）
产品方落地独立 ClickHouse 实例 feedback_warehouse，不与任何 tenant 业务库连通；只有规则维护小组可读
合规授权部署合同明确条款：① 数据用途仅限规则迭代 ② 不与任何第三方共享 ③ 学校随时可关闭 ④ 季度发布"匿名汇总报告"给学校查阅

反馈如何反哺规则

  • 规则维护小组月度查 ClickHouse：哪条规则跨校误报率 > 20%？哪条规则全网无人接单？
  • 季度发版调整规则阈值 / 删除失效规则 / 新增规则
  • 调整理由 + 数据支撑写入产品方内部 xg-platform/alert/rules/CHANGELOG.md

学校可见性

  • 学校管理后台展示"当前已 opt-in 反馈飞轮"开关 + 链接到匿名化字段说明文档
  • 学校可随时关闭，关闭后不影响业务运行，仅影响产品迭代速度
  • 产品方季度发布"规则集变更说明"，公开告诉所有学校"上季度基于跨校反馈调整了 X / Y 阈值"

为什么这件事必须 P1 就有：反馈飞轮不闭环 = 系统部署上线第一天就开始劣化，半年后规则失效却没人知道。比"等学校自己管"风险更可控：至少产品方掌握真相，学校可知。

  • AI 调用超时 5s → 进入"重试一次"，再失败即降级

项设计

展示（v1.2 收口）① 当前未关闭关怀的类型摘要（学业 / 行为 / 生活）
② 全部关闭时显示"目前无主动关心记录"
③ 每条未关闭记录展示"老师建议近期联系"，无具体规则、无具体细节
不展示历史总次数、规则名称、严重度、AI 解释、辅导员姓名、判断词、被下钻信息

数据范围仅自己；后端 scope 强制按 student_id = current_user.id 过滤

9.4.2 为什么 v1.2 改这套（评审王教授）

v1.1 展示"近 6 个月 8 次 / 3 闭环" 看起来是合规知情，但数字本身是判断：

  • 8 次 vs 上次的 3 次 → 学生焦虑"是我做错什么了"
  • 8 次 - 3 闭环 = 5 个老师在背后议论但没找我 → 学生愤怒
  • 跟室友比"你 0 我 8" → 学生心理压力

v1.2 的解决方案：不展示历史数字 / 只展示当前未关闭 / 类型摘要 / 入口深一级。

9.4.3 仍保留的合规边界

  • 学生可知：自己当前是否被列入关心、属于哪类
  • 学生不可知：具体规则、被关心了几次、谁在关心、AI 怎么分析的
  • 完整申诉流程P2 再做

本节立场：教育部《高等学校学生个人信息保护管理办法》要求"学生有权知晓涉及其个人信息的处理情况"。知晓 ≠ 全量披露，给出"当前状态 + 类型"已满足知情权，无需展示评估细节。

9.5 领导下钻个体（v1.2 修订：按角色分级）

v1.2 修正 N=10 一刀切——按角色实际工作量分级。原一刀切对副书记不够（值班 5 + 家长来访 3 + 月度抽查 5 + 突发 5 ≈ 18）。

12.1 任务卡（核心组件）

信息层级（从上到下）

  • 严重度色：紧急 #cf1322 / 高 #fa8c16 / 关注 #faad14 / 提醒 #52c41a
  • AI 区域必须有"AI 助手 · 小夕"标识 + 一句免责声明

12.2 拒绝 / 改期 / 转介对话框

  • 拒绝：必选原因（5 选 1）+ 选填说明 → 写 care_task_feedback
  • 改期：选 1 / 3 / 7 天，自动延 due_date，次数+1
  • 转介：选目标部门（资助办 / 心理中心 / 教务办 / 保卫处），必填说明

12.3 辅导员工作台首页

信息密度

  • 首屏 5 个任务卡
  • 顶部一句话状态（"今日 5 项 · 紧急 1 · 关注 2"）
  • 切换栏：今日 / 进行中 / 已完成 / 超期

避免设计

  • 50 行 alert 列表表格
  • 任何"风险评分"数字柱状图
  • 学生头像 + 红绿灯标签

12.4 院系领导工作台

首屏元素

  • 一句话本周汇总
  • 需要介入的事项（最多 3 条）
  • 本周触发最多的规则 top 3

不出现

  • 个体学生的详细档案入口（除非超期跳转）
