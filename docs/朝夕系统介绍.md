# 第一章 项目定位与整体架构

## 1.1 产品定位

朝夕面向 **高等院校学生工作场景**，是一个 AI 原生的服务平台。"学生工作"覆盖辅导员、院系领导、学工部、校管理员、学生本人在日常事务里需要协作的所有动作 —— 请假、签到、信息收集、通知、违纪、勤工助学、画像与预警等。系统的设计原则：

- **AI 原生（不是事后加挂）**：从第一天起把 AI 助手"小夕"做成一等交互入口。学生说"我下周一请病假三天"、辅导员说"博雅书院的男生有哪些"、校管理员说"事假改成最多 5 天"，都能识别意图、跳到对应表单、自动填写、调起审批 —— 而不是先打开页面再手填。
- **AI 可降级**：把每一处 AI 能力包在 `try/catch` 里。哪怕大模型挂了、配额用完、网络不通，传统界面照常工作。学校不会因为接入了 AI 而失去原有的可靠性。
- **平台化（强主干、弱扩展）**：底座抽象出来 —— 用户/权限/租户/工作流/通知/文件 —— 业务模块按统一约定接入。新增模块（如 P1 的心理健康、P2 的就业）不会改动底座。
- **轻量优先**：能用 PostgreSQL 一库解决的，不引入新组件。所以 P0 只有 3 个数据组件（PG、Redis、MinIO），向量检索、全文搜索、JSONB 文档、审计日志都跑在 PG 上。
- **多端触达**：Web 管理端 + 微信小程序 + 企业微信。同一人在两端的身份是绑定的，不同角色在不同端按场景看到不同功能集 —— 例如学生在微信小程序里申请请假，辅导员在 Web 端审批，企业微信收推送摘要。

## 1.2 阶段策略

朝夕按三阶段交付：

| 阶段 | 时间窗 | 交付目标 | 架构形态 |
|------|--------|---------|---------|
| **P0** | 0–6 月 | 平台底座 + 10 个业务模块上线，1–2 所试点学校验证 | Java 单体 + Python AI sidecar，3 个数据组件 |
| **P1** | 6–12 月 | 扩展至 5–10 所学校，补 3–4 个模块 | 按需拆出工作流引擎为独立服务；引入 RabbitMQ、可视化流程设计器 |
| **P2** | 12–18 月 | 全模块上线、规模化推广、信创适配 | 按需引入 ClickHouse / Milvus / ES / API 网关 |

> **P0 关键决策**：工作流引擎内嵌在 Java 单体里。不是不能拆，是 P0 阶段拆收益负 —— 单库事务、单一部署、无服务边界协议成本。DSL 是稳定契约；未来若需独立演进，引擎实现可换语言/框架而上层零感知。

本文档描述的是 **P0 阶段已交付的状态**，不涉及 P1/P2 的承诺。

### 架构原则（与代码状态严格对齐）

1. **单体优先** — Java 一个进程包全部业务 + 工作流引擎；运维只看一个 jar。
2. **可拆不先拆** — 工作流引擎作为内嵌模块，与 Java 业务靠接口通信；要拆出去时不动 DSL。
3. **AI 与业务解耦** — Python sidecar 是 *外加* 的，Java 不依赖它。sidecar 挂掉系统照样跑。
4. **业务即 Tool（受控暴露）** — 不是所有 CRUD 都给 Agent。只开放 30–40 个 *用户任务级* Tool（"申请请假"是 Tool，"INSERT leave_request"不是）。
5. **轻量优先** — PG 一库多用，不为理论需求预引入重型组件。

## 1.6 跨模块基础设施（P0 平台底座）

| 设施 | 作用 | 存放位置 |
|------|------|---------|
| **多租户路由** | TenantContext + MyBatis-Plus 多租户插件，自动 schema 切换 | `xg-common/tenant/` |
| **学生事件流** | 13 种事件类型（请假提交 / 违纪记录 / 投诉 / 签到迟到 / 通知确认 等），跨模块共享底账 | `xg-platform/event/` |
| **预警引擎** | 5 条内置规则扫描事件流（请假频繁 / 近期违纪 / 投诉偏高 / 迟到模式 / 多模块异常），生成 student_alert | `xg-business/alert/` |
| **通知中心** | INSERT 模板 seed → `NotificationOrchestrator.send(...)` 自动扇出三渠道（站内信 / 小程序订阅 / 企业微信），不允许业务侧绕过 | `xg-business/notification/` |
| **审计日志** | 关键操作（审批、配置改动）落地 audit_log | `xg-platform/audit/` |
| **文件服务** | MinIO 上传下载，预签名 URL | `xg-platform/file/` |
| **AI Sidecar Client** | Java 调 Python sidecar 的封装，超时降级 | `xg-platform/insight/client/` |

> **设计约束（编码时遵守）**：业务模块发通知 = (1) 落一行 `notification_template` seed (2) 调 `NotificationOrchestrator.send(code, sourceType, sourceId, recipients, vars)`。通知中心 UI / AI 助手识别 / 角色偏好 / 双轨去重 / 三渠道扇出 全部自动适配。**禁止业务侧直接 `NotificationService.send`**，否则去重、AI 引用、用户偏好都失效。

# 第二章 工作台与校园页（角色定制入口）

## 2.1 定位

朝夕的"主入口"在两个端：

- **Web 管理端**：进系统第一眼是「工作台」(`/workspace`)。这里 **按角色动态分发** —— 同一个 URL，学生看到的、辅导员看到的、院系领导看到的内容完全不一样。
- **微信小程序**：进入第一眼是「校园」首页 —— 学生看"今日简报 + 我的请假 + 通知"集成视图，教职工看"待审批 + 班级/学院异常学生 + AI 观察员"。

两端的设计共识：**不是给所有人塞同一份信息，而是按角色把"今天最该看到的 5–7 块"放上来**。

## 2.2 角色分发机制

进入 `/workspace` 时，前端按 `useAuth()` 拿到的角色分发组件：

```tsx
// pages/workspace/index.tsx (节选)
if (isStudent) return <StudentWorkspace />;
if (isDean)    return <DeanWorkspace />;
return <CounselorWorkspace />;   // 默认: 辅导员视角
```

> **设计取舍**：没有做"100% 个性化拖拽工作台"那种大件 —— 那玩意学校管不动，每个老师都不会用心搭。换成 **"角色定型 + 数据驱动"**：固定几套布局，进哪一套由角色决定，里面的数字 / 列表来自后端。维护成本和老师培训成本都更低。

P0 阶段已建：`StudentWorkspace` / `CounselorWorkspace` / `DeanWorkspace` 三套主视图。`SchoolAdminWorkspace` / `EmployerWorkspace` 沿用基础卡片组件按需补充。

### 2.3.2 「今日简报」严格规则化

P0 阶段「今日简报」永远 *不走 LLM* —— 它是规则化的数据快照。理由：

- 学生天天看，必须 *秒级* 加载、*稳定文案*、*零幻觉*。
- LLM 调用花成本与延迟。规则化路径每一项都查现成接口，加起来 < 200ms。

### 2.6.1 学院维度统计原则

不直接 `WHERE college=?`，而是走 **org_unit closure table** —— 一次 JOIN 完成"该学院下所有班级所有学生"。Closure 表让任意层级的组织树查询走索引、O(1) 复杂度。

# 第三章 跨模块基础设施

> 本章讲跨业务模块共用的底座 —— 工作流引擎 / 通知中心 / AI 操作能力 / 字段目录。其余业务章（请销假、勤工助学、学生信息库）都以"调这些底座"的姿态接入，不重复定义。

## 3.1 工作流引擎

朝夕的工作流引擎是 **自建轻量状态机**，承载请销假、销假、勤工岗位申请、未来的违纪审批等。不引入 Camunda / Activiti 是为了保证表达式 / DSL 完全在我们可控范围内（多租户 + 安全约束 + 受限语法）。

### 3.1.1 5 类节点

DSL 是 YAML/JSON，每张 workflow 由若干节点串成 DAG：

| 节点类型 | 作用 | 典型用法 |
|---------|------|---------|
| `form_submit` | 学生 / 用户填一份表单 | 请假申请、销假申请、勤工岗位申请 |
| `approval` | 进入待审批，分配给某个角色 / 用户 | 辅导员 / 书记 / 学工部审批 |
| `condition` | 表达式分支 | 请假天数 ≤ 3 天走单档，> 3 天走双档 |
| `notification` | 触发一条通知（调 NotificationOrchestrator） | 审批通过后通知申请人 |
| `end` | 结束节点 | 同意通过、驳回结案 |

### 3.1.2 受限表达式（mini-language）

`condition` 节点的表达式不是通用脚本，是 **受限 mini-language** —— 可控、可静态校验：

| 函数 | 作用 |
|------|------|
| `duration_days(start, end)` | 算两个日期之间的天数 |
| `date_diff(a, b)` | 两个日期相减 |
| `if_then(cond, a, b)` | 条件三元 |
| 数值比较 / 加减 | 标准 |

不开放通用脚本是因为：用户配置的"流程"会被存成 DSL 落到数据库。如果暴露通用脚本引擎，就等于让用户改 SQL 注入 / RCE 等价物。受限语法让 schema 校验与 AI 生成都能静态把关。

### 3.1.3 版本机制：每业务多版本 + college_id 路由

```
workflow_definition 表
  biz_type     leave / leave_return / work_study / ...
  college_id   NULL = 全校默认 / 非 NULL = 学院专属
  version      同 (biz_type, college_id) 自增
  status       draft / published / archived
  yaml         完整 DSL
```

引擎运行期解析逻辑：

1. 学生发起 leave，引擎按 `(biz_type='leave', student.college_id)` 解析
2. 命中学院专属版 → 用这版
3. 否则回落 `(biz_type='leave', college_id IS NULL)` 全校默认

学院差异化 = 多落一行 YAML，**不做 patch、不做合并算法**。这是放弃 v0.4 OrgPatch / TimePatch 设计后的极简方案，详见 5.4。

### 3.1.4 流程驱动事件（与通知中心解耦）

引擎在关键节点抛 Spring `ApplicationEvent`，业务/通知/外部集成都从事件订阅，互不耦合：

| 事件 | 何时抛 | 谁订阅 |
|------|-------|--------|
| `TaskAssignedEvent` | 一个 approval 节点被分配给某审批人 | `TaskAssignedNotifier` → 发"任务到达"通知；前端待办计数 |
| `WorkflowFinishedEvent` | 流程走到 `end` 节点 | `LeaveWorkflowListener` 同步 `leave_request.status`；可选 `GateAutomationListener` 调门禁 |
| `NodeEnteredEvent` | 进入任意节点 | 审计日志 |

## 3.2 通知中心

### 3.2.1 定位

通知中心是 **流程驱动 + 模板化 + 多渠道扇出** 的统一底座。业务侧只调一个方法：

```java
notificationOrchestrator.send(templateCode, sourceType, sourceId, recipients, vars);
```

剩下的事（按收件人角色偏好选渠道、双轨学校去重、AI 引用提示、模板渲染、失败回退）全在底座里做。**禁止业务侧绕开 Orchestrator 直接 `NotificationService.send`**。

### 3.2.2 模板表（notification_template）

每条业务通知 = 一行模板 seed：

```sql
notification_template
  code            LEAVE_SUBMITTED / WORKFLOW_TASK_ARRIVED / ...
  title_tpl       "{{student_name}} 提交了 {{leave_type}} 假申请"
  body_tpl        "时长 {{days}} 天, 起止 {{start}} ~ {{end}}, 点击审批"
  default_channels [ "in_app", "miniapp_subscribe", "wecom" ]
  category         leave / task / system
  ai_pinned        true  -- 通知中心进 AI 时是否自动 pin
```

新模块发通知的代价 = **加一行 seed + 调 Orchestrator.send**。模板可在「系统配置 → 通知中心」里改文案、改默认渠道，不动代码。

### 3.2.3 三渠道扇出

| 渠道 | 实现 | 适用 |
|------|------|------|
| **站内信** | `notification` 表 + 通知中心 UI 列表 | 默认全量入库 |
| **微信小程序订阅消息** | 模板 ID + `subscribeMessage.send` | 学生侧高频 |
| **企业微信应用消息** | 企业微信 webhook + 回链小程序 | 教职工侧 |

每个角色可在「我的偏好」里关掉某些非关键渠道（紧急通知不可关）。Orchestrator 收到 send 后按收件人偏好计算实际渠道集。

### 3.2.4 双轨学校去重

双轨学校（学院制 + 书院制）一名学生同时挂在学院辅导员和书院导师下。同一事件（例：完全通过 leave 通知）发给学生时，Orchestrator 通过 `(source_type, source_id, template_code, recipient_id)` 唯一索引去重，**学生只收一条**，不会因为多个上级而被打扰两遍。

### 3.2.5 任务到达通知（WORKFLOW_TASK_ARRIVED）

工作流引擎抛 `TaskAssignedEvent` 时，`TaskAssignedNotifier` 自动发 `WORKFLOW_TASK_ARRIVED` 模板。这意味着 **任何新业务接入工作流后，待办通知自动就有**，业务侧不需要单独写"通知审批人"的代码。

### 3.2.6 通知中心 UI（用户视角）

![站内通知中心：分类筛选 + 已读未读 + AI 引用提示 + 一键跳转源头](screenshots/notifications_admin.png)

每条通知含：来源（leave #1234）+ 摘要 + 发生时间 + 操作（去审批 / 标记已读）。AI 助手在用户问"我有什么待办"时，从通知中心读出 pinned 条目作为答案上下文。

## 3.3 AI 操作能力（通用 Pattern）

朝夕里 AI 不只是"问答"——它能 **代用户操作系统**。这套能力以两个 pattern 体现，且 **必须保证未来所有配置 / 业务都按这套接入**。

### 3.3.1 Pattern A：路由 + 锚定 + 高亮（Navigate）

校管理员说"我想看一下公假的配置"——AI 不应该再让他点 5 下菜单。实际行为：

```
用户："看一下公假配置"
   │
   ▼
[AI sidecar chat.py · short-circuit]
   识别 "看 X 假别配置 / 规则 / 工作流" 意图,直接 emit action,不进 LLM:

      { type: "navigate",
        payload: { page: "leave", tab: "rule", focus: "official_business" },
        highlights: [ { selector: 'leaveType[code=official_business]' } ] }
   │
   ▼
[前端 AIPanel]
   1. 触发 router 跳转 /leave?tab=rule
   2. 等页面挂载完,scroll 到 focus 的 anchor
   3. CSS class 加 pulse 高亮 2s,然后 fade
```

关键设计：

- **Short-circuit**：识别意图后**跳过 LLM**直接发 action，省 token / 省延迟。
- **三参数**：`page` + `tab` + `focus`，前端路由层和页面内部锚定层分工明确。
- **Highlights 数组**：高亮可指向多个元素，复杂场景（例如"对比一下事假和病假"）能同时圈两条。
- **支持的意图覆盖了所有配置项**：假别、字段、通知模板、岗位、流程节点等。

`navigate` 工具的 system prompt 节选：

```
用户明确说去某个页面时，直接调用 navigate，无需确认。
"看一下事假工作流"   → navigate(page=leave, tab=rule, focus=personal)
"看公假规则"        → navigate(page=leave, tab=rule, focus=official_business)
**禁止用知识库回答、禁止说「无法查看」、禁止反问** — 只要识别出
对象名就直接 navigate。
```

### 3.3.2 Pattern B：自然语言改配置（Propose）

校管理员说"事假改成最多 5 天"——AI 生成新 YAML 草稿 + 弹 diff 预览卡，老师确认后入库。

```
[小夕 chat] 用户："事假改成最多 5 天"
     ↓ tool-call
[Java propose 接口]
     │ 1. 拉当前 published YAML (按 college_id 找,缺失 fallback NULL)
     │ 2. 调 sidecar /workflow/propose-config-change
     ▼
[Python sidecar]
     │ 1. 把 instruction + 当前 YAML 喂给 LLM
     │ 2. LLM 输出修改后的完整 YAML (不是 patch,不是 op)
     │ 3. YAML schema 校验 + 受限表达式语法校验
     ▼
[Java propose 接口]
     │ 1. 收到草稿 → 生成 diff (before/after)
     │ 2. 准备前端 action
     ▼
[前端] navigate 到「请销假配置」+ 弹 diff 预览卡
     ▼
老师 review → 接受 → 落库为下一 version → archive 旧版
       否决 → 关闭,不落任何东西
```

当前已有的 propose 工具：

| 工具 | 业务 | 备注 |
|------|------|------|
| `propose_workflow_config_change` | 请销假 / 销假 / 勤工流程 YAML | 接 `biz_type` + 可选 `college_id` |
| `propose_notification_config_change` | 通知模板 | 文案、渠道、AI pin 状态 |

### 3.3.3 强制铁律：未来所有"可配置项"都要走这两条 pattern

> **设计原则（编码时遵守）**：新增任何"用户能调"的业务（违纪规则、岗位类目、收集单字段、勤工评分项等）时，**必须同时提供**：
>
> 1. 一份后台编辑页（手动配置兜底）
> 2. 一条 `navigate` 路由 + 锚定 anchor（让 AI 能跳进来）
> 3. 一个 `propose_*_change` 工具（让 AI 能自然语言改）
>
> 缺一项的业务不能并入 main。这是为了避免"AI 能力随业务长期碎片化"——一旦某个新模块漏掉 navigate，用户体验立刻分裂。

落库时严格"AI 不直接改数据"原则：所有 propose 都生成草稿 + diff，用户在 UI 上手动确认才入库，并写 changelog。

## 3.4 字段目录（Field Catalog）— 让"加字段"成为单点改动

### 3.4.1 痛点

最初版本里，"在过滤区加一个新字段"涉及 6–9 处编码：

1. 后端 DTO 加字段
2. Mapper SQL 加 `<if>`
3. Service 透传
4. 前端 API 类型加字段
5. 前端 useState
6. 前端 FilterRow + Chip 渲染
7. 前端"已选"chip 条
8. AI tool input_schema 加字段
9. AI prompt 加自然语言映射

任何一处漏掉行为就破。试点学校第二所、第三所提"我们还要按 XX 筛"时工作量线性堆积。

### 3.4.2 方案：一份 yaml 同时驱动后端 SQL + 前端 chip + AI 工具

```yaml
# field-catalog/student.yaml (节选)
page: student
fields:

  - key: gender                   # API 字段名 + 内部 key
    label: 性别                    # 用户看到的名字
    description: 性别;只在用户明确提到时填。   # AI prompt 用
    type: enum                    # 渲染类型 (text / enum)
    sql:
      strategy: simple_eq         # SQL 拼接策略
      column: u.gender
    options:
      static:
        - { value: male,   label: 男 }
        - { value: female, label: 女 }
    aliases:                      # AI 自然语言别名
      - { phrase: 男生, value: male }
      - { phrase: 女生, value: female }

  - key: academy
    label: 书院
    type: enum
    visibility: { tenantSetting: enable_residential_track }   # 双轨制门控
    sql: { strategy: residential_subquery, residentialType: academy }
    options:
      dynamic: { source: endpoint, url: /student-residential-options, key: academies }
```

### 3.4.3 5 种 SQL 策略

| 策略 | 用途 | 示例字段 |
|------|------|---------|
| `simple_eq` | `column = ?` | gender / status / grade |
| `ilike_multi` | 多列 OR 模糊匹配 | keyword（学号 / 姓名 / 手机号） |
| `residential_subquery` | EXISTS 子查询走 student_org_membership | academy / dormBlock |
| `jsonb_path` | `extended_info->>'k' = ?` | 留口子，本期未用 |
| `external_lookup` | 外部 API 拉 user_id 列表后 `IN(...)` | 留口子，例如教务系统拉绩点 |

### 3.4.4 加新字段的成本对比

| 场景 | 之前 | 现在 |
|------|------|------|
| 加"政治面貌"枚举筛选 | 改 9 处 | 改 yaml 一处（10 行） |
| 加"绩点 < 3.2"范围筛选 | 改 9 处 + 设计跨表查询 | 改 yaml + 选 `external_lookup` 策略 |
| 加双轨制学校特有的"楼层" | 改 9 处 + 双轨数据模型适配 | 改 yaml + 选 `residential_subquery` |

> 字段目录与 AI 的整合详见 4.3 — AI 用同一份 yaml 实时生成 tool schema + 别名 prompt，加字段 LLM 自动学会新别名。

# 第四章 学生信息库

## 4.1 模块定位

学生信息库是辅导员、院系领导、学工部使用最频繁的"底账"模块 —— 进任何一个学生相关的动作（请假审批、违纪记录、谈心谈话、画像分析）之前，先要找到这个学生。它的核心要回答三个问题：

1. **怎么找到学生？** 列表 + 多维度筛选 + 搜索 + 自然语言。
2. **学生的全貌是什么？** 学籍数据 + 历史动作（请假、违纪、签到、收集单填报）+ AI 风险标记。
3. **怎么样让"加新字段"代价不爆炸？** —— 不同学校字段诉求不一样，每接一所就改 6 处代码不可持续。

朝夕的解法是 **字段目录 + 双轨支持 + AI 自然语言过滤** 三件套。

## 4.2 数据模型

### 4.2.1 核心表

```
                     ┌──────────────────┐
                     │     sys_user      │  通用用户表
                     │ id, real_name,    │  学生 / 教职工 / 管理员都在这里
                     │ gender, phone,    │  通过 user_role 表挂角色
                     │ email, status     │
                     └──────┬────────────┘
                            │ 1:1 (学生)
                            ▼
                     ┌──────────────────────────┐
                     │      student_profile     │  学生学籍档案
                     │ id, user_id (FK),        │
                     │ student_no, grade,       │
                     │ college, major,          │
                     │ class_name, status,      │
                     │ education_level,         │
                     │ enrollment_date,         │
                     │ extended_info JSONB      │  ← 不固定字段都在 JSONB
                     └──────────────────────────┘
                            │
                            │ N:M  (一个学生可以同时属于"学院专业班"和"书院班")
                            ▼
                ┌────────────────────────────┐
                │  student_org_membership     │  学生 ⨯ org 关联
                │  student_user_id, org_unit_id│
                └────────┬───────────────────┘
                         │
                         ▼
                ┌──────────────────────────┐
                │         org_unit          │  组织节点（双轨）
                │ id, type, name,          │  type=college / major / class
                │ track,                   │  type=academy / dorm_block
                │ parent_id, deleted_at    │  track=academic / residential
                └──────────────────────────┘
```

**关键设计**：

- 学生的 *基础学籍* 在 `student_profile` 上 —— 这套是学校教务系统给的真值。
- 学生的 *组织归属* 在 `student_org_membership` —— 双轨学校允许一个学生既挂在"计算机学院 2023 级软件 1 班"（学术线），又挂在"博雅书院 明德 1 班"（生活线）。
- 学校学术线写死在 `student_profile.college / major / class_name`（直接列），生活线走 `student_org_membership` 关联（可选，由租户开关 `enable_residential_track` 控制是否启用）。
- 不固定的字段（紧急联系人、政治面貌、银行卡号 等）落在 `extended_info` JSONB 列。GIN 索引 + Generated Column 让常用 JSONB 字段也能高效检索。

### 4.2.2 双轨制（学院制 + 书院制）

#### 背景

国内高校近年改革，"书院制"（生活线）在新工科 / 新文科类高校越来越常见 —— 学生的 **学术线**（学院 → 专业 → 班级）由教务管，**生活线**（书院 → 书院班）由学工管。两套组织 **互不隶属**：

- 同一个学生既属于"计算机学院 2023 级软件 1 班"（学术），又属于"博雅书院 明德 1 班"（生活）。
- 老师角色也分两套：**辅导员**（学术线，按学院班分配）+ **书院导师 / 班导师**（生活线，按书院 / 书院班分配）。
- 书院班 **跨学院、跨年级混编** —— 一个明德 1 班里可能同时有计算机学院的大二、人文学院的大三、艺术学院的大一，让不同学科背景的同学一起做思政教育、社区营造。
- 同一件事（请假、违纪、表彰）两线 **都需要知道**，但不能 *重复发两次通知*。

朝夕从底层支持这种双轨结构，**单轨学校无感** —— 老学校 / 还没改革的学校用着完全像没有书院这回事。

#### 数据模型

不为双轨学校单独造一套表。**两轨共享同一份 `org_unit` 树**，仅靠 `track` 字段区分：

```
   org_unit (统一组织节点表)
   ┌──────────────────────────────────────┐
   │ id  / parent_id / type / name /       │
   │ track:                               │
   │   academic     (学术线 — 全校都有)    │
   │   residential  (生活线 — 双轨学校才有) │
   └──────────────────────────────────────┘

         academic                       residential
   ┌──────────────────┐         ┌────────────────────────────┐
   │  全校 (school)    │         │  全校 (school)              │
   │   ↓               │         │   ↓                         │
   │  学院 (college)   │         │  书院 (academy)             │
   │   ↓               │         │   ↓                         │
   │  专业 (major)     │         │  书院班 (dorm_block,技术名) │
   │   ↓               │         │                             │
   │  班级 (class)     │         │  (例:明德 1 班 / 弘毅 2 班) │
   └──────────────────┘         └────────────────────────────┘
```

> **dormBlock 这个 key 的来历**：技术名沿用早期"楼栋"语境的 `dorm_block`，但当前业务语义是"**书院班** — 跨学院 / 跨年级的混编育人班"，**不是物理寝室楼栋**。物理寝室在另一张表（P1 宿舍管理引入时再添）。yaml / API 字段名保留 `dorm_block` / `dormBlock` 是为了向后兼容，UI 标签是"书院班"。

学生与组织的关联表 `student_org_membership` 是 N:M —— 一个学生可同时挂在多个 org_unit 上（一个学院班 + 一个书院班）。

#### 双轨学校的角色补充

引入两个生活线专属角色：

| 角色 code | 中文 | 管辖范围 |
|----------|------|---------|
| `residential_advisor` | 书院导师 | 一个书院（type=academy）下所有学生 |
| `dorm_supervisor` | 班导师（书院班） | 一个书院班（type=dorm_block）下所有学生 |

两者通过 `sys_user_org_assignment`（双轨场景特有）挂到对应的 `org_unit.id`。这套关联让"该书院导师管的学生"用 closure table 一次 JOIN 就能查出来。

> 单轨学校 `sys_user_org_assignment` 表存在但没数据 —— 不影响也不出现在 UI 上。

#### 业务影响：审批、通知、画像 三处都要"双轨自洽"

**①  审批分发**

请假、违纪登记的审批人 *仍然走学术线辅导员*（书记、院领导）—— 双轨学校的请销假规则与单轨一致，不让书院插审批链。否则一名学生请假需要"辅导员 + 书院导师"双签，太重。

但是：

**②  通知抄送 + 双轨去重**

学生发生重要事件（请假提交、违纪记录、被表彰）时，通知中心 *抄送两轨相关人*：

```
     [事件: 张三 请事假 4-18 ~ 4-19]
                    │
        ┌───────────┴───────────┐
        ▼                       ▼
   学术线接收方             生活线接收方
   - 辅导员李老师            - 书院导师王老师
   - 书记 (按假别配置)       - 班导师张老师 (该学生所在书院班)
        │                       │
        └───────────┬───────────┘
                    ▼
            通知中心 ⊳ 双轨去重
  (同一人在两轨上都被命中时,只发一条)
```

**双轨去重** 是底座 `NotificationOrchestrator` 的内置能力 —— 业务侧调一次 `send(...)`，扇出名单合并、去重，再扇出三渠道。如果业务侧 *不* 通过 Orchestrator（直接调 `NotificationService.send`），双轨去重失效，所以在 `CLAUDE.md` 里把这条作为编码红线。

**③  学生画像页双 tab**

进入单个学生的画像页：

```
[张三 · 计算机学院软件 1 班 · 博雅书院明德 1 班]

  [学术线]   [生活线]   ← tab 切换
  ──────────
  辅导员: 李老师
  班主任: 王老师
  累计请假:  ...
  违纪:      ...
  收集单填报率: ...

  [生活线]
  ──────────
  书院导师:    王老师
  班导师:      张老师 (明德 1 班)
  同班同学:    跨学院 28 人,大一/大二/大三混编
  晚归记录:    ...
  社区活动:    ...
  宿舍报修:    ... (P1 模块,届时另起表)
```

学术线 tab 关心 *学习状态*；生活线 tab 关心 *日常起居 + 思政社区*。

#### AI 识别"书院"实体

字段目录里 `academy` 字段附 `visibility: { tenantSetting: enable_residential_track }`。这意味着：

- 单轨学校 → 字段目录拉到的 catalog *不包含* academy 字段 → 学生页过滤 chip 不显示 → AI tool input_schema 也没有 academy 参数 → LLM 看不见"书院"概念，自然不会乱发 action。
- 双轨学校 → catalog 含 academy 字段 → AI 自动学会"博雅书院"是个实体 → 用户说"博雅书院的男生"就能 tool-call。

**整套都是 *配置驱动*，不是 *双轨写两份代码***。同一份代码服务两类学校。

#### 单轨学校的"零打扰"保证

| 切面 | 单轨学校行为 |
|------|------------|
| 数据库 | `org_unit` 表存在但 `track='residential'` 行为 0 条 |
| 字段目录 | catalog API 返回的 academy / dormBlock 字段被 visibility 过滤掉 |
| 列表页 | "书院 / 书院班"两列不渲染（条件判断 `enableResidentialTrack`） |
| 过滤区 | "书院 / 书院班" chip 行不渲染 |
| 画像页 | 不出"生活线"tab |
| 通知扇出 | 接收方名单解析时跳过 residential_advisor / dorm_supervisor 角色 |
| AI tool 工具 | filter_students 工具 schema 不包含 academy / dormBlock |
| AI 提示词 | system prompt 不带"书院"相关别名段 |

只要租户把 `enable_residential_track = true` 翻开，所有切面同步生效；翻回 false 又无感。

#### 配置：书院 / 书院班名单从哪来

迁移到双轨制时，学校把书院列表 + 书院班列表 + 学生归属导入 `org_unit` 与 `student_org_membership`：

- 学校提供 Excel：`书院名,书院班名,学生学号` 三列
- 学工部老师在 Web 端「双轨制配置」页一键导入
- 后续单个学生改归属（如重新分班）走 `PUT /api/v1/students/{id}/residential-class`，body `{"org_unit_id": 1110}`。后端只动 `track='residential' + type='dorm_block'` 那条 membership，不影响学院班绑定，先删后插，幂等。
- 学生画像页对班导师角色直接出"换班"按钮，避免老师走管理后台

## 4.3 AI 自然语言过滤

> 学生信息库是"字段目录"机制的典型应用。字段目录的通用设计见 **3.4 字段目录**。本节讲学生页怎么落地它 + AI 自然语言过滤怎么用它。

### 4.3.1 用户视角

辅导员在小夕里说：

- "学生信息库找一下博雅书院的男生" → 跳到学生页 + 自动套上 `academy=博雅书院, gender=male`
- "2024 级人工智能专业的学生" → `grade=2024级, major=人工智能`
- "博雅书院有哪些专业" → `academy=博雅书院`，让前端列表收敛到那个书院的所有学生，老师从"专业"列肉眼看

### 4.3.2 实现链路

```
用户："博雅书院的男生"
```

# 第五章 请销假与工作流配置

## 5.1 模块定位

请销假是高校学生工作里 **使用频率最高** 的功能。学生频繁请假、辅导员频繁审批、学工部频繁调规则。朝夕的请销假覆盖了：

- **学生侧**：发起请假、看自己的历史、销假回校
- **审批侧**：辅导员/书记/学院领导/学工部按规则分档审批

## 5.4 配置管理：让"调规则"成为非工程动作

### 5.4.1 痛点（对比金智）

竞品系统的请销假配置：

- **金智**：90+ 配置项 × N 学院 ≈ 200–400 实际管理面，学工部老师初次配置 1-2 周
- **根因**：按"页签"切配置 + 学院差异只能"复制一份"配置 —— 不是业务复杂导致，是数据模型抽象不足

朝夕只把请销假 **作为单一具体应用**（不做通用配置平台），把配置面从"几百个表单字段"压成 **一份可编辑、可回滚、可 AI 改的 YAML**。

### 5.4.2 数据模型：workflow_definition 多版本表 + college_id 路由

```
┌──────────────────────────────────────────────────────────────┐
│  workflow_definition                                          │
│    biz_type        leave / leave_return / work_study / ...   │
│    college_id      NULL = 全校默认 / 非 NULL = 学院专属      │
│    version         同 (biz_type, college_id) 自增            │
│    status          draft / published / archived               │
│    yaml            完整 YAML(节点定义 + 假别 + 审批链)         │
│    change_summary  上次改动摘要(谁、为啥)                     │
└──────────────────────────────────────────────────────────────┘

学生发起请假时引擎按 (biz_type, student.college_id) 解析:
  ┌──── 命中学院专属版?  ──── 是 → 用这版           ┐
  │                                                 ▼
  └──── 否 → 回落 (biz_type, college_id IS NULL) → 全校默认
```

- **没有 patch / 没有 diff / 没有合并算法**。学院差异化 = 多落一行 YAML，引擎运行期单条 SELECT 就拿到生效定义。
- **版本是同 (biz_type, college_id) 桶内自增**。每次 apply 写新行，旧行保留为 `archived`，回滚 = 把旧行标回 `published`。
- 这套设计的代价是"改全校默认时各学院专属版不会自动跟"——但实践里学院特例本来就少（每个学校 0–3 条），这点显式更新比"patch 自动叠加却看不清生效结果"更可控。AI 改全校时会主动提示"3 个学院有专属规则，是否一并刷新"。

### 5.4.3 配置概览页（YAML IDE）

![请销假配置概览页：全局参数 + 假别管理卡（每张含审批链摘要 + 启用状态 + 上次修改）](screenshots/admin_leave_config.png)

老师进配置后，默认看到的是「概览态」—— 上半屏是按假别拆开的卡片（启用/禁用 + 上限 + 审批链摘要 + 上次修改人），下半屏是 YAML 源码（只读 + 折叠节点）。任何卡片点开都对应跳到 YAML 的相应 anchor。

切到「编辑态」后，YAML 是首要编辑面：

- **Monaco 编辑器** + YAML schema 校验（错语法即时红线）
- 表达式校验：`condition` 节点的表达式跑受限 mini-language 解析（见 3.1.2）
- 保存即创建新 published 版本，自动 archive 旧版本

### 5.4.4 学院专属规则

校管理员需要为某个学院单独开一份规则时（例：艺术学院事假必须加学工部备案），流程：

1. 概览页右上角切到「学院专属」tab
2. 选学院 + 「以全校默认为基线复制」
3. Monaco 里改 YAML（80% 不变 + 改某个 approval 档）
4. 保存 = 在 `workflow_definition` 表多落一行 `(leave, college_id=艺术学院, version=1)`

之后该学院的学生请假，引擎自动命中这版而不是全校默认。

### 5.4.5 版本回滚

![请销假配置 → 版本历史 tab：每个版本一行（版本号 / 操作人 / 改动摘要 / 操作按钮）](screenshots/leave_config_rule.png)

每次保存留痕。版本列表里支持 **diff 对比** + **一键回滚**：选中某个 archived 版本 → 「设为当前」→ 后端 INSERT 一行新版本，内容拷贝自该 archived，version+1。原 published 自动变 archived。

回滚是"前进式"——历史链路完整保留，回滚自身也是一次新版本。

### 5.4.6 不做 dry-run / 不做健康度巡检（设计取舍）

- **不做 dry-run 预演**：把校验前置到 AI propose 阶段（见 3.3 通用 propose pattern）。LLM 生成 YAML 时强制 schema 合规 + 表达式合法，前端 diff 预览卡里直接展示影响摘要。落库后是 published，不需要"先在 staging 跑 5 个学生"。
- **不做健康度巡检守护进程**：原 v0.4 草案里的"3 条扫描规则"在 multi-YAML 架构下大多无意义（角色不存在/过期时间窗 等校验都在保存时同步做了）。剩余的弱校验（如"某学院 published 版距全校默认已超过 30 天没跟新"）作为下一阶段巡检任务，P0 不上线。

## 5.5 销假：GPS 自助 + 电子围栏 + 人工兜底

销假是请销假闭环里"最容易卡住的环节"——学生回校时已经在路上 / 没空走流程，传统系统让学生再发起一遍审批是无效操作。朝夕把销假拆成三条路径：

### 5.5.1 GPS 自助销假（默认路径）

```
学生回到校园,小程序"销假"按钮
    │
    │  wx.getLocation → ReturnByLocationRequest
    │  { latitude, longitude, captured_at }
    ▼
[Java LeaveReturnService]
    │
    ├─ 比对 campus_geofence (电子围栏:校区中心点 + 半径)
    │   命中? ──── 否 ─→ 返回 OUT_OF_FENCE,提示学生走人工
    │     │ 是
    │     ▼
    ├─ 写入 leave_request.return_latitude / longitude / source='gps'
    ├─ 触发 leave_return workflow (按租户配置:
    │     · 默认免审批直接销假
    │     · 学生处可改为"辅导员后台审核")
    │
    ▼
WorkflowFinishedEvent → LeaveWorkflowListener 同步 status='returned'
                        → NotificationOrchestrator 发 LEAVE_RETURNED
                        → (可选) GateAutomationListener 通知校门禁
```

代码位置：`LeaveReturnService` / `ReturnByLocationRequest` / `LeaveWorkflowListener#case "gps"`。`apply_latitude/longitude` 在请假提交时一并存（用于审批佐证学生确实身在外地），`return_latitude/longitude` 在销假时存。

### 5.5.2 电子围栏配置

```
[请销假配置 → 销假定位 tab]
  ┌─────────────────────────────────────────┐
  │  启用 GPS 销假           [✓]            │
  │  校区中心点              [地图选点]      │
  │  围栏半径                500 m  [可调]   │
  │  超出围栏时的兜底         ⦿ 走人工销假   │
  │                          ○ 拒绝销假     │
  │  保存                                   │
  └─────────────────────────────────────────┘
```

存储：`UpdateCampusGeofenceRequest` 写到租户配置表。多校区学校支持多个围栏 OR 关系（任一命中即销假）。围栏半径默认 500 m，校管理员按校情况调。

### 5.5.3 人工销假兜底

GPS 不命中（学生还没回到围栏 / 手机没开定位 / 隔离观察等特殊情况）走人工兜底：

```
学生填 ManualReturnApplyRequest
    └ reason 必填:"父母车送 已在校门口"
    └ proof_file 可选

  ▼
辅导员收到任务 (TaskAssignedEvent → NotificationOrchestrator)
  ▼
辅导员审核 → 通过 → 走 leave_return workflow → status='returned'
              驳回 → 学生改后重提

leave_request.source = 'manual'
```

入口：`POST /leaves/{id}/return/manual-apply`（学生）+ `POST /leaves/{id}/return/manual-review`（辅导员）。

### 5.5.4 来源标记 + 后续追溯

`leave_request.source` 三值：

| source | 含义 | 触发渠道 |
|--------|------|----------|
| `gps` | GPS 自助销假 | 学生小程序定位 |
| `manual` | 人工销假（学生申请，辅导员审核） | GPS 兜底 |
| `auto` | 系统超时自动销假（如假期末仍未销假，按租户配置自动 close） | LeaveReminderService |

LeaveWorkflowListener 在销假完成的通知里附上 source（"GPS 自助销假" / "人工销假" / "系统自动销假"），便于辅导员追溯异常销假。

### 5.5.5 门禁集成（末端）

销假完成后可选触发 `GateAutomationListener`，调学校现有门禁/宿管系统的简单接入接口（按租户开关）：

- 标记学生从"请假离校"恢复"在校"
- 失败仅 `warn` 不阻塞业务

门禁接口挂掉，业务流程照样走完，留 warn 让运维补；不会因为门禁不稳定锁死销假动作。

## 5.6 累计假期统计

`leave_type_config.annualCapDays` 字段：每个假别可配置"年度上限天数"。学生发起请假时：

- 实时计算"本学年已请假 N 天"
- 提交时校验"加上本次会不会超 annualCapDays"
- 展示"还剩 X 天可请"

## 5.7 通知触达（请销假相关模板）

请销假流程里关键节点都通过 **通知中心** 扇出（通知中心整体设计见 **3.2 通知中心**）。本节只列请销假业务用到的模板清单：

| 事件 | 通知模板 code | 接收方 | 扇出范围说明 |
|------|--------------|-------|-------------|
| 学生提交请假 | `LEAVE_SUBMITTED` | 第一档审批人 | 走任务到达通知（见 3.2.4） |
| 审批通过流转下一档 | `LEAVE_APPROVED_PARTIAL` | 下一档审批人 + 学生 | 学生方为状态更新提醒 |
| 完全通过 | `LEAVE_APPROVED_FINAL` | 学生 + 班导师（双轨学校时） | 班导师同抄是因为他对学生当面有管理责任 |
| 驳回 | `LEAVE_REJECTED` | 学生 | 含驳回理由 |
| 销假提醒 | `LEAVE_RETURN_REMINDER` | 学生（到期前 24h） | LeaveReminderService 调度 |
| 超时未销假 | `LEAVE_RETURN_OVERDUE` | 学生 + 辅导员 | 24h 内 1 次去重 |
| 销假完成 | `LEAVE_RETURNED` | 学生 + 辅导员 + (双轨)班导师 | source 字段标 GPS / 人工 / 自动 |
| 销假被驳回 | （硬编码，P0 未模板化） | 学生 | LeaveWorkflowListener.handleReturnRejected |

业务侧只负责调 `NotificationOrchestrator.send(code, sourceType, sourceId, recipients, vars)` —— 双轨学校时学生若同时归学院辅导员与书院导师，发完全通过通知时**只发给学生一次**（去重靠 `(source_type, source_id, template_code, recipient_id)` 唯一索引），不会同一事件出现两条。

# 第六章 权限与多租户

## 6.1 模块定位

权限系统决定 **谁能看什么、谁能做什么**。学校场景里这个问题不简单：

- 同一个老师可能同时是辅导员 + 学院副书记 + 学工部小组成员，三套权限叠加。
- 一个学生进系统看到的菜单和一个学工部老师完全不同 —— 不是显隐就行，AI tool 列表也得变。
- 跨学校（多租户）时，A 学校的辅导员不能看到 B 学校的学生（schema 级隔离）。

朝夕走的不是"三权分立"那种重型方案，而是 **RBAC + 多角色叠加 + 多租户 schema 隔离 + DEFAULTS in code** 的组合。

## 6.2 角色与权限模型

### 6.2.1 角色清单

P0 阶段定义了这些角色（`sys_role` 表 seed）：

| 角色 code | 中文 | 端 | 典型职责 |
|----------|------|----|---------|
| `student` | 学生 | 小程序为主 | 申请请假、签到、填收集单、看通知 |
| `counselor` | 辅导员 | Web 主，小程序辅 | 审批、统计、谈心谈话、违纪记录 |
| `college_admin` | 院系管理员（学工办） | Web | 学院层级行政管理 + 统计 |
| `dean` | 院系领导（书记 / 院长） | Web | 学院维度看板、跨假别审批 |
| `student_affairs_officer` | 学工处工作人员 | Web | 全校事务执行 |
| `student_affairs_director` | 学工部部长 | Web | 改请假规则、改通知规则 |
| `school_admin` | 校级管理员 | Web | 全校配置、数据看板 |
| `super_admin` | 超级管理员（平台方） | Web | 跨租户运维 |
| `employer` | 用工单位主管 | Web | 勤工助学专属 |
| `residential_advisor` | 书院导师 | Web | 双轨学校生活线 |
| `dorm_supervisor` | 班导师（书院班） | Web | 双轨学校生活线 |

### 6.2.2 多角色叠加（关键设计）

**一个用户可以同时拥有多个角色** —— 在 `user_role` 表里挂多行。例如：

```
sys_user(id=1, name="李老师")
  └─ user_role: counselor      (我管软件 1-3 班)
  └─ user_role: college_admin  (兼任学工办主任)
  └─ user_role: student_affairs_officer  (借调到学工处做兼职工作)
```

李老师进系统时，权限是 *三套并集*：

- 看到的菜单是三个角色覆盖的所有入口
- 工具栏 / AI tool 列表是三个角色暴露的所有 tool 的并集
- 数据可见范围是三个角色 `data_scope` 的并集（自管学生 + 本院学生 + 全校事务）

> **设计取舍**：传统 RBAC 模型是"用户—主角色—权限"单链，副角色拼凑很麻烦。朝夕直接把 `user_role` 设为 N:M。Spring Security 的 `Authentication.authorities` 用并集填，Sa-Token 在 `StpInterface.getRoleList()` / `getPermissionList()` 里返回并集。**老师不需要"切换身份"**，登录后所有身份的权能同时生效。

### 6.2.3 权限码（permission code）

权限码是 *能不能做某动作* 的细粒度控制：

```
student:view                # 看学生信息
student:edit                # 改学生信息（变更审批）
leave:apply                 # 申请请假
leave:approve               # 审批请假
leave_config:edit           # 改请假配置
notification:publish        # 发通知
notification_config:edit    # 改通知规则
violation:record            # 登记违纪
violation:appeal            # 学生发起申诉
work_study:apply            # 申请勤工岗位
work_study:approve          # 审批勤工申请
work_study:settle           # 工资结算
...
```

每个角色挂一组权限码（`role_permission` 表）。Spring 接口上用 `@SaCheckPermission("leave:approve")`，Sa-Token 拦截器决定 200 / 403。

## 6.3 DEFAULTS in code（这套真正的差异点）

> 把"角色 → 权限码"的默认映射写在 Java 代码里 (`RolePermissionDefaults.java`)，租户初始化时一次性 seed 到数据库。后续允许学校改 DB 里的映射，但拥有 *被认证过的初始默认值*。

```java
// xg-platform/auth/RolePermissionDefaults.java (示意)
public final class RolePermissionDefaults {
    public static final Map<String, Set<String>> MAP = Map.of(
        "student",   Set.of("leave:apply", "checkin:scan", "violation:appeal", "work_study:apply"),
        "counselor", Set.of("student:view", "leave:approve", "violation:record", "checkin:manage", "collection:publish", ...),
        "dean",      Set.of("student:view", "leave:approve", "alert:view_college", ...),
        ...
    );
}
```

### 为什么这是真正的差异点

竞品系统的常见路径：
1. UI 给老师勾选"角色 X 拥有 Y / N / Z 权限"。
2. 老师不知道选哪些，要么乱勾、要么留空。
3. 系统行为依赖学校 *初次配置质量*。
4. 老师投诉"为什么辅导员不能审请假？" → 实施工程师远程帮她勾。

朝夕的路径：
1. **代码里把"辅导员该有什么权限"写死成默认值**（基于行业共识 + 试点学校反馈，迭代过 N 轮）。
2. 新租户开通时一键 seed 这套默认。**0 配置即可上线运行**。
3. 学校 *允许* 改默认（如多 / 少给某些权限），但 99% 学校用默认就够。
4. 默认值在代码版本里 —— 用 git diff 能看到 *谁 / 何时 / 为什么* 改的，相当于权限矩阵的 changelog。

这是一个 *小* 创新，不是"AI 革命"那种大词，但对实施成本影响很大 —— 上一个学校上线 = 改一个 yaml 开关 + 一键 seed，不是开 N 个会勾矩阵。

## 6.4 数据可见范围（Data Scope）

权限码控制 *能不能做*，data scope 控制 *看哪部分数据*：

| scope code | 含义 | 适用 |
|-----------|------|-----|
| `self` | 仅自己 | 学生看自己的请假 |
| `same_class` | 自己管的班级 | 辅导员 |
| `same_college` | 自己所在学院 | 院系领导 / college_admin |
| `same_residential` | 自己管的书院 / 书院班 | residential_advisor / dorm_supervisor |
| `school_wide` | 全校 | school_admin / 学工部 |
| `tenant_wide` | 跨租户 | super_admin（仅平台方） |

实现：每条业务查询接 `data_scope` 拦截，按当前用户的角色 scope 加 WHERE 子句（多角色 → 多 scope 取并集 → SQL OR 拼）。

> **ABAC 扩展（P1 后置）**：当前 P0 用 RBAC + scope 二级粒度。P1 阶段需要"按属性筛"（如只看本人指导的研究生）时再引入 ABAC，不预设。

## 6.5 多租户 Schema 级隔离

### 6.5.1 模型

每所学校 = 一个 `tenant`，每个租户独立 PostgreSQL **schema**：

```
postgres database
  ├── public                  # 全局元数据（tenant 列表、平台超管）
  ├── tenant_demo_univ        # demo 学校全部业务表
  ├── tenant_a_university     # A 大学
  ├── tenant_b_university     # B 大学
  └── ...
```

每个租户 schema 里有完整的业务表（user, student_profile, leave_request, ...）。**租户不共享表**，是 *不同的库（schema）*。

### 6.5.2 路由

```
HTTP 请求带 X-Tenant-Id header
   ↓
TenantContextFilter 解析 → TenantContext.set(tenantId) (ThreadLocal)
   ↓
MyBatis 走 SQL 前
   ↓
TenantSchemaInterceptor: SET search_path TO tenant_xxx, public
   ↓
所有后续 SQL 自动落到该 schema (无需改 SQL)
```

### 6.5.3 异步路径的 tenant 恢复

异步线程、定时任务、流程恢复时 **没有 ThreadLocal**。设计约束：所有跨任务表（如 `workflow_instance` / `notification_retry` / `student_event_log`）都把 `tenant_id` 作为 *表字段* 持久化。后台 worker 取一行任务时先 `TenantContext.set(row.tenant_id)`，再走业务逻辑。

> 之前测试期出现过一次幽灵 bug：定时器 scan 出某租户的待销假任务，发的通知却落到 `public.tenant_id` 的 schema 里 —— 就是因为忘了恢复 TenantContext。后来加了 *启动期单测* 强校验"任何后台 task 必须先 setTenant"，把这一类 bug 钉死在 CI。

### 6.5.4 私有化部署退化

学校要求私有化部署时，把 `multi_tenant.enabled = false`，TenantContext 全部强制 `tenant_id = "default"`。代码不变，部署变 —— 一份代码同时支持公有云多租户和私有化单租户。

## 6.6 认证：Sa-Token + Spring Security

P0 阶段两个并存：

| 用途 | 选型 | 为什么 |
|------|------|------|
| Token 颁发 / 校验 / 注解（@SaCheckLogin / @SaCheckPermission） | **Sa-Token** | API 友好、注解轻、Redis 续期方便 |
| AuthenticationContext / 兼容 Spring 生态 | **Spring Security** | Spring Boot 默认依赖，便于按需扩展 |

实现细节：

- 登录入口（`/api/v1/auth/login`）支持 *用户名/密码* 兜底，未来对接 *CAS / OAuth2 / SAML* 走学校自身 IDP（实施成本主要在协议适配，不在朝夕侧）。
- JWT Token 通过 Sa-Token 签发，存在 Redis（支持单点登出 + 续期 + 多端同登）。
- 全局拦截器 `SaInterceptor` 拦所有 `/api/**`，白名单 `/api/v1/auth/login`、`/actuator/**`、`/internal/**`、`/error`。
- `/internal/**` 是 sidecar 调 Java 用的内网通道（Java → Python sidecar 是 sidecar 主动出，反过来 sidecar → Java 走 `/internal`），假定走网络隔离，不需 Token。

## 6.7 与 AI 的耦合

权限不只决定 UI / API 可见，还决定 **AI 看到的工具列表**：

```python
# AI sidecar (chat.py) — 工具按用户角色过滤
user_roles = {r.strip() for r in (role or "").split(",") if r.strip()}
ui: list[dict] = []
for t in UI_TOOLS:
    if t["allowed_roles"] is not None and not (user_roles & t["allowed_roles"]):
        continue   # 用户角色不在白名单 → 工具不进 LLM
    ui.append(...)
```

学生角色看不到 `propose_workflow_config_change` 这种工具 —— LLM 都不知道这个工具存在，自然不会乱发。这种"工具级隔离" 比"输出后过滤"更稳，也更省 token。

# 第八章 AI 助手 小夕

## 8.1 模块定位

"小夕"是朝夕的 AI 助手 —— 全局自然语言入口。它存在的目的不是"为了有 AI 而有 AI"，而是 **降低使用门槛**：

- 学生不用学"病假在哪里点 → 选离校 → 上传证明 → 写紧急联系人"，直接说"我下周一请病假三天，要离校回家"。

- 校管理员不用学"配置在哪个 tab → 哪个字段 → 怎么改"，直接说"事假改成最多 5 天"。

**小夕不是替代 UI 的，是覆盖在 UI 之上的**。任何小夕能做的，UI 也能做。AI 不可用时（API 配额耗尽、网络中断、模型限流），UI 路径完整可用。

| `open_checkin_form` | 打开签到表单 | "为本周班会签到" |
| `open_collection_form` | 打开信息收集表单 | "新建一份国庆去向收集" |
| `filter_students` | 学生信息库套过滤 | "博雅书院的男生" |
| `open_violation_form` | 登记违纪 | "记录张三考试作弊" |
| `open_appeal_form` | 学生发起违纪申诉 | "我对这条违纪不服" |
| `propose_workflow_config_change` | 改请假规则 | "事假改 5 天" |
| `propose_notification_config_change` | 改通知规则 | "停用请假驳回通知" |

### 8.3.2 Query Tool —— 数据查询

LLM 同一轮调工具拿数据，再用工具结果生成回复。

| Tool | 用途 |
|------|------|
| `query_students` | 按条件查学生（不是 UI 过滤，是要用数据回答） |
| `query_leaves_pending` | 查待审批请假 |
| `query_violations` | 查违纪记录 |
| `query_collection_progress` | 查收集单填报进度 |
| `query_alerts_open` | 查待处理预警 |
| `query_unread_notifications` | 查未读通知 |
| `query_today_classes` | 查本班今日课表 |
| `resolve_date` | 把"下周一"翻译成 YYYY-MM-DD |
| ... | （共 ~20 个） |

### 8.3.3 角色门控

每个工具都有 `allowed_roles` 集合：

```python
{ "name": "propose_workflow_config_change",
  "allowed_roles": {"school_admin", "super_admin", "student_affairs_director"} }
```

学生看不到也用不了管理员才有的工具。LLM 看到的 tool 列表是 *按用户角色过滤过的子集* —— 防止越权 + 减少 LLM 幻觉。

## 8.4 数据查询铁律

LLM 自己说"有 N 条" / "有以下名单" 是高风险幻觉源。系统对此设了铁律：

> 任何涉及数据（条数、名单、统计、是否存在、未读数等）的问题，**本轮必须先调对应 query_* 工具**，再基于工具返回结果作答。
>
> - 未调用工具前，**严禁** 输出任何具体数字、名单、人名。
> - 用户单纯问（"有哪些待审批的请假？"）→ 本轮直接调工具，不先反问"需要查看吗？"。
> - 工具返回 0 条 → 如实说"暂无"，不要编。

跨轮一致性：上一轮工具已返回的真实数据优先于 LLM 自己之前说过的数字。

## 8.5 RAG 知识库

知识问答类问题（"国家奖学金的评选条件" / "请假超过几天需要院长审批"）走 pgvector：

| 设施 | 实现 |
|------|------|
| 向量库 | PostgreSQL + pgvector | 不引入 Milvus，P0 规模够用 |
| Embedding | 通义千问 Embedding API |
| 文档来源 | 学校政策文件 / 规章制度 / 奖学金办法 / FAQ |
| 检索粒度 | 段落（512 token chunk） |
| 答案溯源 | 回复附原文摘录 + 文件 ID（让学生进一步查阅） |
| 兜底 | RAG 命中 0 条时，AI 不编 —— 引导转人工或给系统配置 |

## 8.6 上下文与"指代"机制

学生工作的对话经常带 **指代** —— 老师在"待关注学生"列表上点过几个学生，然后说"分析一下这些学生"。这个"这些"必须能解析到具体对象。

### 8.6.1 Pin-to-AI（钉对象）

任何展示对象的卡片（学生、洞察、请假记录）都有 **📌** 按钮。点击后对象进入"右栏 pin 池"。小夕的 system prompt 自动加上：

```
## 用户指代的对象 (来自右侧面板)
1. [student] 张三 (id=9001) — 计算机学院 / 2023级 / 软件工程 / 软件2301班
2. [student] 李四 (id=9002) — 人文学院 / 2024级 / 历史学
3. [insight] 张三 30天考勤异常 (id=insight-7)

- 用户消息中的「这个/这条/他/她」默认指上述对象。
- **指代消歧**：用户限定了修饰词（"高风险"等）时，仅考虑 label 含该关键词的子集。
```

LLM 看到这段后，能正确解析"分析一下他们"= 学生 9001 + 9002，调 `query_*` 工具拿数据。

### 8.6.2 AskAIChip（一键问 AI）

对象旁边的 **？AskAI** 按钮，把"对象 + 一段预设 prompt 模板"打到小夕：

```
点击 AskAI("学生 9001", prompt="分析这个学生最近 30 天的考勤、请假、违纪情况，给出风险建议。")

→ 自动 pin 学生 9001 + 自动发送 prompt
```

老师不用手打长指令，节奏被加速。

## 8.7 多端

小夕在两端都能用：

| 端 | 形态 | 特点 |
|----|------|------|
| **Web 管理端** | 右栏可折叠面板 | 历史 + pin 区 + 输入框；快捷键 ⌘K 唤起 |
| **微信小程序** | 单独 tab "小夕" | 学生侧重；语音输入走微信同声传译，长录音走讯飞 lfasr 后端转写 |
| **企业微信** | （P1）会话式入口 | 推送卡片 + 回链小程序；P0 阶段企微仅做触达，不做对话 |

## 8.8 降级路径

| 故障 | 表现 | 降级 |
|------|------|------|
| Sidecar 起不来 / 不可达 | 小夕入口显示"暂不可用" | UI 完整可用，所有手填路径都走通 |
| 大模型 API 配额耗尽 / 限流 | 单次对话失败 | 自动重试 1 次后切换备用模型（DeepSeek ↔ 通义） |
| RAG 检索 0 命中 | 知识问答类问题 | 不编答案，转人工或给"看上去类似的政策" |
| Tool schema 拉取失败 | filter_students 等动态字段失效 | 降级到 0 字段 schema，LLM 不会乱发 action（前端面对空 action 也无副作用） |
| Pin 上下文异常 | "指代消歧"失败 | LLM 反问"是指 X 还是 Y" |

## 9.1 模块定位

## 9.4 工作流引擎接入

岗位审批 / 工资结算都走第四章的工作流引擎。每个流程的 DSL 配置在租户初始化时 seed，实施期可改：

```
position_review (岗位审核流):
  form_submit (employer 填岗位) → approval (学工部审) → end

application_review (申请审批流):
  form_submit (学生填申请) → approval (employer 审) → notification → end

salary_settlement (工资结算流):
  form_submit (employer 提交工时合计) → condition (金额 > 阈值) →
    approval (学工部) → notification → end
```

走工作流的好处：审批超时自动催促、跨档分流、抄送配置 —— 这些机制和请销假复用，不再为勤工助学单独造。

## 9.6 经济困难学生倾斜

学工部对低收入学生有政策性倾斜：

- 学生在 `StudentWorkStudyPreference` 里勾选"已建档贫困生"（自愿声明 + 学工部一次性认证）
- AI 推荐 / 用工单位筛简历时 *默认* 给经济困难学生 +N 权重
- 学工部看板里专门统计"经济困难学生岗位覆盖率"，低于阈值会 alert

> **隐私保护**：贫困生标识仅在 *后端排序权重计算* 时使用，*不* 暴露给用工单位主管 — 主管侧只看到 AI 推荐结果，看不到"为什么这个人排在前面"的真实原因（避免歧视风险）。

## 9.7 年度设置（YearSetting）

每个学年学工部维护一份「年度勤工设置」：

| 设置项 | 含义 |
|--------|------|
| `default_hourly_rate` | 默认时薪（岗位发布时填的最低值不能低于这个） |
| `max_hours_per_month` | 单学生月度工时上限（防过度劳动） |
| `total_budget` | 年度总预算 |
| `application_deadline` | 当年申请窗口截止 |

`WorkStudyPosition` 通过 `academic_year` 字段绑定到对应的 YearSetting。校历跨年时切年时，规则自动切换。
