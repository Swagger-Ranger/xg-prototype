01定位与背景

关怀模块解决的核心问题：把分散在请假、违纪、考勤、勤工、跟进记录里的弱信号，规则化地聚合成「该不该管、谁来管、什么时候管完」的可闭环任务，并用 AI 降低辅导员的判断与沟通成本，但绝不替辅导员做处置决定。

1.1 从 student_alert 到 care_task

早期系统用 student_alert 做「异常预警」——一条预警只有「开/关」两态，是被动看板。关怀模块用 care_task 取代它：每条命中规则的信号成为一个有 责任人（assigned_to）、SLA（due_at）、7 态状态机、转介 / 改期 / 误报反馈 的工作项。student_alert 在 P0 过渡期并行保留（旧 /alerts 页与后端未删），但已从导航、AI 观察员、schema-catalog 中全面摘除（A1 硬切）。

1.2 设计哲学

  • 人是决策者，AI 是放大器。所有结案、转介、误报判定由辅导员做；AI 只产出「为什么触发 / 可以聊什么 / 不宜碰什么 / 可对接资源」的辅助画像。
  • 规则可运维、可降级。规则集版本化、可启停、可整体调严重度；AI 不可用时全部业务经传统界面完整运行。
  • 安全优先于完整。危机求助走独立通道，刻意不进规则引擎、不喂大模型，宁可少一行分类也不漏一次危机。
  • 最小爆炸半径。取代旧预警时保持对外契约（metric key、ref 类型字面量）不变，把改动锁在数据源与文案层。

02总体架构

关怀是单体 Java 服务内的一个领域模块，AI 能力通过 Python Sidecar 经 REST 调用，前端在 Web 管理端与小程序共享类型与请求层。

信号源请假 / 违纪 / 考勤 / 勤工 / 跟进事件（student_event_log）
→

→
任务库 + 状态机care_task：7 态 / SLA / 转介 / 反馈 + 审计
→
处置触达辅导员工作台 / AI 观察员 / 通知 Orchestrator

04任务生命周期与状态机

状态机是静态常量表，不入库（CareTaskTransitions）——防止规则被运行时改坏。非法 (from, event) 抛 CARE_TASK_INVALID_TRANSITION，全局异常处理器透传中文文案。前端据此网关按钮可见性（isAllowed），避免对非法态发起调用。

4.1 事件

事件触发者必填
ACCEPT 接单辅导员—
REJECT 拒绝 / 误报辅导员reason_code（写 feedback）
RESOLVE 完成辅导员note（可选）
TRANSFER 转介辅导员target_dept + reason_detail
RESCHEDULE 改期辅导员days ∈ {1,3,7}，reschedule_count+1
SAVE_FOLLOWUPService 内部首次保存跟进 → accepted 进 in_progress
OVERDUE_TICK系统定时due_at 过 → overdue

4.2 状态转移表

列＝当前状态，行＝事件；单元格＝目标状态，空＝非法。

事件 \ 状态pendingacceptedin_progressoverdueresolved/rejected/transferred
ACCEPTaccepted——accepted终态
不再迁移
REJECTrejectedrejected——
RESOLVE—resolvedresolvedresolved
TRANSFERtransferredtransferredtransferredtransferred

前端按钮必须按状态网关
受理仅 pending/overdue 合法、误报仅 pending/accepted 合法、完成仅 accepted/in_progress/overdue 合法。任何快捷入口（含 AI 观察员卡片）对非法态调用都会 400——按钮可见性必须与本表一致，输入型转移（转介 / 带备注完成）走详情页。

4.3 SLA 与超期

仅 pending / accepted 会被 OVERDUE_TICK 迁 overdue；in_progress 已在跟进中，超期由 SLA 报表呈现而不进 overdue（避免无谓抛/catch）。

05规则引擎

CareRuleEngine 周期扫描信号，命中内置规则集（R001–R012，产品方维护、rule_version 版本化）即生成 care_task，并把命中证据写入 trigger_data（含 rule_name 中文名，前端展示不暴露 R0xx 代号）。

  • 去重 / 冷却（V125）：trigger_count / last_triggered_at / cooldown_until 防同一规则对同一学生短时间内重复刷任务。
  • 可运维：/care/rules 启停单条规则、/care/rules/severity-offset 全局严重度偏移（−1 / 0 / +1），不改代码。
  • 效果闭环：/care/rules/effect-report 按规则给出命中数、接单率、办结率、平均办结时长、误报率、拒绝原因分布——误报 / 拒绝反馈直接喂回此报表。
DSL 受限
规则表达式受限语法，computed 仅允许内置函数（duration_days / date_diff / if_then），不暴露通用脚本引擎。

06AI 辅助：小夕画像

每个 care_task 可懒加载一份「小夕怎么看」结构化画像，帮辅导员快速进入谈话，不替代判断、不参与状态决策。

// CareBrief（小夕画像，PRD §11.3）
{
  why:               "为什么触发——把规则命中翻译成人能懂的一句话",
  talking_points:    ["可以聊的话题…"],
  avoid_topics:      ["本次不宜触碰…"],
  campus_resources:  ["可对接资源（心理中心 / 资助 / 学业）…"],
  follow_up_days:    7
}
方面设计
三态ready 展示 / pending 懒加载中 / failed 可重试
限流手动「重新分析」命中 5 分钟限流抛中文 BizException
降级AI 区失败 ≠ 任务失败：sanitize 拦截 / 懒加载失败 / 限流 均不影响详情页处置按钮可用性
版本task_ai_brief_history 留历史，current_brief_id 指最新可用版

07多角色与权限边界

关怀的可见 / 可操作范围由 care_task.assigned_to 锚定，统一口径 CareAdminAccess。该口径同时约束：列表 API、首屏摘要、AI 观察员、schema-catalog NL→SQL。

角色口径说明
辅导员 counselor / 班主任 class_masterct.assigned_to = :owner_id只看 / 只可处置分派给本人的任务
院系领导 dean / 学工部部长 / 校级管理员 / 超管""（不过滤）看本校全部（本院收窄留 P1）
学生 studentct.student_id = :owner_id只读知情兜底，不主推 / 不弹窗
role_scope 必须枚举所有角色
schema-catalog 里未列角色，roleScopeFor 返回 "" = 不过滤 = 越权。新增可被 AI 观察员查询的表，role_scope 必须穷举所有角色。

08AI 观察员集成

辅导员 / 院系工作台首屏的「AI 观察员 · 主动关怀」卡片，是关怀任务的智能聚合入口。本节是关怀与 AI 的耦合面，也是本系统改动密度最高处。

8.1 数据链路

WorkspaceMetricsServiceJDBC 聚合 care_task → metrics.recent_alerts（key 名沿用旧契约）
→
InsightService持久化一行 insight，永不抛错（降级空数组）
→
Sidecar /insightsDeepSeek 综合，产出 category=「主动关怀」洞察 + refs
→
InsightCard.tsx渲染行 + 状态网关动作
最小爆炸半径的关键约定
取代旧 student_alert 时，metric key 名（recent_alerts / rule_name …）与 ref 类型字面量 "alert" 全部保持不变，只换底层数据源与文案。"alert" 是对用户不可见的内部字符串，保留它使 sidecar 的 reflection 校验、前端类型、过滤逻辑零改动——id 现在是 care_task.id。

8.2 行渲染以后端为准，不依赖 LLM 完整性

大模型常只为最严重的几条出 ref，导致「标题说 4 条只渲染 2 行」。因此卡片的可操作行直接取后端权威的 recent_alerts（按 SQL 严重度排序 critical→low），只要该洞察引用了关怀任务就把全部待处理 care_task 渲染出来——LLM 只负责文字综述，不负责枚举。

8.3 按状态机网关的快捷动作

status受理误报完成发起谈话
pending✓ accept✓ reject(rule_not_applicable)—✓
accepted—✓✓ 跳详情填备注✓
in_progress——✓ 跳详情填备注✓
overdue✓—✓ 跳详情填备注✓

受理 / 误报为卡内一键（误报＝以 rule_not_applicable 关闭并回写规则效果反馈）；完成＝跳 /care/task/:id 详情页带结案备注正式 RESOLVE，避免一键误关紧急任务；点规则名亦进详情页（同关怀工作台）。

口径必须按 assigned_to，不能按「学生属于本班」
recentAlerts / alerts_open / alerts_critical 的辅导员口径必须 WHERE assigned_to = counselorId（dean=全部，单班抽屉再按本班学生收窄）。曾按 student-in-class 导致：卡片 10 条但首屏 4 条自相矛盾；且对非本人任务调受理/误报会被 requireOwnedTask 拒绝、按钮静默失败。必须与 care role_scope 与首屏摘要完全同口径。

8.4 谈话回链（option B）

从卡片「发起谈话」带 careTaskId 进辅导谈话页，落库时写 counselor_talk.source_care_task_id（单向，与 source_alert_id 平行）。不做 care_task 反向列、不联动状态机——保持关怀状态机纯净。

09危机求助快速通道

危机通道（crisis_signal）是与关怀 R001–R012 并行的 P1 例外通道，处理「学生显式求助 / 需立即人工核实」。它与关怀共享工作台与画像，但在架构上刻意隔离。

三条安全红线
① 不进规则引擎——xg-ai 判定显式求助命中后回调 CrisisSignalService.report，不走 CareRuleEngine。
② 不喂大模型——危机数据不进 insight metrics、不进 LLM 上下文，AI 观察员的危机条由前端直接拉 /crisis-signal 渲染，绝不让模型编辑 / 转述危机内容。
③ 通知不静默失败——首次通知 send 返回 null 语义歧义（去重/无收件人/模板缺失/真失败），一律按失败响亮报错留人工介入，绝不当「已送达」。
方面设计
开关xg.crisis.enabled 默认 false，仅门控 report() 入口（关闭即 no-op，对正常 chat 零影响）；list / detail / close 不受开关门控
幂等同 (tenant, messageId, ruleVersion) 复用同一行，不重复发首次通知
权限crisis_signal 无 assigned_to，越权过滤落在「学生归谁管」——经 AssigneeLookupMapper.findCounselorsOfStudent 反查（与危机通知收件人解析同一条双轨 SQL，口径不漂移）
审计更严详情每次「谁在何时看了」都写一条 view 审计，比普通关怀更严
通知固定模板 CRISIS_SIGNAL_IMMEDIATE，经 NotificationOrchestrator 触达责任辅导员

9.1 观察员中的危机警示条

AI 观察员卡片最顶部（独立于 LLM 输出）渲染红色警示条：⚠ 危机 · 需立即人工核实 · N 例，按学生聚合（同学生多条 pending 折叠成一行，避免刷屏），点「立即核实」跳 /crisis/:signalId——与关怀工作台危机泳道同源同入口、复用危机配色（#cf1322 / #fff2f0 / #ffccc7）。

10关键设计约束与权衡

租户隔离
Schema 级

所有 SQL 经多租户插件路由；裸 JDBC 显式拼 schema 并校验合法。

AI 可降级
try-catch 包裹

AI 不可用时关怀全流程经传统界面完整运行。

通知铁律
必走 Orchestrator

新模块发通知 = seed 一行 notification_template + 调 Orchestrator，禁止绕开。

轻量优先
不加新组件

能用 PG（JSONB / 全文 / 向量）解决的不引入 MQ/ES。

Tool 粒度
用户任务级

AI Tool 是「接单 / 完成」而非 CRUD；用户身份从上下文取，不暴露给 Agent。

文案约束
不暴露 code

UI 全中文，不出现 rule_id / role_code / 英文枚举词。

10.1 取舍记录

  • 保留 ref 类型字面量 "alert"：语义上已是 care，但它是内部不可见串。改它会连锁波及 sidecar reflection、TS 类型、前端过滤——收益为零、风险扩大，故只换数据不换字面量。
  • 谈话回链只做单向：care_task 不加反向列、不因建谈话自动推进状态——关怀状态机是闭环判断的核心，不让旁路写入污染。
  • 危机按学生聚合：观察员是摘要面，6 条同一学生危机折叠成 1 行点开看最近——可读性优先于逐条罗列。

11安全与合规

  • 身份铁律：危机回调的 studentId 必须是调用方重校验已认证 token 解析出的，不接受 xg-ai 自报。
  • 下钻最小化：领导看个体须填 ≥30 字理由、受配额限制、写下钻日志（actor / 学生 / 理由 / 时间）；配额满抛中文 BizException。
  • 督办不泄露：领导界面只显示「已督办」，提醒经 Orchestrator 私下发给责任辅导员，不暴露处置细节给上级界面。
  • 危机详情可追溯：每次查看写 view 审计，比普通关怀严。
  • 脱敏展示：管理摘要脱 rule_id、不点名学生 / 辅导员；前端不展示英文 code。
