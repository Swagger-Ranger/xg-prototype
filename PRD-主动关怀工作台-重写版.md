## 1. 一页摘要

### 1.1 给学工领导

主动关怀工作台不是“预测谁会出事”，而是把散落在请假、考勤、成绩、违纪、宿舍、勤工助学等系统里的异常信号汇总起来，形成可解释、可追溯、可闭环的关怀任务。系统只回答“哪些明确线索需要老师看一眼”，不做学生风险评分，不给学生贴长期标签，不让 AI 做诊断或决策。

学校获得的是一套治理能力：本周产生多少主动关怀、多少闭环、哪些规则误报多、哪些任务超期、哪些下钻访问异常。所有关键动作留痕，便于审计、复盘和持续改进。

### 1.2 给辅导员

辅导员每天打开工作台，看到不超过 5 个优先级最高的关怀任务。每个任务说明为什么进入清单、有哪些事实证据、可以从哪些中性话题切入、有哪些话题本次不宜触碰。处理完后，辅导员用“完成 / 改期 / 拒绝 / 转介”结束任务，系统自动留痕。

这不是新增一个未读红点，而是把“今天最值得主动联系的学生”筛出来，并让记录和闭环更省力。

### 1.3 给产品和研发

核心对象不是 alert，而是 `care_task`。事件流负责提供事实，规则引擎负责生成任务，AI 只负责解释和沟通提纲，通知中心负责触达，审计系统负责留痕，反馈数据负责规则迭代。

北极星指标：

> 每位辅导员每周完成不少于 5 次系统推荐的主动关怀，且其中不少于 80% 形成闭环记录。

## 2. 产品定位

### 2.1 一句话

主动关怀工作台是一套面向高校辅导员和学工管理者的“学生异常线索发现 + 主动关怀建议 + 闭环跟进留痕”系统。

### 2.2 做什么

- 汇总学生相关业务系统中的异常事件，形成标准事件流
- 用产品方内置规则识别需要主动关怀的组合线索
- 自动生成带责任人、SLA、触发证据的关怀任务
- 用 AI 提供中性解释和沟通提纲，但不参与判定
- 通过统一通知中心分层触达
- 支持辅导员处理、反馈、转介和闭环记录
- 支持院系和学校管理者查看汇总、超期、趋势和审计
- 支持学生本人查询当前未关闭关怀类型摘要

### 2.3 不做什么

- 不做学生风险总分、排名或红黄绿榜
- 不给学生打“高风险 / 低风险 / 重点关注”永久标签
- 不让 AI 判断学生是否有心理风险
- 不自动通知学生家长
- 不把心理咨询、医疗、家庭情况接入 P1 规则
- P1 不开放学校侧规则编辑器
- P1 不引入 ClickHouse、ES、RabbitMQ 等新基础设施

## 3. 用户角色

| 角色 | 主要使用场景 | 权限边界 |
|---|---|---|
| 辅导员 | 看今日关怀清单，接单、记录、闭环 | 只能看自己负责学生的任务和审计 |
| 院系学工领导 | 看本院汇总、督办超期、查看趋势 | 默认看汇总；下钻个体需理由和额度控制 |
| 学校学工管理者 | 看全校汇总、规则效果、异常下钻审计 | 默认看汇总；下钻个体需理由和额度控制 |
| 校领导 | 突发事件研判、全校治理审阅 | 下钻不限频，但全部留痕并进入审计 |
| 规则管理员 | 看规则启停、效果报表、全局严重度偏移 | P1 只能启停和偏移，不能改阈值和 DSL |
| 学生本人 | 查询当前未关闭关怀类型摘要 | 只能看自己，且不展示历史次数和规则细节 |
| 心理中心 | P2 独立模块接收心理类线索 | P1 不接入心理数据 |

角色实现遵守现有角色 / 团队模型：长期岗位走 `sys_role.kind = 'role'`，临时业务编组走 `sys_role.kind = 'team'`。本模块不新增“每个学院一类角色”之类的数据膨胀模型。

## 4. 核心原则

### 4.1 可解释

每条关怀任务必须回答：

1. 这是哪个学生
2. 为什么进入清单
3. 触发依据是什么
4. 谁负责跟进
5. 建议何时完成
6. 最终如何闭环

不可解释的输出不能生成关怀任务。

### 4.2 不评分、不排名、不贴标签

- 不展示风险分
- 不生成全校风险榜
- 不在学生档案留下永久风险标识
- 任务关闭后，从学生日常视图消失，仅在审计后台保留

### 4.3 AI 不参与判定

AI 可以：

- 解释规则为什么触发
- 生成中性沟通话题
- 整理谈话纪要
- 摘要群体趋势

AI 不可以：

- 判定学生有风险
- 做心理诊断或情绪定性
- 决定是否联系家长、上报或转介
- 自动外发通知给学生或家长
- 直接修改任务状态

### 4.4 必须闭环

任务不能停留在“提示过”。每条任务必须进入以下终态之一：

- 已完成
- 已拒绝
- 已转介

改期不是终态。改期次数达到阈值后，任务自动进入院系领导视图。

### 4.5 敏感数据最小化

| 数据类型 | 规则使用 | AI 使用 | 展示 |
|---|---|---|---|
| 学籍、班级、年级、专业 | 可以 | 可以 | 可以 |
| 出勤、请假、成绩、违纪 | 可以 | 可以 | 可以 |
| 勤工助学申请和履职事件 | 可以 | 可以使用中性字段 | 可以 |
| 欠费、资助申请 | P1 暂不接入规则；P2 评审后接入 | 不可以 | 仅限有权限角色 |
| 一卡通消费 | P2 再评审 | 不可以 | P1 不展示 |
| 心理咨询、医疗、家庭情况 | 不可以 | 不可以 | P1 不展示 |
| 谈话自由文本 | 不用于规则 | 仅用于纪要整理，且用户主动触发 | 只在任务详情和审计中授权查看 |

## 5. 典型场景

### 5.1 辅导员早晨 5 分钟

辅导员 8:30 打开“主动关怀”工作台，看到今日 5 个任务：

- 张三：近 5 天缺课 3 次，建议 24 小时内主动联系
- 李四：近 14 天夜不归宿 2 次，建议 24 小时内主动联系
- 王五：勤工岗位纪律类离岗，建议 24 小时内主动联系

辅导员点开张三任务，看到：

- 触发证据：具体课程、日期、缺课记录
- AI 解释：该任务由连续缺课规则触发，依据为近 5 天累计缺课 3 次
- 沟通话题：身体状态、作息变化、课程适配度
- 避免话题：家庭经济、感情状况
- 可对接资源：学业导师、学习帮扶小组

辅导员接单并完成谈话后，录入纪要，任务关闭。

### 5.2 院系领导周一查看

院系领导打开本院视图，看到：

- 本周新增关怀任务 28 项
- 已完成 22 项
- 进行中 5 项
- 超期 1 项
- 触发最多的规则：连续缺课、夜不归宿、长期无跟进

超期任务展示学生姓名和任务类型，但不展示责任辅导员姓名。领导可点击“督办”，系统通过通知中心私下提醒责任辅导员，领导界面只显示“已督办”。

### 5.3 突发事件回溯

突发事件发生时，授权人员可通过下钻查看该学生近 90 天关怀任务摘要和审计记录。下钻必须填写不少于 30 字理由，并写入审计。

系统不展示“被下钻”信息给学生本人，避免对学生形成反向压力。

## 6. 产品视图

### 6.1 辅导员关怀工作台

首屏内容：

- 今日待处理任务，最多展示 5 个
- 已接进行中
- 本周已完成
- 超期任务

任务卡信息：

- 学生姓名、班级
- 一句话触发摘要
- 严重度标签：紧急 / 高 / 关注 / 提醒
- SLA
- AI 解释区域
- 接单、改期、详情按钮

设计约束：

- 不展示风险分
- 不展示永久标签
- AI 区默认展开，但允许用户设置默认折叠
- 触发证据默认折叠，详情页可展开

> 首屏 wireframe、区块上限、字段映射 → [docs/W1-信息架构与任务卡.md](docs/W1-信息架构与任务卡.md) §2

### 6.2 院系 / 学校管理视图

默认只看汇总：

- 任务总数
- 完成数
- 进行中数
- 超期数
- 高发规则
- 严重度分布
- 周 / 月趋势

允许展示的个体入口：

- 超期任务
- 明确下钻理由后的学生档案

禁止展示：

- 辅导员效能排行
- 学生风险排行榜
- 单个辅导员任务积压点名建议
- 单个班级或单个学生的系统建议

系统建议只能基于聚合维度生成：

- 全院 / 全校
- 规则
- 严重度
- 时间窗口

### 6.3 规则运维视图

P1 学校侧只提供：

- 规则列表
- 单条规则启停
- 全局严重度偏移：-1 / 0 / +1
- 30 天效果报表
- 当前规则集版本
- 下次预计更新日期

P1 不提供：

- 学校侧规则创建
- 学校侧 DSL 编辑
- 学校侧 shadow / canary
- 学校侧规则版本回滚
- 学校侧阈值调整

## 7. 核心业务流程

```text
业务系统产生事件
  -> 写入 student_event_log
  -> 规则引擎评估
  -> 生成 care_task
  -> AI 批量预生成 task_ai_brief_history
  -> 通知中心分层触达
  -> 辅导员接单 / 改期 / 拒绝 / 转介 / 完成
  -> 写入 care_task_audit 和 care_task_feedback
  -> 效果报表与匿名反馈回流
```

事件写入失败不能阻塞原业务事务。事件流是观测层和治理层，不是业务主流程的强依赖。

## 8. 事件中心

### 8.1 事件来源

P1 先接入已有或短期可得的数据源：

| 来源 | 事件类型 | 信号含义 |
|---|---|---|
| 请假 | `leave_overdue` | 请假到期未销假 |
| 考勤 | `course_absence`、`checkin_late` | 课堂出勤异常 |
| 违纪 | `violation_recorded` | 行为问题 |
| 成绩 | `exam_failed`、`grade_drop` | 学业风险 |
| 宿舍 / 查寝 | `dorm_late_return`、`dorm_check_absent` | 晚归、夜不归宿 |
| 谈话记录 | `counselor_talk_recorded` | 跟进基线 |
| 勤工助学 | `workstudy_offboarded`、`workstudy_no_show`、`workstudy_apply_rejected`、`workstudy_timesheet_disputed` | 履职异常和隐性经济压力信号 |

### 8.2 勤工助学事件

| 事件 | 触发动作 | severity | 说明 |
|---|---|---|---|
| `workstudy_offboarded` | 勤工申请 `engagement_status` 变为 `offboarded` | 0-7 动态 | 由离岗原因派生 |
| `workstudy_no_show` | 有上岗周期但工时为 0 | 6 | P1 可先不做调度器，作为 P2 增量 |
| `workstudy_apply_rejected` | 申请被雇主驳回 | 3 | 单次不触发，30 天累计用于 R012 |
| `workstudy_timesheet_disputed` | 工时表进入 disputed | 4 | 反映履职冲突 |

离岗 severity 派生：

| offboard_reason | dismissal_category | severity | 是否入规则 |
|---|---|---|---|
| `completed` | - | 0 | 否 |
| `resigned_by_student` | - | 2 | 否 |
| `terminated_by_employer` | `position_dissolved` | 2 | 否 |
| `terminated_by_employer` | `mismatch` | 2 | 否 |
| `terminated_by_employer` | `performance` | 5 | 是，关注 |
| `terminated_by_employer` | `other` | 5 | 是，关注 |
| `terminated_by_employer` | `discipline` | 7 | 是，紧急 |
| `terminated_by_employer` | NULL | 5 | 是，关注兜底 |

`dismissal_category` 由雇主在业务表单中选择，不由 AI 后置分类。

### 8.3 事件存储

主表：`student_event_log`

| 字段 | 类型 | 说明 |
|---|---|---|
| id | bigint PK | 主键 |
| tenant_id | varchar | 租户 |
| student_id | bigint | 学生 |
| event_type | varchar | 事件类型 |
| event_source | varchar | 来源模块 |
| source_id | bigint | 来源业务记录 |
| event_data | jsonb | 结构化字段，不存姓名、身份证、自由文本 |
| severity | int | 0-10 |
| occurred_at | timestamptz | 事件发生时间 |
| created_at | timestamptz | 写入时间 |

索引：

- `(tenant_id, student_id, occurred_at desc)`
- `(tenant_id, event_type, occurred_at desc)`
- `(tenant_id, severity, occurred_at desc) where severity >= 5`

事件保留：在校期间全量保留；毕业 / 离校后归档到只读区，不再参与规则评估。

## 9. 规则引擎

### 9.1 规则归属

P1 规则由产品方维护，作为版本化产品能力发布。学校只能：

- 启停单条规则
- 设置全局严重度偏移
- 查看规则效果报表

学校不能：

- 修改阈值
- 修改 DSL
- 修改冷却期
- 修改 SLA
- 新建规则
- 回滚规则版本

### 9.2 P1 内置规则

P1 上线 13 个 rule_id：

| ID | 名称 | 条件 | 严重度 | SLA | 冷却 | 类别 |

| R002 | 挂科叠加 | 本学期挂科不少于 3 门 | 关注 | 7d | 30d | 学业 |
| R003 | 阶段成绩下降 | 最近 2 次阶段成绩连续下降不少于 15% | 关注 | 7d | 30d | 学业 |
| R004 | 晚归异常 | 近 14 天晚归不少于 3 次 | 提醒 | 14d | 7d | 行为 |
| R005 | 夜不归宿 | 近 14 天夜不归宿不少于 2 次 | 紧急 | 24h | 7d | 行为 |
| R006 | 重复违纪 | 近 30 天违纪不少于 2 次 | 紧急 | 24h | 14d | 行为 |
| R007 | 请假超期 | 请假到期 48h 未销假 | 高 | 48h | 3d | 生活 |
| R008 | 长期无跟进 | 近 60 天无谈话记录，且存在历史关怀任务 | 提醒 | 14d | 30d | 生活 |
| R009 | 多模块异常 | 30 天内命中不少于 3 类异常事件 | 紧急 | 24h | 14d | 跨类 |
| R010 | 趋势性下滑 | 学业成绩下滑且课堂出勤低于 70% | 关注 | 7d | 30d | 跨类 |
| R011a | 勤工履职异常（纪律） | 30 天内纪律类离岗，或 `workstudy_no_show` 不少于 2 次 | 紧急 | 24h | 14d | 勤工 |
| R011b | 勤工履职异常（表现） | 30 天内表现类 / 其他类离岗 | 关注 | 7d | 30d | 勤工 |
| R012 | 隐性经济压力 | 30 天内勤工申请被拒不少于 3 次且无成功上岗 | 关注 | 7d | 30d | 勤工 |

说明：

- `workstudy_no_show` 如果 P1 调度器赶不上，可在灰度时关闭 R011a 的该子条件，但保留 rule_id。
- R012 不向 AI 输入家庭、资助、欠费字段。
- 所有关怀文案使用中性表达，不出现“风险学生”“问题学生”等词。

### 10.2 状态机

```text
pending
  -> accepted
  -> in_progress
  -> resolved

pending / accepted
  -> rejected

pending / accepted / in_progress
  -> transferred

pending / accepted
  -> pending, reschedule_count + 1

pending / accepted
  -> overdue, when due_at exceeded
```

规则：

- `resolved`、`rejected`、`transferred` 为终态
- `reschedule_count >= 2` 自动进入院系领导“需要介入”视图
- `overdue` 不等于关闭，仍需辅导员或授权人处理
- 所有状态变化写入 `care_task_audit`

## 11. AI 解释与建议

### 11.1 触发时机

AI brief 采用批量预生成 + 懒加载兜底：

| 时间 | 范围 | 用途 |
|---|---|---|
| 06:00 | 所有 `due_at < now + 24h` 且无有效 brief 的任务 | 覆盖夜间规则扫描结果 |
| 08:00 | 06:00 后新增任务 | 覆盖上班前新增事件 |
| 13:00 | 08:00 后新增任务 | 覆盖上午新增事件 |
| 用户打开详情 | 当前 brief 缺失或生成失败 | 懒加载兜底 |
| 用户点击重新分析 | 当前任务，5 分钟限流 | 手动刷新 |

### 11.2 输入

允许输入：

- 任务触发证据
- 学生基础学籍信息
- 最近 30 天结构化事件
- 已关闭关怀历史摘要

禁止输入：

- 心理咨询内容
- 医疗信息
- 家庭情况
- 身份证号
- 自由文本谈话原文，除非用户主动触发纪要整理

### 11.3 输出 schema

```json
{
  "why": "为什么触发此关怀，只陈述事实",
  "talking_points": ["中性话题"],
  "avoid_topics": ["本次不宜触碰的话题"],
  "campus_resources": ["可对接资源"],
  "follow_up_days": 7
}
```

输出要求：

- 不使用“疑似”“高危”“心理风险”等诊断性词汇
- 不使用“必须”“应当”“请立即”等命令式表达
- 不预设辅导员行动顺序
- 不写“该生需要”
- 不生成联系家长、转介心理、上报等决策建议

### 11.4 brief 留存

表：`task_ai_brief_history`

| 字段 | 类型 | 说明 |
|---|---|---|
| id | bigint PK | 主键 |
| task_id | bigint | 任务 |
| brief | jsonb | 完整 AI 输出 |
| generated_at | timestamptz | 生成时间 |
| generation_trigger | varchar | batch_06 / batch_08 / batch_13 / lazy / manual_refresh |
| prompt_version | varchar | Prompt 版本 |
| llm_model | varchar | 模型 |
| sanitize_result | varchar | pass / blocked / redacted |

规则：

- append-only，不覆盖
- `care_task.current_brief_id` 指向最新可用版本
- 任务关闭后归档，不直接删除
- 保留期至少与任务审计一致

### 11.5 降级

- AI sidecar 故障：任务照常展示触发证据和操作按钮
- 单条生成失败：打开详情时懒加载重试
- sanitize 不通过：展示“建议生成失败”，不展示违规内容
- 超时：5 秒内失败则进入降级，不阻塞业务动作

## 12. 通知扇出

### 12.1 项目铁律

本模块所有通知必须遵守：

1. Flyway migration 里 seed `notification_template`
2. 业务侧调用 `NotificationOrchestrator.send(code, sourceType, sourceId, recipients, vars)`

禁止：

- 直接调用 `NotificationService.send`
- 直接调用短信、企微、小程序客户端
- 业务侧拼 title/body 字符串发送

统一参数：

- `sourceType = care_task`
- `sourceId = care_task.id`

### 12.2 通知模板

P1 必须 seed 4 条模板：

| code | 场景 | 触达对象 | 内容约束 |
|---|---|---|---|
| `care_task_high_immediate` | high 任务即时提醒 | 责任辅导员 | 不含学生姓名、规则名、判断词 |
| `care_task_daily_digest` | medium 日聚合 | 责任辅导员 | 只写数量和入口 |
| `care_task_critical_dashboard` | critical 看板提示 | 值守看板 / 站内 | 不自动外发企微 |
| `care_task_urge` | 领导督办 | 责任辅导员 | 不暴露领导界面上的责任人姓名 |

模板示例：

```text
care_task_high_immediate
标题：您有 1 项主动关怀任务
正文：您有 1 项需在 48 小时内跟进的主动关怀任务，请登录主动关怀工作台查看。

care_task_daily_digest
标题：今日主动关怀清单
正文：您今日有 {{n}} 项主动关怀任务，请登录主动关怀工作台查看。

care_task_critical_dashboard
标题：紧急关怀线索待研判
正文：{{college_name}} 当前有 {{n}} 项紧急关怀线索，请值守人员登录看板查看。

care_task_urge
标题：您有 1 项主动关怀任务待处理
正文：该任务已进入督办，请尽快登录主动关怀工作台处理。
```

### 12.3 通知策略

| 严重度 | 策略 |
|---|---|
| critical | 站内 + 看板，不自动外发企微 |
| high | 企微即时 + 站内 |
| medium | 每日 09:00 企微聚合 + 站内 |
| low | 仅站内 |

补充规则：

- 节假日不推 medium 日聚合
- 单辅导员当日任务数超过 5 项时自动合并摘要
- 用户 quiet hours 对 high 生效，对站内待办不生效
- critical 的外部升级必须由人工触发

### 13.1 审计

表：`care_task_audit`

| 字段 | 类型 | 说明 |
|---|---|---|
| id | bigint PK | 主键 |
| task_id | bigint | 任务，可为空用于学生下钻类审计 |
| actor_id | bigint | 操作人 |

action 包括：

- viewed
- accepted
- rescheduled

- rejected
- transferred

- notified
- urged
- drilled_down

审计保留 5 年。到期后按学校合同和数据保留策略归档或删除。

### 13.2 领导下钻

默认额度：

| 角色 | 默认额度 |
|---|---|
| 辅导员 | 不算下钻 |
| 院系学工领导 | 20 / 天 |
| 学校学工管理者 | 50 / 天 |
| 校领导 | 不限，但全量审计 |

强制门槛：

- 必填不少于 30 字理由
- 写入 `care_task_audit.action = drilled_down`
- 额度达到 90% 时提示
- 超额拒绝，并提示紧急情况可申请更高权限人员协助

异常审计：

- 任意角色单周下钻超过自身 4 周均值 3 倍，进入异常审计
- 30 天内同一学生被同一用户下钻不少于 5 次，进入高频审计
- 每周一 09:00 向学工部部长推送异常摘要，不推全量日志

### 13.3 学生只读查询

API：`GET /api/v1/care/me/active-summary`

**入口位置**（产品决策，避免反向引发学生焦虑）：

- 路径：小程序"我的" → "个人信息保护" → "我的关怀记录"
- **不主推**：不在小程序首屏、不弹窗、不在消息中心提示
- **不主动通知**：被列入关怀任务时不发学生侧通知
- 仅作为"知情权兜底"满足合规：学生主动查询时能查到，但不引导查询

学生端展示：

- 当前未关闭关怀的类型摘要：学业 / 行为 / 生活 / 勤工助学
- 全部关闭时展示：“目前无主动关心记录”
- 每条未关闭记录文案：“你当前有 1 项学业类的主动关心，老师可能近期联系你”

学生端不展示：

- 历史总次数
- 闭环总数
- 规则名称
- 严重度
- AI 解释
- 辅导员姓名
- 被谁查看或下钻
- 具体触发证据

权限：

- 后端强制 `student_id = current_user.id`
- 不允许通过传 studentId 查询他人
- 任何越权返回 403

## 14. 反馈回流与噪声治理

### 14.1 学校内反馈

规则效果报表展示：

- 触发数
- 接单率
- 完成率
- 平均闭环时长
- 拒绝原因分布
- 误报反馈率

治理提示：

| 条件 | 提示 |
|---|---|
| 单规则 30 天误报率超过 20% | 提示产品方规则维护小组复核 |
| 单规则 30 天“已私下处理”超过 30% | 提示阈值可能偏紧 |

### 14.2 跨校反馈回流

P1 不引入 ClickHouse。

回流机制：

- 学校侧默认 opt-in，可在后台关闭
- 每周一 02:00 批量导出脱敏反馈汇总
- 产品方接收后落 PostgreSQL 汇总库
- 只用于规则迭代和匿名汇总报告

导出前删除：

- student_id
- counselor_id
- 自由文本 note / reason_detail
- 身份证、姓名等直接标识

保留字段：

- rule_id
- rule_version
- severity
- feedback_type
- reason_code
- time_to_close_hours
- 匿名 tenant 哈希

合规要求：

- 合同明确数据用途
- 学校可随时关闭
- 产品方不向第三方共享
- 每季度向学校发布匿名汇总规则变更说明

- 接单

拒绝：

- 必选原因
- 选填说明
- 写入 feedback

改期：

- 可选 1 / 3 / 7 天
- 自动延长 due_at
- 次数 +1

转介：

- 选择目标部门：资助办 / 心理中心 / 教务办 / 保卫处

- 转介给心理中心时，辅导员不再看到心理处理细节

首屏元素：

- 本周汇总
- 需要介入的超期任务
- 触发最多规则 top 3
- 聚合管理建议

不出现：

- 辅导员排行榜
- 学生风险榜
- 单个辅导员点名建议
- 个体学生详细档案入口，除非走下钻

| 单规则误报率 | 不超过 20% |
| 辅导员 NPS | 不少于 30 |
| 产品方规则复盘 | 每月至少 1 次 |
| Prompt eval | 每月至少 1 次 |
| 异常下钻审计 | 每周自动生成 |

## 18. 实施路径

P1 预计 8 周。

| 周 | 主题 | 产物 |
|---|---|---|
| W1 | 信息架构与任务卡 | 工作台首页、任务详情、状态机设计 |
| W2 | 事件流和内置规则 | `student_event_log`、13 个 rule_id、任务生成 |
| W3 | AI brief 和降级 | 批量预生成、history 表、sanitize、懒加载 |
| W4 | 通知和闭环 | notification_template seed、Orchestrator 调用、反馈入口、**任务卡 / 旧 alert 列表切换 toggle**（UAT 阶段降级渠道，上线后保留至少 1 个月）|
| W5 | 管理视图和下钻 | 汇总、超期、督办、下钻额度、审计 |
| W6 | 学生侧和规则设置 | active-summary、规则启停、严重度偏移 |
| W7 | UAT 招募 + 合规走查 | 联系学院、协调 3-5 名辅导员、培训物料（视频 + cheatsheet）；同周并行：文案走查、法务 / 学工业务专家 / 心理中心（如涉及）签字 |
| W8 | UAT 观察 + 灰度上线 | 1 周观察期、每日反馈访谈、prompt eval 回归；1 个学院灰度运行；监控 + 上线 |

### 18.1 不可压缩工期底线

下列环节都被评审过认定为"不可压缩"，工程提速时不能从这里挤时间：

| 项 | 不可压缩原因 |
|---|---|
| W1 prompt eval 一轮过不去 | 业务专家评分不达标即合规底线未过；prompt 调优本身是反复试错过程，必须留 buffer |
| W6 学生端合规走查 | 文案直接面向学生，错一个词就是舆情；不能等 W8 才走查 |
| W7 + W8 UAT 完整 2 周 | 从招募到辅导员真的用起来需要培训 + 信任建立；1 周打不出真实反馈 |
| W8 灰度 1 周不出 P0 / P1 | 不灰度直接全量 = 上线即翻车，且 critical 通道不外发的设计需要看板侧观察验证 |
| 全程"切回旧视图" toggle | 上线后保留至少 1 个月，给"用户接受度不达预期"留退路 |

**项目管理立场**：以上 5 项任何一项被压时间，必须经过产品 + 学工业务专家书面同意。

## 19. P2 / P3 边界

P2 可考虑：

- 完整学生申诉流程
- 心理中心独立线索箱
- 经济类规则接入
- 一卡通消费异常
- 学校侧高级规则治理，仅对有运维能力的高校开放
- 月报 / 学期报

P3 可考虑：

- 群体基线对比
- 无监督异常检测
- 规则效果自动调优建议
- 历史案例回溯

## 20. 风险登记

| 风险 | 影响 | 缓解 |
|---|---|---|
| AI 建议被当成决策 | 责任事故 | 中性 prompt、sanitize、免责声明、AI 不改状态 |
| 通知文案泄露隐私 | 舆情 | 模板不含学生姓名、规则名、判断词 |
| 管理层把系统当绩效工具 | 辅导员抵触 | 禁止排行榜，建议只做聚合 |
| 规则噪声过大 | 使用率下降 | 灰度、误报反馈、产品方月度复盘 |
| 下钻被滥用 | 隐私风险 | 理由、额度、异常审计 |
| 学生看到数字产生焦虑 | 信任受损 | 学生侧只显示当前类型摘要 |
| 新增基础设施拖慢交付 | 工期风险 | P1 不引入 ClickHouse，先用 PostgreSQL 汇总库 |

## 21. 术语

| 术语 | 说明 |
|---|---|
| 关怀任务 | 由规则生成、需要辅导员主动跟进的任务 |
| 触发证据 | 任务生成时命中的事件和阈值快照 |
| AI brief | AI 生成的事实解释和中性沟通提示 |
| 下钻 | 管理者基于具体理由查看个体学生档案 |
| 闭环 | 任务进入完成、拒绝或转介终态 |
| 规则偏移 | 学校对全局严重度做 -1 / 0 / +1 调整 |

## 22. 最终定位

主动关怀工作台不回答“谁是高风险学生”，它回答：

> 今天哪些明确线索值得老师主动看一眼，应该由谁跟进，如何合规地记录和闭环。

系统的价值不是制造更多预警，而是让主动关怀真正发生、可被审计、可被改进。
