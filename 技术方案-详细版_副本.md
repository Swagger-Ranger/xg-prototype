# 高校学生工作管理工作流引擎 — 技术方案（详细版）

## 一、项目概述

### 1.1 项目定位

为高校学生工作管理场景构建一套**通用工作流引擎**。引擎本身只有一套，通过配置文件驱动不同的业务流程。支持两种配置方式：

- **AI Agent**：用户用自然语言描述需求，Agent 做语义理解后生成配置文件
- **可视化设计器**：用户通过拖拽和表单界面从零配置，或对 Agent 生成的配置进行二次修改

两种方式输出的是**同一种格式的配置文件（YAML/JSON）**，引擎统一解析执行。

### 1.2 核心挑战

| 挑战 | 说明 |
|------|------|
| 动态表单存储 | 不同流程的表单字段完全不同，需灵活存储且支持高效查询 |
| 数据关联 | 表单字段（如学号）需关联学校已有业务数据（如学生信息表） |
| 跨流程整合 | 多个相关流程之间存在数据依赖和流转关系（如勤工助学的发布→申请→结算） |
| 配置驱动执行 | 引擎需要能解析 AI Agent 或可视化设计器输出的配置文件并驱动流程运行 |

## 三、技术选型

### 3.1 后端主服务：NestJS (TypeScript)

**选型理由：**

- 工作流引擎的核心工作是**动态配置解析 + JSON 数据流转**，TypeScript 操作 JSON 零摩擦
- 前后端统一 TypeScript，流程定义和表单定义的类型可前后端共享
- 配合 Prisma 操作 PostgreSQL JSONB 非常自然
- 高校场景并发量有限，Node.js 性能完全满足

**与备选方案对比：**

| 对比项 | NestJS (Node.js) | Spring Boot (Java) | Go (Gin) |
|--------|-------------------|---------------------|-----------|
| JSON 处理 | 天然亲和 | 需序列化/反序列化 | 需 marshal |
| 动态表单适配 | 极佳 | 强类型反而是负担 | 一般 |
| 全栈统一 | 前后端同语言 | 不能 | 不能 |
| 生态 | xstate 状态机可直接用 | Flowable 成熟但重 | 需自建 |

### 3.2 前端：Next.js + Ant Design + AntV X6

| 组件 | 选型 | 职责 |
|------|------|------|
| 框架 | Next.js (App Router) | SSR、文件路由、BFF 层 |
| UI 组件库 | Ant Design 5.x | 表单、表格、审批等中后台组件 |
| 流程设计器 | AntV X6 | 可视化流程拖拽编辑 |
| 动态表单渲染 | Formily | JSON Schema 驱动表单渲染，与我们的表单定义天然匹配 |
| 状态管理 | Zustand | 流程设计器等复杂交互状态管理 |

**流程设计器选型对比：**

| 方案 | 适配度 | 理由 |
|------|--------|------|
| **AntV X6** | 最高 | 轻量、高度可定制、中文文档好，适合自定义 DSL |
| bpmn.js | 低 | 绑定 BPMN 标准，定制困难 |
| React Flow | 中 | 偏展示，编辑能力弱 |

选 AntV X6 的原因：我们的流程定义是自定义 DSL，不遵循 BPMN 标准，X6 的自由度最高。

### 3.3 AI Agent 服务：Python + FastAPI + LangChain

| 选型 | 理由 |
|------|------|
| Python | AI/LLM 生态最成熟 |
| FastAPI | 异步高性能，自动 OpenAPI 文档 |
| LangChain | 结构化输出（JSON 配置生成）、Prompt 管理 |
| 大模型 | Claude API（结构化输出质量高） |

Agent 作为独立微服务，与主服务通过 REST API 通信。主引擎完全不依赖 AI，AI 服务挂掉不影响核心流程运转。

### 3.4 工作流引擎内核：自建（xstate 状态机驱动）

**为什么不用现成引擎：**

| 现有引擎 | 问题 |
|----------|------|
| Flowable / Camunda | 吃 BPMN XML，要加配置转换层；动态表单、业务对象、ref 关联在 BPMN 中无原生支持，削足适履 |
| Temporal | 代码驱动流程定义，与"配置驱动"需求冲突 |
| n8n / Node-RED | 偏自动化集成，不是审批流场景 |

**为什么用 xstate 做内核：**

xstate 是 TypeScript 生态最成熟的状态机库，其能力与工作流引擎的需求精确对应：

| 工作流需求 | xstate 能力 |
|-----------|-------------|
| 节点按顺序流转 | state + transition |
| 条件分支（通过/拒绝） | Guards（条件守卫） |
| 等待审批（人工介入） | invoke + 异步事件 |
| 并行会签 | Parallel states |
| 超时自动处理 | After（延迟转换） |
| 子流程 | invoke child machine |
| 持久化与恢复 | 状态序列化/反序列化 |

**关键：用户和 Agent 永远不接触 xstate，它们只操作 YAML/JSON 配置。xstate 是引擎的内部实现细节。**

### 3.5 数据库：PostgreSQL 15+ (JSONB)

**选型理由：**

| 对比方案 | 问题 |
|----------|------|
| 每流程建表（纯关系型） | 表爆炸、DDL 运维风险、改字段需 ALTER TABLE |
| EAV 模型 | 查询极慢、SQL 复杂、报表不可用 |
| MongoDB | 跨流程关联查询弱、事务弱、与学校已有系统集成成本高 |
| **PostgreSQL + JSONB** | **兼顾灵活与结构化，JSONB 支持 GIN 索引，生成列可提取高频字段** |

核心优势：
- 动态表单数据存 JSONB，完全灵活
- 高频查询字段（如学号）通过**生成列（Generated Column）**提取为物理列并建索引
- 完整的事务和 JOIN 能力，天然支持与学校已有数据关联
- 高校信息化系统几乎都是关系型生态，兼容性最好

### 3.6 中间件

| 组件 | 选型 | 用途 |
|------|------|------|
| 缓存 | Redis 7 | 会话管理、待办计数、流程实例状态缓存、分布式锁 |
| 消息队列 | RabbitMQ | 异步通知（短信/微信）、跨流程触发、超时调度 |
| 文件存储 | MinIO | 表单附件，兼容 S3 协议，可私有化部署 |
| ORM | Prisma | 类型安全、JSONB 支持好、自动迁移 |

RabbitMQ 而非 Kafka：高校场景消息量不大，RabbitMQ 的延迟队列天然适合审批超时提醒和流程到期自动关闭。

### 3.7 认证与权限

| 层面 | 方案 |
|------|------|
| 统一认证 | 对接学校 CAS / OAuth2 / LDAP |
| Token | JWT (access + refresh)，Redis 存 session |
| 权限模型 | RBAC + 数据权限（学生、辅导员、院领导、职能部门管理员） |
| 审批权限 | 基于组织架构自动分配（同班辅导员、同院领导） |

## 四、数据库设计

### 4.1 分层结构

```
第一层：流程定义层（元数据）
├── workflow_definition    — 流程定义
├── node_definition        — 节点定义
└── form_definition        — 表单定义（含字段列表、关联规则）

第二层：流程实例层（运行时数据）
├── workflow_instance      — 流程实例
├── form_data              — 表单数据（JSONB）
└── task_instance          — 任务/节点实例（待办、审批记录）

第三层：业务对象层（跨流程共享数据）
├── business_object_definition  — 业务对象定义
└── business_object_data        — 业务对象数据（JSONB）

第四层：基础数据层（学校已有数据）
├── student_info           — 学生信息
├── teacher_info           — 教师信息
├── department             — 部门/院系
└── ...
```

### 4.2 核心表设计

#### workflow_definition（流程定义）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | UUID | 主键 |
| name | VARCHAR | 流程名称 |
| description | TEXT | 描述 |
| version | INT | 版本号 |
| status | ENUM | draft / published / archived |
| config | JSONB | 完整的流程配置（即 YAML 解析后的 JSON） |
| created_by | UUID | 创建人 |
| created_at | TIMESTAMP | 创建时间 |
| updated_at | TIMESTAMP | 更新时间 |

#### form_definition（表单定义）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | UUID | 主键 |
| workflow_definition_id | UUID | 所属流程定义 |
| node_id | VARCHAR | 所属节点 ID |
| fields | JSONB | 字段定义列表 |

**fields 结构示例：**

```json
[
  {
    "key": "student_id",
    "label": "学号",
    "type": "text",
    "required": true,
    "ref": {
      "table": "student_info",
      "column": "student_id",
      "display": "student_name",
      "validate": true,
      "auto_fill": [
        { "source": "student_name", "target": "name" },
        { "source": "grade", "target": "grade" },
        { "source": "class_name", "target": "class" }
      ]
    }
  },
  {
    "key": "name",
    "label": "姓名",
    "type": "text",
    "readonly": true
  },
  {
    "key": "reason",
    "label": "补办原因",
    "type": "select",
    "options": ["遗失", "损坏", "其他"]
  }
]
```

这里的 `ref` 定义解决了**表单字段如何关联外部数据库**的问题：
- `table` + `column`：指定关联的目标表和字段
- `validate`：提交时校验值是否存在于目标表
- `auto_fill`：用户填入学号后，自动查询目标表回填姓名、年级、班级
- `display`：在下拉选择场景中显示的字段

引擎在运行时看到字段有 `ref`，自动处理校验、回填和关联查询，**不需要每个流程单独写关联逻辑**。

#### workflow_instance（流程实例）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | UUID | 主键 |
| workflow_definition_id | UUID | 所属流程定义 |
| definition_version | INT | 发起时的流程版本 |
| status | ENUM | running / completed / rejected / cancelled |
| current_state | JSONB | xstate 状态机的序列化快照（用于持久化和恢复） |
| context | JSONB | 流程上下文数据（跨节点传递） |
| initiated_by | UUID | 发起人 |
| created_at | TIMESTAMP | 发起时间 |
| completed_at | TIMESTAMP | 完成时间 |

#### form_data（表单数据）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | UUID | 主键 |
| workflow_instance_id | UUID | 所属流程实例 |
| form_definition_id | UUID | 所属表单定义 |
| data | JSONB | 实际表单数据 |
| student_id | VARCHAR GENERATED | 从 data->>'student_id' 提取的生成列，可建索引 |
| submitted_by | UUID | 提交人 |
| submitted_at | TIMESTAMP | 提交时间 |

**生成列的作用：** 对"学号"这种跨流程通用的高频关联字段，从 JSONB 提取为物理列并建索引，既保持动态表单的灵活性，又获得关系型查询的性能。

```sql
-- 生成列定义示例
ALTER TABLE form_data
ADD COLUMN student_id VARCHAR(20)
GENERATED ALWAYS AS (data->>'student_id') STORED;

CREATE INDEX idx_form_data_student_id ON form_data(student_id);
```

| 字段 | 类型 | 说明 |
|------|------|------|

| created_at | TIMESTAMP | 创建时间 |

#### business_object_definition（业务对象定义）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | UUID | 主键 |
| name | VARCHAR | 对象名称，如 work_study_position |
| label | VARCHAR | 显示名称，如 "勤工助学岗位" |
| fields | JSONB | 字段定义（同 form_definition 的 fields 结构） |

### 4.3 关联查询示例

**查询某学生所有补办申请及其班级信息：**

```sql
SELECT
  fd.data,
  si.class_name,
  si.grade,
  wi.status,
  wi.created_at
FROM form_data fd
JOIN workflow_instance wi ON fd.workflow_instance_id = wi.id
JOIN student_info si ON fd.student_id = si.student_id
WHERE fd.student_id = '2024001'
  AND wi.workflow_definition_id = '<补办学生证流程ID>';
```

利用生成列 `fd.student_id` 直接 JOIN，无需 JSONB 操作符，性能等同普通关系型查询。

## 五、工作流引擎核心设计

### 5.1 引擎处理流程

```
1. 加载配置
   用户发起流程 → 从 workflow_definition 读取 config

2. 解析配置
   Config Parser 解析 config → 校验节点、表单、关联定义的合法性 → 生成内部流程模型

3. 编译状态机
   Machine Compiler 将内部流程模型编译为 xstate 状态机定义
   每个 node → 一个 state
   每个 next → 一个 transition
   条件分支 → guards
   超时 → after

4. 创建实例
   Runtime Engine 创建状态机实例 → 写入 workflow_instance

5. 执行节点
   进入 state 时调用对应的 Node Executor：
   - FormExecutor：渲染表单、处理 ref 关联和 auto_fill
   - ApprovalExecutor：按规则分配审批人、创建待办
   - TriggerExecutor：操作业务对象 / 触发其他流程
   - NotificationExecutor：发送通知
   - ComputationExecutor：执行计算逻辑

6. 等待与恢复
   遇到需要人工介入的节点（审批）→ 序列化状态机 → 存入 DB → 挂起
   审批人操作 → 从 DB 恢复状态机 → 发送事件 → 继续流转

7. 结束
   到达 end 节点 → 更新 workflow_instance 状态 → 触发通知
```

### 5.2 配置文件格式

以"补办学生证"为完整示例：

```yaml
workflow:
  name: "补办学生证"
  version: 1
  trigger: "manual"

  form:
    fields:
      - key: student_id
        label: 学号
        type: text
        required: true
        ref:
          table: student_info
          column: student_id
          auto_fill:
            - source: student_name
              target: name
            - source: grade
              target: grade
            - source: class_name
              target: class
      - key: name
        label: 姓名
        type: text
        readonly: true
      - key: grade
        label: 年级
        type: text
        readonly: true
      - key: class
        label: 班级
        type: text
        readonly: true
      - key: reason
        label: 补办原因
        type: select
        options: [遗失, 损坏, 其他]

  nodes:
    - id: submit
      type: form_submit
      name: 学生提交申请
      next: counselor_approve

    - id: counselor_approve
      type: approval
      name: 辅导员审批
      assignee:
        role: counselor
        scope: same_class
      timeout:
        duration: 48h
        action: remind
      next:
        approved: card_office
        rejected: end_rejected

    - id: card_office
      type: approval
      name: 一卡通中心确认
      assignee:
        role: card_admin
      next:
        approved: notify_student
        rejected: end_rejected

    - id: notify_student
      type: notification
      name: 通知学生领取
      channel: [sms, wechat]
      template: "您的学生证补办申请已通过，请至一卡通中心领取。"
      next: end_success

    - id: end_success
      type: end
      status: completed

    - id: end_rejected
      type: end
      status: rejected
```

### 5.3 节点类型体系

| 节点类型 | 用途 | 说明 |
|----------|------|------|
| `form_submit` | 表单填写提交 | 发起节点，渲染动态表单，处理 ref 关联 |
| `approval` | 审批 | 支持会签（全部通过）、或签（任一通过）、按角色/层级/组织架构分配 |
| `notification` | 通知 | 短信 / 微信 / 站内信 |
| `condition` | 条件网关 | 根据表单数据或外部数据走不同分支 |
| `trigger` | 触发器 | 跨流程核心——触发另一个流程、创建/更新业务对象 |
| `computation` | 计算节点 | 数据计算（如工资结算 = 工时 × 时薪） |
| `subprocess` | 子流程 | 嵌入另一个完整流程作为子节点 |
| `end` | 结束 | 终止节点，标记最终状态 |

### 5.4 持久化与恢复

审批流程通常需要等待数小时甚至数天，状态机不可能常驻内存：

```
挂起：状态机当前 state + context → 序列化为 JSON → 存入 workflow_instance.current_state
恢复：从 DB 读取 current_state → 反序列化 → 重建状态机实例 → 发送事件继续流转
```

xstate 原生支持状态的序列化（`machine.getPersistedSnapshot()`）和反序列化（`createActor(machine, { snapshot })`），这个能力对工作流引擎至关重要。

## 六、跨流程业务整合

### 6.1 业务场景：勤工助学

勤工助学涉及三个关联流程：

```
流程A：发布岗位    →  创建岗位记录
流程B：岗位申请    →  学生申请 + 审批 → 创建分配记录
流程C：工资结算    →  按分配记录 + 考勤数据 → 计算工资
```

三个流程独立运行，但数据存在依赖关系。

### 6.2 整合机制一：业务对象

**业务对象**是跨流程共享的数据实体，不属于任何单一流程，多个流程共同读写：

```yaml
business_objects:
  - name: work_study_position
    label: 勤工助学岗位
    fields:
      - key: position_id
        type: auto_id
      - key: position_name
        label: 岗位名称
        type: text
      - key: department_id
        label: 用工部门
        type: text
        ref:
          table: department
          column: id
          display: name
      - key: salary_rate
        label: 时薪（元）
        type: number
      - key: quota
        label: 招聘名额
        type: number
      - key: status
        label: 状态
        type: select
        options: [open, filled, closed]

  - name: work_study_assignment
    label: 岗位分配
    fields:
      - key: assignment_id
        type: auto_id
      - key: position_id
        type: text
        ref:
          object: work_study_position
          column: position_id
      - key: student_id
        type: text
        ref:
          table: student_info
          column: student_id
      - key: status
        type: select
        options: [applied, approved, rejected, terminated]
      - key: start_date
        type: date
      - key: end_date
        type: date

  - name: salary_record
    label: 工资记录
    fields:
      - key: record_id
        type: auto_id
      - key: assignment_id
        type: text
        ref:
          object: work_study_assignment
          column: assignment_id
      - key: month
        type: text
      - key: work_hours
        type: number
      - key: amount
        type: number
      - key: status
        type: select
        options: [pending, paid]
```

### 6.3 整合机制二：Trigger 节点（流程间数据写入）

流程 B 审批通过后，通过 trigger 节点自动创建业务对象记录：

```yaml
# 流程B 岗位申请 — 审批通过后的触发节点
- id: create_assignment
  type: trigger
  action: create
  target_object: work_study_assignment
  mapping:
    position_id: $form.position_id
    student_id: $form.student_id
    status: "approved"
    start_date: $form.start_date
  next: notify_student
```

`$form.xxx` 表示引用当前流程的表单数据，引擎自动将值写入业务对象。

### 6.4 整合机制三：数据源引用（流程间数据读取）

流程 C 发起时，表单从业务对象中读取数据作为选项：

```yaml
# 流程C 工资结算 — 表单字段的数据来源是业务对象
form:
  fields:
    - key: position_id
      label: 选择岗位
      type: select
      source:
        object: work_study_position
        filter: { status: "open" }
        display: position_name
    - key: month
      label: 结算月份
      type: month_picker
    - key: students
      label: 结算学生
      type: table
      source:
        object: work_study_assignment
        filter: { position_id: "$form.position_id", status: "approved" }
        columns: [student_id, student_name, start_date]
```

### 6.5 数据流全景

```
          流程A: 发布岗位
          (职能部门发起)
               │
               ▼  trigger: create
        ┌──────────────────┐
        │ work_study_       │
        │ position          │  ← 业务对象
        │ (岗位信息)         │
        └────────┬─────────┘
                 │  source: 读取可申请岗位
                 ▼
          流程B: 岗位申请
          (学生发起)
               │
               ▼  trigger: create
        ┌──────────────────┐
        │ work_study_       │
        │ assignment        │  ← 业务对象
        │ (学生-岗位分配)    │
        └────────┬─────────┘
                 │  source: 读取已分配学生
                 ▼
          流程C: 工资结算
          (职能部门发起)
               │
               ▼  trigger: create
        ┌──────────────────┐
        │ salary_record     │  ← 业务对象
        │ (工资记录)         │
        └──────────────────┘
```

每个流程只关心自己的配置，通过业务对象实现解耦的数据流转。新增流程（如"岗位评价"）只需读写对应的业务对象，不需要修改已有流程。

## 八、技术选型总览

| 层 | 技术 | 核心理由 |
|----|------|---------|
| 前端框架 | Next.js (App Router) | SSR + 文件路由 + BFF |
| UI 组件 | Ant Design 5.x | 动态表单和中后台场景最强 |
| 流程设计器 | AntV X6 | 轻量可定制，自定义 DSL 友好 |
| 动态表单 | Formily | JSON Schema 驱动，天然匹配 |
| 后端框架 | NestJS (TypeScript) | 全栈统一 + JSON 亲和 |
| 工作流内核 | 自建 (xstate) | 直接解析自定义配置，灵活度最高 |
| AI Agent | FastAPI + LangChain | Python AI 生态 + 结构化输出 |
| 数据库 | PostgreSQL 15+ | JSONB + 生成列 + 完整关系能力 |
| 缓存 | Redis 7 | 会话 + 待办 + 状态缓存 |
| 消息队列 | RabbitMQ | 延迟队列 + 异步通知 |
| 文件存储 | MinIO | 私有化 S3 兼容 |
| 工程管理 | pnpm + Turborepo | Monorepo 共享类型 |
| 认证 | CAS / OAuth2 + JWT | 对接高校统一身份认证 |
