# xg1 AI 原生方案（v2，经 5 轮多角色讨论收敛）

> 目的：把 xg1 从「AI 增强（AI 套壳）」推进到「AI 原生 + 可感知降级」。
>
> **5 轮讨论打动率趋势**（辅导员 / 学生代表 / 副院长 / 学工业务专家 四方平均）：
>
> R3 45% → R4 63% → **R5 75.25%**（已到敢签字门槛）

---

## 一、核心原则

**AI 原生 ≠ AI 独裁**。"AI 原生"与"可降级"并不矛盾。矛盾的是把**降级做成无缝等价替代** —— 那等于公开承认 AI 没有独有价值。

> **判定真·AI 原生的唯一硬指标：用户对 AI 离线是否"有感知且有损失"。无感 = 套壳，有感 = 原生。**

---

## 二、分层框架（L1 / L2 / L3）

| 层 | 定义 | 是否依赖 AI | 降级策略 |
|----|------|-----------|----------|
| **L1** 数据 / 合规 / 审计 | 核心数据、审计日志、工作流状态、文件存储 | **绝不依赖** | AI 不能篡改、删除、影响任何合规数据 |
| **L2** 业务 CRUD | 请销假 / 审批 / 通知 / 签到 / 违纪等传统业务 | **不应依赖** | AI 加速（自然语言预填、快速查询），AI 挂了走传统 UI 照常办 |
| **L3** AI 独占价值 | 生成、推理、决策协助、批量智能、个性化 | **必须依赖** | AI 挂了**功能明确 disable + UI 告知用户 + 给出人工备份路径** |

**关键**：L3 的降级**不是偷偷回退到规则**，而是**明确"当前不可用"** —— 让用户意识到 AI 曾经在为他创造价值。

## 三、xg1 当前分层诊断

| 功能 | 代码位置 | 当前层 | 应在层 | 诊断 |
|------|---------|--------|--------|------|
| 6 个 UI-Tools（`open_leave_form` 等） | `xg-ai/app/api/chat.py:46-135` | L2 加速 | L2 | ✅ 接受它是 L2，别包装成 L3 |
| 10 个 `query_*` 工具 | `xg-ai/app/tool/query_tools.py` | L2 加速 | L2 | ✅ 同上 |
| `/insights` 端点 | `xg-ai/app/api/insight.py:255-346` | **伪 L3**（像 L3 其实是 L2） | 应升级真 L3 | ❌ 关掉 AI 辅导员察觉不到 —— 病症 |
| `task_recommendation` 审批建议 | `xg-ai/app/api/task_recommendation.py` | L2+ 润色 | L2+ | ⚠️ 核心决策在 Java 规则引擎，AI 只翻译 |
| `alert_rule_author` 自然语言→DSL | `xg-ai/app/api/agent.py:44-46` | **真 L3 萌芽** | L3 | ✅ 全项目少数真 L3，人工写 DSL 慢 10 倍 |
| `AlertRuleEngine.applyAiHooks` | `xg-backend/.../AlertRuleEngine.java:44-94` | L1+L3（干净） | 保持 | ✅ 全项目最健康的降级姿势，值得复制 |
| RAG pgvector + 关键词回落 | `xg-ai/app/rag/retriever.py:60-72` | L2 问答 | L2 | ✅ 无需改动 |
| `LeaveService` 所有业务 CRUD | `xg-backend/.../LeaveService.java` | L2 纯业务 | L2 | ✅ 零 AI 依赖，正确降级 |

**结论**：除 2 处真 L3 外，**xg1 所有 AI 都在 L2 做加速**。这是"套壳感"的根因。

## 四、真 AI 原生的 7 条验收 checklist

| # | 标准 | 及格线 | xg1 当前 |
|---|------|--------|---------|
| C1 | AI 有**独立数据资产** | ≥3 张 AI 自有表，且 ≥1 张被业务消费 | ❌ 只有 `knowledge_chunk` |
| C2 | **关键路径不可替代** | 停掉 sidecar，≥2 条路径不可用 | ❌ 停掉 AI 业务 100% 正常 |
| C3 | **反馈闭环可度量** | trace_id 贯穿 + 👍👎 + 周采纳率曲线 | ❌ Langfuse 0 行，无反馈写入 |
| C4 | **三类人各有 AI 主动场景** | 管理/执行/被服务侧各 1 个 | ❌ 学生侧 AI 能力池严重偏窄 |
| C5 | **Reflection** 跑在 ≥1 核心场景并留痕 | self-critique 产出可追溯 | ❌ 只有引用校验，不是真 self-critique |
| C6 | Prompt / Tool 作为**产品资产治理** | 版本化 + owner + 变更评审 | ❌ Prompt 硬编码在 `chat.py:17-42` |
| C7 | **降级可感知** | UI 明示 Full / Assist / Offline 三态 | ❌ `chat.py:342` 只返一句错误 |

**0 / 7 合格**。目标：3 个月内 ≥ 4 / 7，6 个月内 6 / 7。

## 五、5 个可复用设计模式

| 模式 | 何时用 | xg1 举例 |
|------|--------|---------|
| **Progressive Enhancement 渐进增强** | 高频基础功能 | 请假表单永远可手填，AI 帮预填 reason |
| **AI-only Capability AI 独占能力** | 人力成本高、个性化维度爆炸 | 280 学生个性化周报；AI 挂了整页灰掉 |
| **Shadow Mode 影子模式** | 高风险 AI 决策 | AlertRule AI severity 先写 `ai_suggested_severity` 不覆盖主字段，采纳率 >80% 2 周再 promote |
| **Explicit Degradation Badge 显式降级徽章** | 所有含 AI 的页面 | `<AIStatusBadge />` 订阅 `/api/v1/ai/health` 三色态 |
| **Human-in-the-Loop Checkpoint 人机协作断点** | AI 产出要对外发声 | 周报 / 申诉回复 AI 起草，人编辑确认才落库 |
| **Feedback-Driven Promotion 反馈驱动晋升** | 新 AI 能力孵化 | trace_id + 👍👎 + 采纳率达阈值自动 L3→L2 |

## 六、3 个 P0 高频 L3 能力（5 轮讨论锁定，v2 核心变更）

> **为什么是 3 个 P0 而不是 13 条清单？** —— 前 4 轮曾扩到 13 条（含迎新/毕业/汇报 PPT/学生画像等），Round 5 收敛到**只做真正高频（日/周级）且合规低风险**的 3 条，其余降级为 P0.5 或推迟。
>
> **筛选标准**：高频（日/周）× 三方共识 × 真 L3（人工慢 10 倍以上）× 业务专家合规通过

### P0-1 · 辅导员谈话前 60 秒背景卡 🥇 首批上线

| 项 | 内容 |
|----|------|
| 频次 | 辅导员**每天 3-5 次**，每次省 8-10 分钟 |
| 输入 | 近 90 天跨模块事件（请假/违纪/签到/谈话）+ 未闭环跟进项 |
| 输出 | **只列事实时间线**（禁"心测分数/画像标签/谈话策略建议"） |
| 辅导员共识 | Round 5 排第 1：**"每天省 40 分钟"** / 业务专家排第 2 |
| 合规风险 | 低 |

**必须嵌入的红线**：
- 🔒 约谈前 **2 小时告知学生**主题（系统时间戳审计）
- 🔒 学生侧 UI 有"**查看本次 briefing**"入口（老师看到什么学生看到什么）
- 🔒 **禁显示字段**：心测 SCL-90 分数 / 画像 severity 标签 / 性格评价措辞
- 🔒 AI 产出带"**AI 辅助草稿**"水印

**副院长派生数据**：本院每周谈话次数 / 人均密度 / **连续 14 天未被谈话学生占比**

**多校翻车教训（业务专家）**：
> "Z 云" 2023 年同类功能**信息过载 10 个模块塞一屏 + 心测分数直出** → 学生瞥见屏幕 → 家长告到教育厅 → 被迫下线心理字段

**待团队画线**：**"事实时间线"字段白名单**由产品 + 学工处共同定（成绩算事实、排名算不算？待决）

---

### P0-3 · 家长消息回复草稿 🥈 首批上线（高风险但必做，靠硬拦截兜底）

| 项 | 内容 |
|----|------|
| 频次 | 辅导员**每周 5-8 次**晚 10 点场景 |
| 输入 | 学生真实数据（受授权白名单过滤） |
| 输出 | 事实框架 + **三种回复风格供选** —— 文字辅导员自己敲 |
| 辅导员共识 | 情绪劳动砍半；但**最怕 AI 出错** |
| 合规风险 | 中（需 18+ 学生授权白名单 + 严控输出字段） |

**必须嵌入的红线**：
- 🔒 **AI 输出禁止项**：评价性描述 / 系统数据转述 / 承诺性语句 / 情绪安抚油腻话术
- 🔒 **心理/成绩/处分类关键词强制人工改写**，禁一键发送
- 🔒 草稿页顶部红字："**AI 生成，发送前请核对**"
- 🔒 白名单**按事件粒度或学期续签**（**拒绝一揽子永久授权**，学生代表硬要求）
- 🔒 草稿生成时**直接标红"未授权字段已屏蔽"**（避免辅导员晚 10 点去系统查授权，辅导员硬要求）
- 🔒 每学期开学**二次确认授权** + 学生可**单方面撤销** + 撤销后历史 **T+1 匿名化**
- 🔒 发送记录保留：**原始草稿 + 修改 diff + 辅导员工号**，留痕 **3 年**

**副院长派生数据**：家长咨询 **TOP 5 话题** + 高频投诉辅导员匿名分布

**事故想定（副院长 Round 5 提出）**：
> 辅导员赶时间一键发 AI 草稿 "该生心理状态稳定" → 次日学生跳楼 → "你们是不是早就知道？AI 凭什么判断心理稳定？"
>
> **必须靠关键词强制拦截 + 禁一键发送**

### P0-2 · 口述录音转结构化日志 🥉 **灰度邀请制**，不进首批

| 项 | 内容 |
|----|------|
| 频次 | 辅导员**每天谈话后** |
| 输入 | 辅导员开车路上 2 分钟口述 |
| 输出 | 规范跟进日志 + 自动创建待办 + 识别未闭环 |
| 辅导员共识 | "防出错"核心维度 |
| 合规风险 | **高**（《个保法》第 28 条敏感信息处理） |

**业务专家为什么建议 P0-2 不进首批**：
> "H 平台" 2023 年同类功能**默认开录音 + 未告知学生** → 学生事后发现 → 被认定违反《个保法》第 28 条
>
> 建议：**首批只推 P0-1 + P0-3**；P0-2 做**灰度邀请制**，只给**带生比 > 1:300 的普通本科**试点 3 个月，跑通再全量 —— 签约概率能推到 **80%+，且零舆情风险**

**必须嵌入的红线（灰度期硬约束）**：
- 🔒 **开录音前 10 秒口头告知 + 学生点屏幕确认**（双轨留痕）
- 🔒 **多方对话自动暂停**（检测到多人声纹即提示）
- 🔒 **敏感词强制人工复核**：转写出现"自杀/自残/举报/性"等词 → 草稿不入库 → 转人工
- 🔒 原始音频保留 **不超过 7 天**，转写完成即加密归档
- 🔒 涉及学生的条目学生本人可申请查阅

**副院长派生数据**：**焦虑/失眠/退学/经济困难**关键词词云按月趋势

## 七、待决：学生代表要的独立 L3 能否进 P0？

**学生代表 Round 5 原话**：
> "所有 L3 都是'辅导员用 AI 服务学生'，缺的是 **'学生用工具监督系统'** 那一条。没有它，三条红线全是自证清白，没有外部校验。**这条必须独立，不能捆绑在辅导员工作流里 —— 等于让被告管法官**"

### 她要的：P0.5 · 学生数字画像自查 + 异议通道

- 学生可**随时查看**系统对自己打的所有标签（预警等级、风险类型、被提及关键词）
- 对不准确条目**发起异议** → 辅导员 + 督导**双签回复**
- 加上这条：学生代表从 72% → **82%**

**当前决策：不加为 P0，作为 P0.5 优先进入第二批**。
**妥协方案**：P0-1 的"**学生同步查看权**"本身就是小版本的"自查"雏形 —— 学生能查看针对自己生成的 briefing。作为谈判筹码告诉学生代表：**P0-1 的学生 UI 入口就是数字画像自查的种子**。

## 八、降级可感知 UX 的技术实现

### 8.1 AI 健康协议（新增 `/api/v1/ai/health`）

```json
{
  "level": "full" | "degraded" | "offline",
  "reason": "deepseek_timeout" | "rag_store_down" | null,
  "disabled_features": ["p0_1_briefing", "p0_3_parent_reply", "p0_2_voice_log"],
  "last_ok_at": "2026-04-22T10:12:00Z"
}
```

- Python 侧真实探活（ping LLM + pgvector），Redis 缓存 30s
- Java 侧转发同形状给前端（前端只打 Java 一个入口）

### 8.2 前端统一降级组件

- `<AIStatusBadge />` 全局订阅 health，三色态（绿/黄/灰）
- `<AIOfflineNotice />` 挂到所有含 AI 的页面
- 离线时 banner **必须给出人工备份路径**，不能只说"不可用"

### 8.3 数据层区分来源

所有 AI 产物表新增 `source` 枚举：

```sql
ALTER TABLE ai_generated_artifact ADD COLUMN source VARCHAR(16) NOT NULL DEFAULT 'ai';
-- 'ai' | 'rule' | 'human' | 'hybrid'
```

**前端每张 AI 卡右上角小 tag**："AI 生成" / "规则生成" / "人工编辑过"。

### 8.4 反馈闭环表

```sql
CREATE TABLE ai_feedback (
  id BIGSERIAL PRIMARY KEY,
  trace_id VARCHAR(64) NOT NULL,
  user_id BIGINT NOT NULL,
  kind VARCHAR(32) NOT NULL,  -- 'briefing' | 'parent_reply' | 'voice_log' | ...
  score SMALLINT NOT NULL,    -- -1 (驳回) | 0 (修改) | 1 (采纳)
  edited_text TEXT,
  created_at TIMESTAMPTZ DEFAULT NOW()
);
```

## 十、合规抽屉（业务专家核心建议，v2 新增）

> **业务专家 Round 4 原话**："这份清单最需要的不是加功能，而是做一本可以交给省教育厅的 **AI 使用合规抽屉** —— 每条 L3 对应哪条法规 / 留痕字段 / 人工复核点 / 学生异议路径，打印出来 20 页。**这比再加 5 条功能值钱 10 倍**。"

### 合规抽屉必须包含（初步清单）

| 红线 | 对应法规 | 留痕字段 | 人工复核点 | 学生异议路径 |
|------|---------|---------|-----------|------------|
| P0-1 "只列事实" | 教育部《学生管理规定》第 55 条 | `briefing_id`+生成时间戳+查看学生 ID | 辅导员使用前签字确认 | 学生可申请查看本次 briefing |
| P0-1 "2 小时告知" | 《深化新时代教育评价改革》 | `notify_at` 时间戳 | 系统自动校验 | 超时未告知学生可拒谈 |
| P0-3 "授权白名单" | 《个人信息保护法》第 13/31 条 | `authz_event_id`+学期续签 diff | 每条发送前显式确认 | 学生可单方撤销 + 历史 T+1 匿名化 |
| P0-3 "关键词拦截" | 教育部 AI 应用指引（征求意见稿） | `sensitive_words_hit`+强制改写 diff | 命中即转人工 | 学生可查询历史发送记录 |
| P0-2 "录音告知" | 《个人信息保护法》第 28 条 | `audio_consent_ts`+声纹确认 | 检测多人声纹自动暂停 | 学生可撤销录音同意并申请删除 |
| 全局 "AI 水印" | 省教育厅"AI 辅助人工决定"口径 | 每条 AI 产出带 `source` tag + 辅导员签字 | 辅导员未签字不得入档 | 外界问询有"AI 辅助，人工决策"标准答复 |

**系统登录页和家长告知书必须印一句**：
> **"本系统用 AI 归集事实、生成草稿。所有涉及评价和决策的文字均由辅导员本人签字负责，AI 不对学生下判断。"**

## 十一、每周自检清单

产品 owner 每周例会前 2 分钟走一遍，**任一条中招即亮红灯**：

- [ ] 本周迭代会**没人提 AI 指标**（采纳率 / 改写率 / 离线分钟数）
- [ ] 改一版 prompt **需要走代码发版**
- [ ] 团队里有人说 **"反正 AI 不可用用户也不会发现"**
- [ ] AI 离线 1 小时，**0 个客服工单**
- [ ] 最近 4 周**没有任何功能从 L3 晋升到 L2**
- [ ] `ai_feedback` 表本周写入 **< 人均 1 条**
- [ ] 新需求第一反应是**"加个 CRUD 页面"** 而不是"这能不能只有 AI 能做"
