# 高校学生工作服务系统 — 总体设计文档 v2

> **设计原则：强主干、弱扩展的单体底座。把 AI 和工作流都设计成可拆而不是先拆。**

---

## 一、产品定位

面向高等院校的 **AI 原生学生工作服务平台**。以"平台底座 + 业务模块"架构为核心，为高校学生工作提供全场景、智能化的数字化解决方案。

### 核心理念

- **AI 原生**：系统从第一天起为 AI 参与而设计，AI 是一等交互入口，而非事后附加
- **AI 可降级**：AI 不可用时，所有业务通过传统界面完整运行
- **平台化**：底层能力高度抽象复用，业务模块插件化扩展
- **混合云部署**：公有云托管与私有化部署并存，支持全私有化
- **轻量优先**：能用 PostgreSQL 解决的不加新组件，能不拆的服务先不拆
- **多端触达**：**微信小程序**与**企业微信**均可覆盖**学生与教职工**；同一人在两端的身份需绑定。按**角色与场景**呈现不同能力——例如学生侧重请销假、收集单、签到；辅导员/领导在小程序侧重移动审批、统计与轻量查询，在企微侧重推送摘要、日报与会话式入口（具体菜单与权限由 RBAC 控制，而非「一端只给某一类人」）

### 目标用户

| 角色 | 典型使用场景 |
|------|------------|
| 学生 | 请销假、收集单、签到、知识问答等（**微信小程序**为主办事入口；**企业微信**可同步收通知/催办，视学校是否开通学生企微而定） |
| 辅导员 | 审批、统计、日常事务（**Web + 微信小程序**移动办公；**企业微信**收待办/通知摘要、日报、会话式问数） |
| 院系领导 | 数据概览、审核、报告（**Web + 微信小程序**看板与审批；**企业微信**收日报与预警摘要） |
| 校级管理员 | 全校策略与运维（**Web**为主；**微信小程序**可做移动处置；**企业微信**收通报与任务回执） |
| 超级管理员（平台方） | 租户管理、模块管理、运维监控 |

### P0 业务模块详细说明

#### B. 学生日常事务 — 请销假

请销假是学生日常使用频率最高的功能，需要支持多种假别子类型，每种假别有不同的表单字段和审批规则。

**假别子类型与差异字段：**

| 假别 | 差异字段 | 审批特殊规则 |
|------|---------|-------------|
| 病假（在校） | 需上传校医院证明（附件必填） | 标准审批 |
| 病假（离校） | 需上传校医院证明 + 离校去向 + 紧急联系人 | 辅导员审批后需院系备案 |
| 事假 | 需上传证明材料（附件必填）+ 事由分类 | 3天以上需院系领导加签 |
| 周末离校 | 离校时间（周五/周六）+ 返校时间 + 去向 + 交通方式 | 辅导员审批即可 |

**公共字段：** 请假时间段（开始时间、结束时间）、请假天数（自动计算）、请假原因

**销假：** 学生返校后需主动销假或到期自动提醒销假，辅导员可查看未销假名单

#### H. 信息收集

辅导员日常需要频繁统计学生信息（如国庆去向、寒假留校情况等），需要一套灵活的收集单机制。

**核心功能：**

- **收集单设定（辅导员侧）：** 辅导员可自定义收集表单（标题、说明、字段列表），支持文本/单选/多选/日期/手写签名等字段类型
- **收集单填写（学生侧）：** 学生在**微信小程序**端填写收集单（与角色绑定；若学校为学生开通企微，可同步收到催填等企微消息），支持手写签名组件，填写后可查看自己的提交记录
- **校级任务发布：** 校级管理员可创建信息收集任务，选择下发范围（按学院/年级/班级），自动分发至对应辅导员管辖的学生
- **统计看板：** 辅导员实时查看填报进度（已填/未填），支持一键导出、催填提醒

**典型场景：** 国庆去向统计 → 校级发布任务 → 辅导员转发给管辖学生 → 学生填写（含是否离校、去向、手写签名）→ 辅导员查看汇总

#### I. 签到

用于班会、讲座、活动等场景的出勤管理。

**核心功能：**

- **二维码签到：** 辅导员/活动组织者生成限时二维码，学生扫码签到
- **签到记录：** 自动记录签到时间、地点（可选），辅导员可查看签到/缺勤名单
- **二维码策略：** 支持定时刷新（防截图转发）、限定地理围栏（可选）
- **补签：** 特殊情况辅导员可手动补签

#### J. 通知任务

区别于消息通知中心（底层能力），通知任务是一个面向辅导员和校级管理的**业务模块**，重点在于确保通知触达和已读确认。

**核心功能：**

- **通知分级：** 普通 / 重要 / 紧急，不同级别对应不同的展示和推送策略
- **任务类别：** 省厅任务（如"开学第一课"）、学校通知、学院通知等分类标签
- **附件支持：** 通知可携带附件（文件/图片）
- **强制确认（按接收对象配置）：** 对配置了「须确认」的接收方（常见为学生），重要/紧急通知在**微信小程序**内置顶浮窗展示，**必须点击"确认收到"后才能继续其他操作**；辅导员/领导若也在小程序侧接收同一任务，则按角色展示相应操作（如已读回执、跳转详情），不强求与学生同一套遮罩策略
- **企业微信触达（全员可达）：** 向**学生与教职工**推送摘要、卡片或应用消息（以学校通讯录与绑定关系为准）；卡片可**回链微信小程序**打开完整正文或完成填报/确认（避免在企微内重复实现全部复杂页面）
- **已读追踪：** 发布方（如辅导员）可查看每条通知的确认情况（已确认/未确认名单），支持一键催促

#### K. 接诉即办

学生向校方反映问题的正式渠道，对标"校长信箱"。

**核心功能：**

- **投诉提交：** 学生提交投诉（分类 + 描述 + 可选附件），可选择匿名
- **流转处理：** 投诉提交后按类别分派至对应部门，限时处理
- **进度查看：** 学生可查看投诉处理进度（已受理/处理中/已回复）
- **满意度反馈：** 处理完成后学生可评价满意度

#### L. 辅导员工作日志

辅导员日常工作记录的结构化管理工具。

**核心功能：**

- **分类主题：** 工作日志按类别分类，不同类别有不同的记录模板
- **内置模板：**
  - 谈心谈话：谈话对象、时间地点、谈话内容摘要、后续跟进计划
  - 违纪记录：违纪学生、违纪类型、事实描述、处理意见
  - 班级活动：活动名称、参与人数、活动总结
  - 家校沟通：沟通对象（家长）、沟通方式、沟通内容、沟通结果
  - 自定义模板：辅导员可创建自己的日志模板
- **统计汇总：** 按时间段/类别汇总工作日志，辅助生成工作报告

#### M. 学生信息库

辅导员管理其负责学生信息的核心模块。

**核心功能：**

- **学生全量信息：** 按年级 / 院系 / 班级组织，包括基本信息（姓名、学号、联系方式、照片等）、家庭信息、在校状态
- **多维度筛选：** 按年级、院系、班级、在校状态等条件快速检索
- **信息变更审批：** 学生申请修改个人信息（如联系方式、家庭住址等），辅导员审批后生效
- **数据导入/导出：** 支持 Excel 批量导入学生信息，支持按条件导出

#### N. 勤工助学

覆盖岗位发布 → 学生申请 → 审批录用 → 工资结算的完整流程。

**核心功能：**

- **岗位发布：** 职能部门/学院发布勤工助学岗位（岗位名称、工作内容、工时要求、时薪、招聘人数）
- **岗位申请：** 学生浏览可申请岗位，提交申请（含个人简介），经审批后录用
- **工资结算：** 管理员按月录入/确认学生工时，系统自动计算工资（工时 × 时薪），生成结算单

#### O. 学生违纪

学生违纪行为的记录、分类和处理。

**核心功能：**

- **违纪分类：** 考试违纪、宿舍违规、旷课、打架斗殴、其他（可扩展分类）
- **违纪记录：** 记录违纪学生、时间、地点、事实经过、证据材料（附件）
- **处分流程：** 根据违纪严重程度走不同的处分审批流程（警告/严重警告/记过/留校察看/开除学籍）
- **处分解除：** 到期后可申请解除处分
- **与工作日志联动：** 违纪记录自动关联辅导员工作日志中的违纪记录模板

#### P. 知识问答

基于 AI 的学校政策知识库问答系统。

**核心功能：**

- **知识库管理（管理员侧）：** 录入/导入学校政策文件、规章制度、奖学金评选办法、常见问题等
- **智能问答（学生侧）：** 学生通过自然语言提问，AI 基于知识库（pgvector 语义检索）返回精准答案
- **典型问题：** "国家奖学金的评选条件是什么？" "请假超过几天需要院长审批？" "勤工助学时薪是多少？"
- **答案溯源：** 回答附带原始文件/政策出处，方便学生进一步查阅
- **人工兜底：** AI 无法回答时，引导学生转人工（辅导员或相关部门）

**P0 关键决策：工作流引擎内嵌在 Java 单体中。** 自建轻量状态机（基于 Spring StateMachine 或纯 Java 实现），配置驱动，YAML/JSON DSL 格式保持不变。未来若需独立演进，DSL 格式是稳定的契约，引擎实现可以换语言/框架而不影响上层。

### 各阶段新增组件

| 阶段 | 新增组件 | 触发条件 |
|------|---------|---------|
| P0 | **Langfuse**（自托管） | Agent 可观测：trace、评估、token 用量分析 |
| P1 | **RabbitMQ** | 审批超时调度需要可靠的延迟队列 |
| P1 | **AntV X6 + Formily** | 非技术人员需要可视化配置流程 |
| P2 | **ClickHouse** | 数据分析查询 PG 物化视图扛不住时 |
| P2 | **Milvus** | 知识库规模超出 pgvector 能力时 |
| P2 | **Elasticsearch** | 日志量/搜索复杂度超出 PG tsvector 能力时 |
| P2 | **Spring Cloud Gateway** | 开放平台需要独立的 API 网关时 |

### P0 阶段异步通知和超时调度方案（不用 RabbitMQ）

| 场景 | 方案 |
|------|------|
| 异步通知（短信 / 小程序订阅类消息 / 企业微信应用消息） | Java 线程池异步 + 失败重试表（PG），简单可靠 |
| 审批超时提醒 | 定时任务扫描 task_instance.due_at（Spring @Scheduled，每分钟一次） |
| 流程到期自动关闭 | 同上，定时任务扫描 |

高校场景并发量不大，定时扫描完全够用。P1 阶段流程量增长后，切换为 RabbitMQ 延迟队列，业务代码改动很小。

### 架构核心原则

1. **单体优先**：P0 阶段 Java 单体包含所有业务逻辑和工作流引擎，避免服务边界带来的复杂度
2. **可拆不先拆**：工作流引擎作为内嵌模块，接口清晰，未来可独立拆出为 NestJS 或 Java 微服务
3. **AI 与业务解耦**：Python AI 作为 sidecar 存在，Java 层不依赖它即可完整运行
4. **业务即 Tool（受控）**：不是所有 CRUD 都暴露给 Agent，只开放 30-40 个用户任务级 Tool
5. **轻量优先**：PG 一库多用，不为理论需求预引入重型组件

#### Agent 可观测：Langfuse

Langfuse 自托管部署（Docker），通过 LangChain/LangGraph 原生 callback handler 自动采集，无需手动埋点。

| 采集维度 | 说明 | 用途 |
|----------|------|------|
| Trace | 每次用户对话完整链路（Agent → Tool → LLM） | 问题排查、对话质量回溯 |
| Span / Token | 每步 LLM 调用的 token 数、耗时 | 成本核算、性能瓶颈定位 |
| Score | Tool 执行成功率、用户反馈评分 | Agent 效果评估 |
| Prompt 版本 | Prompt 变更与效果关联 | Prompt 迭代管理 |

**P0 接入方式：**

```python
from langfuse.callback import CallbackHandler

langfuse_handler = CallbackHandler()  # 读取环境变量 LANGFUSE_HOST / LANGFUSE_PUBLIC_KEY / LANGFUSE_SECRET_KEY

# LangChain / LangGraph 调用时传入 callback
agent.invoke({"input": user_message}, config={"callbacks": [langfuse_handler]})
```

**部署：** Langfuse 自托管加入 docker-compose，依赖 PostgreSQL（可复用现有实例，独立 schema）。

### 6.4 工作流 / 审批流引擎

#### P0 定位

工作流引擎 **内嵌在 Java 单体中**，作为一个内部模块（如 `workflow` package），不是独立服务。通过清晰的内部接口设计，确保未来可以拆出。

#### 为什么 P0 不独立部署

| 考虑 | 说明 |
|------|------|
| 减少服务边界 | 少一个服务 = 少一套部署/调试/错误排查/API 对接 |
| 团队成本 | 不需要同时精通 Java 和 TypeScript |
| P0 流程简单 | 5 类节点、手写 YAML 配置，Java 自建状态机完全胜任 |
| 可拆契约 | YAML/JSON DSL 格式是稳定的，未来换引擎实现不影响上层 |

#### 自建状态机（Java 实现）

P0 阶段流程节点类型只有 5 种，状态转换逻辑不复杂，自建轻量状态机比引入 Spring StateMachine 更可控：

```java
public class WorkflowStateMachine {
    // 核心逻辑：根据当前节点类型，执行对应 Executor，然后按 next 配置转到下一个节点
    // 遇到 approval 节点 → 序列化状态到 DB → 挂起
    // 审批完成 → 反序列化恢复 → 继续流转
}
```

| 设计点 | 方案 | 阶段 |
|--------|------|------|
| 引擎内核 | 自建轻量状态机（Java） | P0 |
| 配置格式 | YAML/JSON 配置文件 | P0 |
| 配置方式 | 手写 YAML + AI 辅助生成（P1 加可视化设计器） | P0 |
| 数据存储 | PostgreSQL JSONB（共用数据库，workflow 相关表） | P0 |
| ORM | MyBatis-Plus（与主系统统一） | P0 |

#### P0 节点类型（5 种）

| 节点类型 | 用途 | 说明 |
|----------|------|------|
| `form_submit` | 表单填写提交 | 发起节点，渲染动态表单 |
| `approval` | 审批 | 支持会签/或签，按角色/组织架构分配 |
| `condition` | 条件网关 | 根据表单数据走不同分支 |
| `notification` | 通知 | 站内信 / 微信小程序消息 / 企业微信应用消息 / 短信（扩展） |
| `end` | 结束 | 终止节点，标记最终状态 |

#### P1 新增节点类型（3 种）

| 节点类型 | 用途 | 为什么延后 |
|----------|------|-----------|
| `trigger` | 触发其他流程、创建业务对象 | 跨流程整合是 P1 需求 |
| `computation` | 数据计算（如工时×时薪） | P0 的请假等流程不需要 |
| `subprocess` | 嵌入子流程 | 复杂度高，P0 不需要 |

#### P1 新增能力

| 能力 | 说明 |
|------|------|
| 业务对象（Business Object） | 跨流程共享数据实体 |
| 可视化流程设计器 | AntV X6 拖拽配置 |
| Formily 动态表单 | JSON Schema 驱动表单渲染 |
| AI 生成流程配置 | Agent 自然语言 → YAML 配置 |

#### ref 机制：受控查询接口（非直连业务表）

**P0 不做 cross-schema JOIN。** ref 通过 Java 内部的受控查询接口实现：

```
表单字段声明 ref:
  ref:
    source: student_info
    column: student_id
    auto_fill: [student_name, grade, class_name]

引擎处理流程：
  1. 前端用户输入学号
  2. 前端调用 /api/data-source/student_info?student_id=xxx
  3. Java 层的 DataSourceService 处理：
     - 识别当前租户 → 路由到对应 Schema
     - 检查数据权限（该用户能不能查这条数据）
     - 返回 { student_name, grade, class_name }
  4. 前端自动回填字段
```

**好处：**
- 租户隔离自然实现（DataSourceService 内部处理）
- 数据权限可控（不是引擎直接访问业务表）
- 工作流引擎与业务表解耦（表结构变了只需改 DataSourceService）
- 未来引擎拆成独立服务时，这个接口直接变成 HTTP API，零改动

#### P0 表达式约束

P0 的 DSL 中出现表达式的地方只有三类，均采用**受限语法**，不引入通用脚本引擎：

| 表达式场景 | 支持的操作 | 不支持 | 求值位置 |
|-----------|-----------|--------|---------|
| `visible_when`（字段可见性） | `==`、`!=`、`in [...]`、`and`、`or` | 函数调用、算术、嵌套括号 | 前端 |
| `computed`（字段计算） | 收敛为内置函数，如 `duration_days(start, end)`，不暴露任意表达式 | 自由算术字符串 | 后端 |
| `condition.when`（条件网关） | 同 `visible_when`，加 `>=`、`<=`、`>`、`<` | 函数调用、正则 | 后端 |

P1 引入可视化设计器时，再评估是否需要更丰富的表达式能力；届时统一定义表达式规范，确保跨语言（Java/NestJS）一致性。

#### 配置文件示例（请假流程）

```yaml
workflow:
  name: "学生请假"
  version: 2
  trigger: "manual"

  form:
    fields:
      # --- 公共字段 ---
      - key: student_id
        label: 学号
        type: text
        required: true
        ref:
          source: student_info
          column: student_id
          auto_fill:
            - source: student_name
              target: name
            - source: grade
              target: grade
            - source: class_name
              target: class
      - key: name
        label: 姓名
        type: text
        readonly: true
      - key: leave_type
        label: 假别
        type: select
        required: true
        options:
          - label: 病假（在校）
            value: sick_on_campus
          - label: 病假（离校）
            value: sick_off_campus
          - label: 事假
            value: personal
          - label: 周末离校
            value: weekend_leave
      - key: start_time
        label: 开始时间
        type: datetime
        required: true
      - key: end_time
        label: 结束时间
        type: datetime
        required: true
      - key: duration_days
        label: 请假天数
        type: number
        computed: "ceil((end_time - start_time) / 86400000)"
        readonly: true
      - key: reason
        label: 请假原因
        type: textarea
        required: true

      # --- 病假差异字段 ---
      - key: medical_cert
        label: 校医院证明
        type: attachment
        required: true
        visible_when: "leave_type in ['sick_on_campus', 'sick_off_campus']"

      # --- 离校类差异字段（病假离校 + 周末离校） ---
      - key: destination
        label: 离校去向
        type: text
        required: true
        visible_when: "leave_type in ['sick_off_campus', 'weekend_leave']"
      - key: emergency_contact
        label: 紧急联系人电话
        type: phone
        required: true
        visible_when: "leave_type == 'sick_off_campus'"
      - key: transport
        label: 交通方式
        type: select
        options: [公共交通, 私家车, 其他]
        visible_when: "leave_type == 'weekend_leave'"

      # --- 事假差异字段 ---
      - key: personal_reason_type
        label: 事由分类
        type: select
        options: [家庭事务, 个人事务, 其他]
        required: true
        visible_when: "leave_type == 'personal'"
      - key: supporting_doc
        label: 证明材料
        type: attachment
        required: true
        visible_when: "leave_type == 'personal'"

  nodes:
    - id: submit
      type: form_submit
      name: 学生提交申请
      next: counselor_approve

    - id: counselor_approve
      type: approval
      name: 辅导员审批
      assignee:
        role: counselor
        scope: same_class
      timeout:
        duration: 48h
        action: remind
      next:
        approved: check_need_dean
        rejected: end_rejected

    - id: check_need_dean
      type: condition
      name: 是否需要院系审批
      conditions:
        - when: "leave_type == 'sick_off_campus'"
          next: dean_record
        - when: "leave_type == 'personal' and duration_days >= 3"
          next: dean_approve
        - default: true
          next: notify_approved

    - id: dean_approve
      type: approval
      name: 院系领导审批
      assignee:
        role: dean
        scope: same_department
      timeout:
        duration: 72h
        action: remind
      next:
        approved: notify_approved
        rejected: end_rejected

    - id: dean_record
      type: notification
      name: 院系备案通知
      target:
        role: dean
        scope: same_department
      channel: [in_app, wecom]
      template: "{{name}} 因病假离校，请知悉。"
      next: notify_approved

    - id: notify_approved
      type: notification
      name: 通知学生
      channel: [miniprogram, in_app]
      template: "您的请假申请已通过，请按时返校销假。"
      next: end_success

    - id: end_success
      type: end
      status: completed

    - id: end_rejected
      type: end
      status: rejected
```

#### 持久化与恢复

```
挂起：当前节点 ID + 流程上下文 → JSON → 存入 workflow_instance.current_state
恢复：从 DB 读取 → 反序列化 → 定位到当前节点 → 接收事件继续流转
```

**版本绑定规则（P0）：** workflow_instance 创建时，将当前流程定义完整冻结到 `definition_snapshot`（JSONB 字段）。实例恢复、审批继续时读快照，不读最新配置。这样即使流程 YAML 被修改，挂起中的实例仍按发起时的定义执行。P1 再补充正式的发布/下线/版本迁移机制。

#### 未来拆分路径

```
P0：Java 内嵌模块（workflow package）
  ↓ 如果需要独立演进
P1：拆为独立 Java 服务（Spring Boot，共用 PG）
  ↓ 如果 JSON 处理确实成为痛点
P2：可选迁移到 NestJS（DSL 格式不变，引擎实现替换）
```

**微信小程序载体结论：** 采用**同一套微信小程序（Taro）**，通过**登录身份与 RBAC** 区分**学生端能力集**与**辅导员/领导等管理端能力集**（菜单、路由、页面不同）；并非「小程序仅给学生用」。Web 管理后台与小程序在能力上可互补：重配置、大屏分析偏 Web；外勤审批、签到、收通知偏小程序。

**企微内小程序（可选增量）：** 企业微信支持在工作台打开**企业微信小程序**；若学校希望从企微统一入口跳转，可在 P1 及后评估「企微内打开同一小程序 / 独立企微小程序」方案。P0 以**公众号/开放平台体系下的微信小程序 + 企微应用消息**为主干。

**身份与绑定：** 租户管理员配置本校的企微 CorpId、应用 Secret、AgentId 等；**学生与教职工**均在个人设置中完成「企微 UserID ↔ 平台账号」绑定（OAuth 或管理员预绑定）；未绑定则不接收企微侧推送。

**与消息通知中心的关系：** 通知中心在路由层实现 `wecom` / `miniprogram` 等渠道；业务模块只调用统一发送接口。须强制确认类任务以**微信小程序**落合规交互为主；企微侧可发**摘要 + 打开小程序**路径，触达对象**不限于教职工**。

**与「独立机器人」形态的关系：** 优先走**企业微信自建应用 + 接收消息回调**（或官方群机器人能力）实现对话式入口；避免个人微信外挂协议作为正式交付形态。

## 七、AI 原生架构设计

### 7.1 "AI 原生"的定义

AI 原生 ≠ AI 耦合。含义是：**系统从第一天就按照"有 AI 参与"来设计交互和数据流，AI 是一等公民，但不是唯一公民。**

### 7.2 双路径设计

```
路径一（AI 对话）：
  学生说"帮我请明天一天事假，肚子疼"
    → Agent 解析意图和实体
    → 调用 start_leave_request Tool
    → 返回确认信息

路径二（传统表单）：
  学生打开请假页面，手动填写，点提交
    → 前端调用请假 API
    → 返回提交成功

两条路径调用同一个服务，数据完全一致。
```

### 7.3 Tool 注册机制

```java
@Tool(name = "start_leave_request",
      description = "为当前学生发起请假申请")
public LeaveResult startLeave(
    @Param(desc = "开始日期") String startDate,
    @Param(desc = "结束日期") String endDate,
    @Param(desc = "请假原因") String reason,
    @Param(desc = "假别: 事假/病假/公假") String type) {
    // 内部调用工作流引擎发起流程
    // student_id 从当前登录用户上下文自动获取，不需要 Agent 传入
}
```

**Tool 设计原则：**
- Tool 粒度是"用户任务"而非"数据 CRUD"
- 用户身份从上下文获取，不暴露给 Agent
- 返回值是面向用户的结果描述，不是原始数据结构

### 7.4 AI 增强层降级策略

```java
public LeaveRequest createLeave(CreateLeaveDTO dto) {
    // 核心业务逻辑（永远执行）
    validate(dto);
    LeaveRequest request = save(dto);
    startApprovalFlow(request);

    // AI 增强（可降级）
    try {
        aiEnhancer.analyzeRisk(request);
        aiEnhancer.suggestApprover(request);
    } catch (AIServiceUnavailableException e) {
        log.warn("AI 增强不可用，走默认逻辑");
        fallbackToDefault(request);
    }

    return request;
}
```

### 7.5 AI 壁垒战略：从效率工具到领域智能

当前 AI 能力（意图识别 + 表单预填 + RAG 问答）本质上是"通用 LLM 能力的场景包装"，技术门槛不高，竞品可在数周内复刻。**真正的 AI 壁垒 = 领域数据 × 场景深度 × 时间积累**，需要从 P0 开始有意识地构建。

#### 三层壁垒体系

```
┌─────────────────────────────────────────────────────────────────┐
│  第三层（P1）：竞品无法快速复制的智能化能力                        │
│  • 辅导员 AI 周报（跨模块数据自动生成工作报告）                   │
│  • 审批智能建议（基于历史审批模式推荐通过/驳回及理由）             │
│  • 个性化知识问答（同一问题按学生画像给出不同深度回答）            │
│  • 班级健康度评分（综合考勤/违纪/活动参与等多维指标）             │
├─────────────────────────────────────────────────────────────────┤
│  第二层（P0.5）：一个"尖刀功能"做到极致                          │
│  → 学生异常关注预警                                              │
│    数据源：请假频率 + 签到缺勤 + 违纪记录 + 投诉情绪              │
│    P0 用规则引擎即可达到 80% 效果，P1 叠加 LLM 增强               │
├─────────────────────────────────────────────────────────────────┤
│  第一层（P0）：埋好数据桩，零额外开发成本                         │
│  → student_event_log 统一行为事件流                               │
│  → form_data 中保存 AI 草稿与最终提交的对比                       │
│  → 审批耗时、审批模式等决策数据                                   │
└─────────────────────────────────────────────────────────────────┘
```

#### P0 数据桩设计

P0 不需要额外开发 AI 模型，只需在业务流程中**多存几个字段**，为后续智能化积累数据。现在不存，以后想做就没有历史数据。

| 数据桩 | 落地位置 | 成本 | 未来用途 |
|--------|---------|------|---------|
| 审批决策耗时 | `task_instance` 增加 `decision_duration_ms` | 1 字段 | 训练"审批建议"模型，识别审批模式 |
| AI 预填 vs 最终提交对比 | `form_data` 增加 `ai_draft` JSONB | 1 字段 | 衡量 AI 预填准确率，持续优化 Prompt |
| 知识问答用户反馈 | `qa_message.feedback`（已设计） | 0 | 构建高校领域的 RLHF 数据集 |
| 学生统一行为事件流 | 新增 `student_event_log` 表 | 1 表 + AOP | 学生画像、异常检测、班级健康度的基础 |

**student_event_log 表设计：**

```sql
CREATE TABLE student_event_log (
    id          BIGINT PRIMARY KEY,
    tenant_id   VARCHAR(32),
    student_id  BIGINT NOT NULL,
    event_type  VARCHAR(32) NOT NULL,   -- leave_submit / checkin_absent / violation / complaint / ...
    event_source VARCHAR(32) NOT NULL,  -- 来源模块
    event_data  JSONB,                  -- 事件详情
    occurred_at TIMESTAMPTZ NOT NULL,
    created_at  TIMESTAMPTZ DEFAULT NOW()
);

CREATE INDEX idx_student_event_student ON student_event_log(student_id, occurred_at);
CREATE INDEX idx_student_event_type ON student_event_log(event_type, occurred_at);
```

各模块关键操作通过 AOP 或 EventListener 写入，业务代码无感知。

#### 数据飞轮埋点策略

数据飞轮的核心逻辑：**用户使用 → 产生数据 → 优化模型/规则 → 体验提升 → 更多使用**。P0 阶段在关键环节埋点，让飞轮从上线第一天开始转。

```
┌──────────────────────────────────────────────────────────┐
│                    数据飞轮闭环                            │
│                                                          │
│   用户使用 ──→ 埋点采集 ──→ 数据沉淀 ──→ 分析优化        │
│      ↑                                      │            │
│      └──────── 体验提升 ←────────────────────┘            │
└──────────────────────────────────────────────────────────┘
```

**P0 埋点清单：**

| 埋点事件 | 采集位置 | 数据内容 | 飞轮用途 |
|----------|----------|----------|----------|
| `ai_intent_parsed` | Python sidecar | 原始输入、识别意图、置信度、是否回退人工 | 意图识别准确率 → 优化 Prompt/Few-shot |
| `ai_tool_invoked` | Python sidecar | tool 名、入参摘要、执行结果、耗时 | Tool 调用成功率 → 修正参数提取逻辑 |
| `ai_draft_accepted` | Java 业务层 | AI 预填字段、用户修改字段、最终提交值 | 预填采纳率 → 逐字段优化预填策略 |
| `ai_fallback_triggered` | Java 业务层 | 降级原因（超时/异常/拒识别）、回退路径 | 降级频次 → 定位稳定性瓶颈 |
| `qa_feedback` | 前端 + Java | 问题、回答、用户评分（👍/👎）、修正内容 | 问答质量 → 知识库补充 + Prompt 调优 |
| `student_event` | Java AOP | 行为事件流（已有 `student_event_log`） | 学生画像 → 异常预警 + 个性化服务 |

**埋点实现原则：**

- **业务无感知**：通过 LangChain callback（AI 侧）和 Spring AOP / EventListener（业务侧）自动采集，不侵入业务代码
- **Langfuse 自动覆盖 AI 侧**：trace/span/token/latency 由 Langfuse callback handler 采集，上述 `ai_intent_parsed`、`ai_tool_invoked` 等作为 Langfuse trace 的 metadata 附加
- **业务侧埋点写 PG**：`student_event_log` + `ai_draft` 字段等复用已有数据桩设计，不额外引入埋点系统
- **脱敏优先**：埋点数据不存原始敏感信息（姓名、身份证等），仅保留结构化摘要

#### P0.5 尖刀功能：学生异常关注预警

竞品都在做"帮你填表""帮你查信息"（效率工具），但辅导员最痛的需求是**"哪些学生需要我关注"**——这是只有 AI 才能做到的洞察。

```
数据输入（全部来自 P0 已有模块，无需新数据源）：
  - 请假频率突增（请销假模块）
  - 签到连续缺勤（签到模块）
  - 违纪记录（违纪模块）
  - 投诉情绪倾向（接诉即办模块）

处理方式：
  P0 规则引擎（开发量约 2-3 天）：
    - 30 天内请假 ≥ 5 次 → 标记
    - 连续 3 次签到缺勤 → 标记
    - 同一学生多维度命中 → 提升关注等级
  P1 LLM 增强：
    - 综合多维度数据生成自然语言关注建议
    - 结合学生画像给出个性化干预建议

输出触达：
  - 辅导员工作台 → "需关注学生"卡片（3-5 人/周）
  - 企微日报 → "本周异常关注 N 人，点击查看"
  - 点击展开 → 完整行为时间线 + 关注原因摘要
```

**为什么这是壁垒：**
- 需要跨模块关联数据，竞品即使抄界面也没有数据基础
- 数据越多越准，先发者的规则/模型持续优化
- 这是辅导员**考核指标直接相关**的能力（学生安全稳定是辅导员第一责任）
- P0 用规则引擎就能做到 80% 效果，不依赖复杂 AI 模型

### 7.6 AI 增强层降级与前端感知

AI 不可用时，系统需让用户**明确感知降级状态**而非无声失败：

| 场景 | 降级策略 |
|------|---------|
| 全局 AI 入口不可用 | 输入框置灰 + 提示"智能助手暂时不可用，请使用菜单操作" |
| 场景内 AI Tab 不可用 | 自动切换到手动填写 Tab，AI Tab 显示"暂不可用" |
| 企微会话式问答不可用 | 自动回复"系统繁忙，请打开小程序操作" + 附小程序链接 |
| AI 增强功能不可用（风险标记、审批建议等） | 静默降级，走默认逻辑，不影响用户操作 |

#### 核心设计原则

1. **数据互通**：两个 Tab 共享同一份表单数据模型
2. **结构化卡片**：Agent 返回可交互的确认卡片，非纯文本
3. **默认 Tab 可配置**：AI 不可用时自动回退手动模式
4. **对话内嵌组件**（P1）：AI 追问时嵌入日期选择器等 UI 组件
5. **历史可追溯**（P1）：AI 对话记录与业务单据关联

## 九、部署架构

### 9.1 三种部署模式

| 模式 | 适用场景 | 说明 |
|------|---------|------|
| 公有云托管 | 中小规模/预算有限 | 数据加密存储，多租户隔离，包装为"托管服务" |
| 混合云 | 有一定 IT 基础 | AI 对话在公有云，敏感数据在本地 |
| 全私有化 | 安全要求高/双一流 | 所有组件部署在学校机房，数据不出校园 |

### 9.2 私有化部署参考配置

所有服务通过 **docker-compose** 一键拉起（Java + Python + PG + Redis + MinIO）。

| 部署档位 | 服务器配置 | 适用规模 | 阶段 |
|---------|-----------|---------|------|
| 基础版 | 1台（8核16G），无 GPU | 5000 人以下，AI 走公有云 API | P0 |
| 标准版 | 2台 + 1张推理卡（A10/L20） | 5000-20000 人，本地 14B 模型 | P1 |
| 高级版 | 3台 + GPU 节点（4×A100） | 20000 人以上，本地 72B+ 模型 | P2 |

**P0 基础版：单台服务器跑完所有服务。** 对中小规模学校极为友好。

## 十、大模型选型与评测方案

### 10.1 分级策略

| 档位 | 候选模型 | 显存需求 | 适用场景 | 阶段 |
|------|---------|---------|---------|------|
| 公有云 API | 通义千问 / DeepSeek API | 无 | P0 阶段全部走这个 | P0 |
| 基础版（7B） | Qwen2.5-7B, GLM-4-9B | 16-24GB | 意图识别、简单表单辅助 | P1 |
| 标准版（14B） | Qwen2.5-14B, InternLM2.5-20B | 40-48GB | 大部分日常场景 | P1 |
| 高级版（72B+） | Qwen2.5-72B, DeepSeek-V3 | 4×A100 | 复杂分析、心理陪伴 | P2 |

### 10.2 评测体系

#### 评测节奏

1. **P0**：全部用公有云 API，快速出功能
2. **P1**：在开发过程中积累 200-300 条测试用例，形成评测集
3. **P1 末期**：用评测集跑模型对比，出《模型选型评测报告》

#### 评分标准

- 意图识别准确率 >95%
- 信息抽取完整率 >90%
- 安全合规：敏感场景预警触发率 100%
