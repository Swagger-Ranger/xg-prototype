# 吴恩达 · Agent 实战建议(项目落地版)

> 整理自吴恩达 2024-2026 在 DeepLearning.AI 周报 (The Batch)、红杉 AI Ascent 演讲、AI Fund 公开课的核心观点。
> 已对照本项目(高校学生工作 AI 原生平台)的具体场景,给出落地纪律。

---

## 一、四大 Agent 设计模式("AI 的农业革命")

按上手难度从低到高,**项目应严格按此顺序推进**:

| 模式 | 一句话定义 | 难度 | ROI | 本项目对应 |
|------|----------|----|-----|---------|
| **Tool Use 工具使用** | LLM 调外部 API/函数 | ⭐ | ⭐⭐⭐⭐⭐ | 30-40 个 P0 Tool |
| **Reflection 反思** | LLM 评审自己的输出再改 | ⭐⭐ | ⭐⭐⭐⭐ | AI 观察员的自校验循环 |
| **Planning 规划** | LLM 拆解多步任务执行 | ⭐⭐⭐ | ⭐⭐⭐ | 任务编排 Agent("帮我搞定迎新") |
| **Multi-Agent 多智能体** | 多个角色 Agent 协作 | ⭐⭐⭐⭐ | ⭐⭐ | 路由 Agent + 事务助手 + 专家 Agent |

吴恩达原话:
> "**Reflection 和 Tool Use 是当前最被低估的 —— 很多团队直接冲 Multi-Agent 是搞反了顺序。**"

### 项目纪律
- **P0**:只做 Tool Use
- **P0.5**:加 Reflection
- **P1**:才考虑 Planning
- **不到 P2 不要碰 Multi-Agent**

## 二、最反直觉的实证:便宜模型 + Agent loop > 贵模型单跑

吴恩达在红杉演讲展示的 HumanEval 数据:

| 配置 | 准确率 |
|------|------|
| GPT-4 单次生成 | 67% |
| GPT-3.5 + Reflection 迭代 | **95%** |

### 三条启示

1. **不要一上来就用最贵的模型**
2. 模型选择 ≠ 效果选择;**架构选择 = 效果选择**
3. 同样的预算,Agent loop 跑便宜模型 3 次 > 跑贵模型 1 次

### 对本项目的应用

- AI 观察员**不必上来就用 GPT-4o / Claude Opus**
- 用 DeepSeek-V3 / Qwen-72B + Reflection 循环,效果可能更好,成本压到 1/10
- **私有化部署友好** — 学校更买账,合规更轻松

## 三、5 个成功要素

### 要素 1:Eval 是 Agent 项目最大 ROI 的投资

> "**不会写 eval 的团队,做 Agent 永远在原地踏步。**"

很多团队上来调 Prompt,凭感觉迭代。**改一版下一版可能更差也不知道**。

#### 项目落地
- 第一周建 50 条 eval 集(辅导员标注)
- 每次 Prompt 改动跑一遍,看准确率有没有掉
- 上线后用辅导员 👍👎 反馈持续扩充
- **详细方案见同目录 [`AI观察员-标注手册.md`](./AI观察员-标注手册.md)**

---

### 要素 2:迭代速度 > 一次做对

> "**做 Agent 像写软件,不像做研究。能一天迭代 5 次的团队会赢一天迭代 1 次的团队。**"

#### 项目落地
- 别花两周憋大版本,**每天上线一次微调**
- Prompt 改完 1 分钟内能跑全量 eval
- Langfuse 的 Prompt 版本化就是为这个准备的(改坏了 1 分钟回滚)
- CI 自动跑 eval,不达标阻止合并

---

### 要素 3:流式输出 比 等待更聪明

> "**用户能容忍 30 秒的流式回答,但容忍不了 10 秒的空白。**"

#### 项目落地
- AI 观察员生成 100 字叙述,流式吐字 vs 一次性返回,**感知速度差 5 倍**
- 任务编排 Agent 边规划边告诉用户"我现在在做第 2 步"
- 知识问答的 RAG 答案必须 SSE 流式

---

### 要素 4:Human-in-the-Loop 不是退步,是产品力

> "**最好的 Agent 不是全自主,而是知道什么时候问人。**"

#### 项目落地
- 任务编排 Agent **每步执行前必须给辅导员 review**
- AI 观察员的卡片必带 [👎 不对] 按钮
- **高风险动作(发通知、改流程、自动批假)永远不能 Agent 自动触发**

---

### 要素 5:不要纠结"这算不算真 Agent"

> "**Agent 是个连续光谱,不是二元判断。能完成任务就行,别陷进定义之争。**"

#### 项目落地
- "AI 观察员"严格说是 Reflection + Tool Use,不是 Multi-Agent,**没关系**
- 价值不在多 fancy,在辅导员真的省时间
- **内部沟通避免**"这才叫 agent / 那不叫 agent"的争论

## 四、6 个最容易翻车的坑

| 坑 | 表现 | 怎么避 |
|----|----|------|
| **Eval 太晚做** | 上线后才发现回归,改一版坏一版 | 写第一个 Prompt 之前先写 10 条 eval |
| **盲目追新框架** | LangGraph、CrewAI、AutoGen 全试一遍,啥都没产出 | P0 用最朴素的 Python 函数 + LangChain Tool |
| **过早 Multi-Agent** | 4 个 Agent 互相聊天,效果不如 1 个 Agent + 好 Prompt | 永远先用单 Agent 把场景跑通再考虑拆 |
| **不做成本/延迟监控** | 一个月烧几万 Token,延迟 30 秒,用户跑了 | 第一天就接 Langfuse,设日预算硬上限 |
| **指望 Agent 学习记忆魔法** | 以为 Agent 能"自己越用越聪明" | 必须有显式的反馈采集 + Prompt 迭代流程 |
| **Prompt 越写越长** | 加规则加到 5000 字,效果反而下降 | 反向:用 Reflection 让模型自己悟,Prompt 短反而稳 |

## 五、对本项目的 5 条关键纪律(浓缩版)

### ① P0 只做 Tool Use,Multi-Agent 留 P1
30-40 个 P0 Tool 全部写好 schema、做好权限边界、接好 Langfuse —— 这一步做透就值了。

### ② AI 观察员用 Reflection 模式,不要一次出
```
Step 1: 模型生成初稿
Step 2: 模型自评"我引用的 event_id 都能对上吗?有没有推测词?"
Step 3: 不达标就重写,达标才入库
```
这一招能把幻觉率压到 5% 以下。

### ③ 第一周就建 eval 集
50 个辅导员标注的真实案例 → 跑准确率 / 召回率 / 有用率三个指标 → 每次 Prompt 改动必跑。
**这是项目能不能持续优化的命门。**

### ④ 用国产开源模型试试,能省一大笔
公有云 API 跑通后,P1 切私有化 72B 做反思迭代,效果可能更好。
**私有化部署的学校还会更买账。**

### ⑤ 流式 + HITL 是 UX 的两个胜负手
- 所有 Agent 输出走 SSE 流式,辅导员看着字一个个蹦出来,体感快 5 倍
- 任何"自动批准 / 自动发通知"功能都要二次确认
- **永远不要让 Agent 自己按下扳机**

## 六、一句话纪律(贴墙上)

> **Agent 不是炼丹,是工程。**
>
> 评估、迭代、流式、HITL、用便宜模型反复跑 —— 这五件事做好了,90% 的 Agent 项目都能成。
>
> **本项目最大的风险不是技术,是没建 eval 就开始迭代。**
