# 销假门禁自动化 · 设计与实施计划

> 状态：草案 v0.1（2026-04-29）
> 范围：P0 增量功能，针对学生离校请假后的"销假"环节加自动化兜底
> 作者：claude（待评审）

## 1. 目标 (Goal)

把"假期到期后未销假 → 辅导员人工催销"的流程，**改为系统自动检测 + 自动销假**：

- 假期到点先发提醒（学生 + 辅导员）
- 给学生一段宽限期手动销假
- 宽限期过了还没销，且这是**离校假**，自动查门禁记录
- 找到回校记录 → 系统代替学生销假，附时间证据通知双方
- 没找到记录 → 升级提醒辅导员（不强制结束，等人工处理）

## 2. 非目标 (Non-Goals)

明确**不做**的事情，避免范围爬坡：

- ❌ 不引入新的 workflow 节点类型（`external_data_check` 等）。引擎保持 6 种节点不变。
- ❌ 不在 workflow DSL 里表达"门禁回滚"。这是引擎外的定时调度逻辑，不是审批逻辑。
- ❌ 不让 AI 决策是否自动销假。这是确定性规则，不走 LLM。
- ❌ 不接管现有手动销假流程。学生主动销假优先级永远高于自动销假；自动销假只是兜底。
- ❌ P0 不做真实门禁系统对接（海康/大华 API），只做数据接入契约 + mock。真接入是 P1 的事。

## 4. 架构总览

```
                    ┌────────────────────────────┐
                    │   外部门禁系统（学校）       │
                    └──────────────┬─────────────┘
                                   │ webhook (POST)
                                   ▼
            ┌──────────────────────────────────────────┐
            │  POST /api/v1/gate/records (内部)          │
            │  GateRecordIngestController                │
            │  写入 gate_record 表                        │
            └──────────────────────────────────────────┘

            ┌──────────────────────────────────────────┐
            │  LeaveAutoReturnScheduler (每 30 min)      │
            │  ├─ 扫描"假期已结束 & 未销假"的 leave        │
            │  ├─ 调用 LeaveAutoReturnService             │
            │  └─ 状态机推进                               │
            └──────────────────┬───────────────────────┘
                               │
                ┌──────────────▼───────────────┐
                │  LeaveAutoReturnService       │
                │  ├─ T+0  → 发提醒              │
                │  ├─ T+24h 离校假 → 查门禁      │
                │  │    ├─ 命中 → 调用 LeaveService.autoReturn() │
                │  │    └─ 未命中 → 升级提醒辅导员                 │
                │  └─ T+24h 在校假 → 升级提醒                       │
                └──────────────┬───────────────┘
                               │
                ┌──────────────▼───────────────┐
                │  LeaveService.autoReturn()    │
                │  复用现有销假入口，actor=system │
                │  发起 leave_return_v1 实例    │
                └──────────────────────────────┘
```

**核心设计原则：**
- 门禁数据接入与销假流程**解耦**——`gate_record` 表是只追加（append-only）数据池，调度任务按需查询
- 自动销假复用**现有**销假 workflow（`leave_return_v1`），不旁路审批逻辑
- 调度任务幂等——多次跑同一条 leave 不会重复发通知/重复销假（用 `leave.auto_return_state` 字段做状态判断）

## 5. 数据模型

### 5.1 新表 `gate_record`

```sql
CREATE TABLE gate_record (
    id              BIGSERIAL PRIMARY KEY,
    tenant_id       VARCHAR(64) NOT NULL,
    user_id         BIGINT NOT NULL,
    event_time      TIMESTAMP NOT NULL,
    direction       VARCHAR(8) NOT NULL,  -- 'in' / 'out'
    gate_id         VARCHAR(64),          -- 门禁设备标识，可空
    source          VARCHAR(32) NOT NULL, -- 'webhook' / 'mock' / 'manual'
    raw_data        JSONB,                -- 原始上报报文，留作审计
    created_at      TIMESTAMP DEFAULT NOW()
);
CREATE INDEX idx_gate_record_user_time ON gate_record (tenant_id, user_id, event_time DESC);
```

- **append-only**：永不更新或删除已有记录
- **多租户**：schema 级 + 字段双重隔离
- **索引**：按 (tenant, user, time) 倒序，支持"找该学生最近一次入校"这类查询

### 5.2 `leave_request` 表新增字段

```sql
ALTER TABLE leave_request ADD COLUMN is_off_campus BOOLEAN DEFAULT FALSE;
ALTER TABLE leave_request ADD COLUMN auto_return_state VARCHAR(32) DEFAULT 'pending';
ALTER TABLE leave_request ADD COLUMN auto_return_data JSONB;
```

`auto_return_state` 状态机：
```
pending → reminded → checking → auto_returned (终态)
                              ↘ escalated     (终态——等人工)
                              ↘ manually_returned (学生自己销假，状态机停)
```

`auto_return_data` 存：提醒发送时间、查到的 gate_record_id、升级时间等审计信息。

### 5.3 表单字段 `is_off_campus`

请假提交表单加一个布尔字段（"是否需要离校"），默认 false。学生填这个字段，决定后续自动销假是否走门禁查询路径。

## 6. 状态机详细规则

```
[T0 = leave.end_time, 假期结束时间点]

scheduler 每 30 min 扫一次：
  WHERE auto_return_state IN ('pending', 'reminded', 'checking')
    AND status = 'in_leave'  -- 已生效但未销假
    AND end_time IS NOT NULL

按状态分派：

case 'pending' (从未提醒过):
  if now() >= T0:
    发通知 → 学生 + 辅导员："假期已结束，请尽快销假"
    set state = 'reminded'
    set auto_return_data.reminded_at = now()

case 'reminded' (已提醒，等宽限):
  grace = 24h  -- 配置项
  if now() < T0 + grace:
    skip (继续等)
  else:
    if is_off_campus:
      set state = 'checking'
    else:
      // 在校假，没法用门禁验证 → 直接升级
      升级通知辅导员
      set state = 'escalated'

case 'checking' (查门禁):
  records = SELECT FROM gate_record
            WHERE user_id = leave.student_id
              AND direction = 'in'
              AND event_time > T0
            ORDER BY event_time ASC LIMIT 1;
  if records 命中:
    调用 LeaveService.autoReturn(leave_id, gate_record_id)
      → 创建 leave_return_v1 实例，actor=system
      → 发通知："系统已自动销假，依据：{event_time} 门禁入校记录"
    set state = 'auto_returned'
    set auto_return_data.gate_record_id = ...
  else:
    if now() < T0 + grace + 24h:
      // 再宽限 24h（门禁数据可能延迟上报）
      skip
    else:
      升级通知辅导员
      set state = 'escalated'

case 'manually_returned':
  // 学生自己点了销假按钮 → 监听 leave_return_v1 实例完成事件
  // 在 LeaveWorkflowListener 里把 auto_return_state 改成这个，调度任务下次跳过
  noop
```

## 8. 接入契约（webhook 协议）

为了 P1 真接入海康/大华，先冻结 webhook 协议：

```
POST /api/v1/gate/records
Headers:
  Content-Type: application/json
  X-Tenant-Id: <tenant_code>
  X-Signature: <hmac_sha256(body, shared_secret)>  -- 防伪
Body:
{
  "events": [
    {
      "user_id": "2104",            // 学校的学号或人员 id
      "event_time": "2026-04-29T15:30:21+08:00",
      "direction": "in",            // 'in' / 'out'
      "gate_id": "north_gate_01",
      "raw": { /* 厂商原始数据 */ }
    }
  ]
}
Response: { "ingested": <int>, "rejected": <int>, "errors": [...] }
```

**P0 mock 模式**：
- 没真接入时，调度任务也不能干活
- 提供管理后台一个"录入门禁记录"小工具（手动表单），方便在 demo / 测试时模拟门禁数据
- 或者提供一个 SQL 脚本灌测试数据

## 10. 风险与边界

| 风险 | 缓解策略 |
|------|---------|
| 调度任务漏跑 / 重复跑 | `auto_return_state` 字段 + DB 行锁 (`SELECT FOR UPDATE`) 保证幂等 |
| 学生从侧门进/门禁离线导致漏报 | 默认走升级路径（不自动失败）；辅导员手动销假兜底 |
| 时钟漂移（门禁系统 vs 业务系统） | 比较时间时给 ±5 min 容差 |
| 门禁数据延迟（凌晨批量上报） | `checking` 状态多给 24h 宽限再升级 |
| webhook 被伪造 | HMAC-SHA256 签名校验，shared_secret 配在 application.yml |
| 假期开始当天被自动销假 | scheduler 必须先校验 `now() >= end_time`，且 `end_time IS NOT NULL` |
| 学生跨租户上学 | gate_record 按 tenant_id 隔离，扫描时严格按当前 tenant 查 |
| 多次入校（出去又回来再出去） | 取**第一次** event_time > end_time 的 in 记录即可 |

## 12. 兼容性 & 回滚

- 所有 leave_request 老数据 `auto_return_state` 默认 'pending'，但 `is_off_campus = false`，所以老数据不会触发自动销假（在校假 → 直接升级，等于现有人工流程）
- 关停开关：application.yml 加 `leave.auto-return.enabled=false`，scheduler 跳过整个逻辑
- 回滚步骤：禁用 scheduler → 数据保留（不清表） → 后续可重新启用
