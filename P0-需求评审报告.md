## 一、阻塞项（必须修复后才能进入开发）

### 🔴 B-01：P0 范围过大，6个月交付风险极高
**来源：项目经理 + AI产品经理**

10个业务模块 + 平台底座 + AI sidecar + 三端前端，保守估计约 **44人月**，6个月交付需要至少8人全职团队且无缓冲。

**推荐 MVP 最小集（5+1 模块）：**

| 优先级 | 模块 | 理由 |
|--------|------|------|
| 必保 | 平台底座 | 所有模块基础 |
| 必保 | 请销假 | 学生最高频，验证工作流引擎 |
| 必保 | 信息收集 | 辅导员最高频，验证动态表单 |
| 必保 | 签到 | 高频刚需，开发量小 |
| 必保 | 通知任务 | 验证消息通知中心 |
| 必保 | 学生信息库 | 基础数据底座，其他模块依赖 |
| P0.5 | 接诉即办、辅导员工作日志 | 2-3月后补齐 |
| 推迟P1 | 勤工助学、学生违纪、知识问答 | 流程复杂/合规要求高/RAG开发量大 |

**建议开发顺序：**
```
阶段1 (月1-2): 平台底座（认证/RBAC/组织架构/工作流引擎/通知中心/文件服务）
阶段2 (月3-4): 核心业务并行（请销假/信息收集/签到/通知任务 + AI sidecar框架）
阶段3 (月4-5): 学生信息库 + 企微接入 + AI Tool实现
阶段4 (月5-6): 联调测试 + 试点学校数据导入 + UAT
```

### 🔴 B-02：假别类型不完整，缺少政策必要假别
**来源：学工业务专家**

当前仅支持病假/事假/周末离校，缺少：

| 缺失假别 | 紧迫性 | 说明 |
|---------|--------|------|
| **公假** | P0必须 | 学生代表学校参赛/征兵体检等，高频且审批规则完全不同 |
| **丧假** | P0必须 | 直系亲属去世，3-7天，辅导员审批即可 |
| 婚假 | P1 | 研究生群体 |

**改进**：假别改为可配置字典表（类似 `violation_category`），而非硬编码枚举。P0至少补充公假和丧假。

---

### 🔴 B-03：接诉即办流程不符合教育部要求
**来源：学工业务专家**

教育部《关于加强高等学校"接诉即办"工作的通知》要求：

| 要求 | 当前设计 | 差距 |
|------|---------|------|
| 24小时内响应 | 72小时才提醒 | 不合规 |
| 分派到职能部门 | 只分派给个人 | 不合规 |
| 督办机制 | 仅超时提醒 | 缺失 |
| 不满意重新办理 | 评价后直接关闭 | 缺失 |

**改进**：
1. 时限改为24小时受理、5工作日办结
2. 增加部门维度，按分类分派到部门
3. 满意度 ≤2 分自动重新打开
4. 超期自动通知校级领导

---

### 🔴 B-04：学生信息库缺少关键字段
**来源：学工业务专家**

`student_profile` 缺少以下高校管理必需字段：

| 缺失字段 | 影响 |
|---------|------|
| 经济困难认定等级 | 勤工助学优先安排的核心依据 |
| 学生类型（本/硕/博） | 不同层次管理规定不同 |
| 籍贯/生源地 | 招生就业基础数据 |
| 银行卡信息 | 勤工助学工资发放 |
| 导师信息 | 研究生请假需导师知晓 |

**改进**：核心字段不应放在 `extra` JSONB 中，补充为正式列。

---

### 🔴 B-05：勤工助学不符合管理办法
**来源：学工业务专家**

不符合《高等学校学生勤工助学管理办法》（教财〔2018〕12号）：

1. 无经济困难学生优先机制
2. 未区分固定岗位/临时岗位（计薪方式不同）
3. 缺少用工部门角色（实际流程：用工部门申报→学工部审批→发布）
4. 无工资最低保障校验

> 如采纳 B-01 建议将勤工助学推迟到 P1，此项可暂缓。但若保留在 P0，必须修复。

---

### 🔴 B-06：computed 表达式与总体设计约束矛盾
**来源：项目经理 + AI产品经理**

总体设计规定 `computed` 收敛为内置函数，但 YAML 中写了 `ceil((end_time - start_time) / 86400000)`，属于自由算术表达式。工作流引擎按哪个规范实现会产生分歧。

**改进**：统一为 `computed: "duration_days(start_time, end_time)"`，引擎实现内置函数。

---

### 🔴 B-07：微信小程序/企微应用注册需立即启动
**来源：项目经理**

小程序首次审核 1-2 周（教育类目可能更长），企微自建应用配置+审核也需要提前。这是**外部依赖**，不启动会阻塞月3-4的前端联调。

**行动**：立即启动主体注册、认证、应用创建。

## 四、三方共识点（文档做得好的地方）

1. **架构决策务实**：单体优先、可拆不先拆、AI可降级，三位专家一致认同
2. **工作流 YAML DSL 设计合理**：配置完整可执行，是整份文档的亮点
3. **Tool 粒度正确**："用户任务级"而非 CRUD 级，身份从上下文获取不暴露给 Agent
4. **请销假模块设计最完整**：业务规则详尽（时间重叠校验、销假提醒定时任务等），可作为其他模块的标杆
5. **数据模型设计扎实**：JSONB + Generated Column 方案适合 P0 阶段
6. **审计日志设计完整**：不可删除、变更 diff、AOP 自动采集，合规必备
