# 主动关怀任务去重与冷却落地方案

> 面向问题：  
> 1. 今天命中 10 个学生，但辅导员首页只显示 5 个，剩下 5 个第二天怎么办？  
> 2. 同一个学生、同一类事件如果连续几天发生，会不会天天推一条新的关怀告警？

## 1. 先给结论

落地上不要把主动关怀做成“每天生成一批新告警”，而要做成“持续维护一个待处理任务池”。

也就是说：

1. 同一学生 + 同一规则，在未关闭前只能有 1 个 `care_task`
2. 新事件再次命中时，不新建任务，而是给已有任务追加证据、必要时升级严重度
3. 任务关闭后进入冷却期，冷却期内即使再次命中，也不重新开任务
4. 辅导员首页“今日 5 条”只是从未关闭任务池里选优先级最高的 5 条，不是“今天只生成 5 条”

这样就能同时回答两个产品问题：

- 今天没排进前 5 的学生，不会丢；第二天还会继续参与排序
- 一个学生的同类事件不会天天变成一条新告警；只会让原任务变“证据更多 / 更紧急 / 更临期”

- `student_event_log` 已支持 `source_id`，能从事件回溯到业务单据

- 通知有统一出口 `NotificationOrchestrator.send(...)`
- 学生对应辅导员已有统一查法 `AssigneeLookupMapper.findCounselorsOfStudent(...)`

### 2.2 当前真正缺的不是“表”，而是“生成入口”

现在 `care_task` 侧缺的不是查询接口，也不是状态机，而是下面这件事：

> “规则命中一次后，到底是新建任务、合并到旧任务、还是被冷却压住？”

这层逻辑目前还没有统一入口。

换句话说，现在已有：

- 用户处理任务的能力
- 任务状态迁移能力
- 任务审计能力

但还没有：

- 规则命中 -> `care_task` 的 upsert / merge 能力
- open 唯一约束
- closed 后 cooldown 抑制
- “证据追加但不重复通知”的策略

### 2.3 旧 `student_alert` 里其实已经有一半经验

旧预警引擎 `StudentAlertService` 里已经有几条可直接借用的思路：

1. open 时不重复插入
2. resolved 后按 `cooldown_days` 抑制
3. `acknowledged` 时允许 re-fire

但主动关怀和旧 alert 的差别在于：

- 旧 `student_alert` 更像“告警记录”
- 新 `care_task` 是“老师要处理的任务”

所以新方案不能直接照搬“re-fire 重新打开一条”，而应该改成：

- 未关闭时永远只维护同一条任务
- 新命中只追加证据 / 升级任务

## 3. 目标行为定义

建议把“规则命中”后的处理分成 4 种情况。

| 情况 | 处理方式 | 是否新建任务 | 是否发通知 |
|---|---|---:|---:|
| 没有 open 任务，也不在冷却期 | 新建 `care_task` | 是 | 是，按严重度 |
| 已有 open 任务 | 合并到已有任务，更新证据 | 否 | 否，除非严重度升级 |
| 最近已关闭，且仍在冷却期 | 压住，不新建 | 否 | 否 |
| 最近已关闭，但冷却已过，再次命中 | 新建新一轮任务 | 是 | 是 |

再补两条产品语义：

1. “今日 5 条”取自全部未关闭任务池，不是今天新建任务池  
2. “会不会天天推”取决于通知策略，不取决于是否又命中了规则

## 4. 最小改造方案

## 4.1 数据库：只补最必要字段，不再新建一套大表

建议新增一个 migration：

- `xg-backend/xg-app/src/main/resources/db/migration/tenant/V125__care_task_dedup_and_cooldown.sql`

### 4.1.1 `care_task` 补 3 个字段

```sql
ALTER TABLE care_task
    ADD COLUMN IF NOT EXISTS last_triggered_at TIMESTAMPTZ,
    ADD COLUMN IF NOT EXISTS trigger_count INT NOT NULL DEFAULT 1,
    ADD COLUMN IF NOT EXISTS cooldown_until TIMESTAMPTZ;
```

字段含义：

- `last_triggered_at`：最近一次规则再次命中的时间
- `trigger_count`：这条任务累计被同一规则命中过几次
- `cooldown_until`：这条任务关闭后，在什么时间之前不允许重开

为什么只加这 3 个：

- `trigger_data` 里已经能放证据快照，不需要再单独建一张 evidence 表
- `created_at` 已能表示首次生成时间，不必再加 first_triggered_at
- `updated_at` 会跟着证据追加变更，前端排序也能用

### 4.1.2 加 open 唯一索引

```sql
CREATE UNIQUE INDEX IF NOT EXISTS uq_care_task_open_student_rule
    ON care_task(tenant_id, student_id, rule_id)
    WHERE status IN ('pending', 'accepted', 'in_progress', 'overdue');
```

这条索引是整个方案的关键。

它保证：

- 同一租户
- 同一学生
- 同一规则
- 在“未关闭状态”下

数据库层面只能存在一条任务。

这样即使定时扫描和事件触发并发撞车，也不会插出两条 open 任务。

### 4.1.3 加冷却查询索引

```sql
CREATE INDEX IF NOT EXISTS idx_care_task_rule_closed
    ON care_task(tenant_id, student_id, rule_id, closed_at DESC);
```

用途：

- 查最近一次关闭任务
- 判断是否还在冷却期

### 4.1.4 `trigger_data` 结构约束

不建议把所有历史证据无限堆进 `trigger_data`，否则 JSON 会越来越大。

建议 `trigger_data` 只保留：

```json
{
  "rule_name": "连续缺课",
  "window_days": 5,
  "threshold": 3,
  "actual_count": 4,
  "matched_event_ids": [101, 102, 103, 104],
  "matched_events": [
    {
      "id": 104,
      "source_id": 90021,
      "event_type": "checkin_absent",
      "occurred_at": "2026-05-16T08:30:00+08:00",
      "summary": "2026-05-16 第1节缺课"
    }
  ],
  "last_hit_at": "2026-05-16T08:30:00+08:00"
}
```

控制原则：

- 保留计数
- 保留最近几条样本证据
- 保留 event_id / source_id 方便回溯
- 历史全量明细继续以 `student_event_log` 为准

## 4.2 服务层：新增一个“规则命中入口”，不要把逻辑塞进 `CareTaskService`

当前 `CareTaskService` 更适合处理“辅导员动作”和“状态流转”。

建议新增一个独立服务：

- `CareTaskRuleMatchService`
  或
- `CareTaskUpsertService`

职责只做一件事：

> 接收一条规则命中结果，决定是 create / merge / suppress

建议接口：

```java
public interface CareTaskRuleMatchService {
    CareTaskUpsertResult upsert(CareRuleHit hit);
}
```

其中 `CareRuleHit` 至少包含：

- `studentId`
- `ruleId`
- `ruleVersion`
- `severity`
- `cooldownDays`
- `dueAt`
- `triggeredAt`
- `triggerData`

### 4.2.1 建议的核心流程

伪码如下：

```java
@Transactional
public CareTaskUpsertResult upsert(CareRuleHit hit) {
    CareTask open = mapper.findOpenTask(hit.studentId(), hit.ruleId());
    if (open != null) {
        return mergeIntoOpenTask(open, hit);
    }

    CareTask latestClosed = mapper.findLatestClosedTask(hit.studentId(), hit.ruleId());
    if (latestClosed != null
            && latestClosed.getCooldownUntil() != null
            && latestClosed.getCooldownUntil().isAfter(hit.triggeredAt())) {
        writeAuditSameStatus(latestClosed, "suppressed_by_cooldown", hit);
        return SUPPRESSED;
    }

    return createNewTask(hit);
}
```

### 4.2.2 merge open task 的规则

如果已存在 open 任务：

1. 不新建任务
2. 更新 `trigger_data`
3. 更新 `last_triggered_at`
4. `trigger_count + 1`
5. `updated_at = now`
6. 如有需要，升级 `severity`
7. 如有需要，收紧 `due_at`
8. 写 `care_task_audit`

### 4.2.3 严重度升级规则

定义顺序：

```text
low < medium < high < critical
```

若新命中的严重度更高：

- `severity` 升级到更高值
- `due_at = min(当前 due_at, 新 severity 对应 SLA 截止)`
- 记一条 `severity_upgraded` 审计
- 发一次升级通知

若新命中的严重度相同或更低：

- 不降级
- 不额外发即时通知
- 只更新证据

这里一定不要做“来一条事件就来一条通知”，否则马上回到天天推。

### 4.2.4 冷却期规则

当任务进入终态 `resolved / rejected / transferred` 时：

```text
cooldown_until = closed_at + rule.cooldown_days
```

这样再次命中时只要查 `cooldown_until` 即可，不必每次重新回算。

### 4.2.5 审计怎么写，才能不改大表

当前 `care_task_audit` 设计本来偏“状态迁移”。

为了低影响，不建议另起新表；直接复用 `care_task_audit`，但对非状态变化动作这样处理：

- `from_status = 当前状态`
- `to_status = 当前状态`
- `action` 改为语义动作

新增动作建议：

- `created`
- `evidence_appended`
- `severity_upgraded`
- `suppressed_by_cooldown`
- `notified`
- `urge_sent`

这样既不需要重建审计模型，也能把“为什么今天没新建 / 为什么任务变严重了”留痕出来。

2. 但要新增一个“单责任人”选择策略

建议新增 mapper 方法：

1. 优先 `counselor_org_mapping.is_primary = true`

## 4.4 规则扫描：先桥接旧引擎，减少影响面

如果目标是“尽快落地、少动大流程”，不建议第一步就重写整套 13 条规则引擎。

建议分两步：

### 方案一：过渡期复用旧 `student_alert` 规则扫描

现有：

- `AlertScanScheduler`
- `StudentAlertService`
- `AlertRuleEngine`
- `alert_rule.config.cooldown_days`

都已经能跑。

可以先把“命中规则后写 `student_alert`”改成：

1. 规则照旧扫描
2. 扫描结果不直接当成最终展示对象
3. 转成 `CareRuleHit`
4. 调 `CareTaskRuleMatchService.upsert(...)`

这样能复用：

而不用一上来同时改：

- 规则引擎

### 方案二：等 W4 再逐步退出旧 `student_alert`

这比一次性切换稳很多。

## 4.5 通知策略：把“是否推送”和“是否命中”彻底解耦

这块是用户体感最敏感的地方。

### 4.5.1 原则

规则命中 != 立刻发一条新通知

通知只在以下场景触发：

1. 新建任务
2. 严重度升级
3. 进入超期
4. 被领导督办

普通证据追加不发即时通知。

### 4.5.2 与 PRD 4 个模板的对应关系

继续按 PRD 保持这 4 条模板：

- `care_task_high_immediate`
- `care_task_daily_digest`
- `care_task_critical_dashboard`
- `care_task_urge`

建议触发规则：

| 场景 | 模板 | 备注 |
|---|---|---|
| 新建 high 任务 | `care_task_high_immediate` | 即时 |
| 新建 medium 任务 | `care_task_daily_digest` | 进 09:00 聚合 |
| 新建 low 任务 | 无额外模板 | 只进站内待办 |
| 新建 critical 任务 | `care_task_critical_dashboard` | 看板/站内，不自动外发企微 |
| open 任务只追加证据 | 不发 | 防噪音 |
| open 任务升到 high/critical | 对应升级模板 | 只发一次升级提醒 |
| 任务超期 | 先只进站内 / 待办 | 不建议额外天天催 |
| 领导督办 | `care_task_urge` | 明确人工触发 |

### 4.5.3 “第二天还会不会再提醒”

建议明确成两层：

1. 列表层面：会  
   未关闭任务第二天继续出现在任务池里，仍然可能进前 5
2. 通知层面：不一定  
   不因为“第二天到了”就重复发同一条即时提醒

medium 的日聚合可以统计“当前待处理任务数”，而不是“今天新建任务数”。

这样老师第二天仍能看见它，但不会被同一学生同一规则连续轰炸。

## 4.6 首页“只看 5 个”怎么落地

首页的“今日关怀清单”建议不是查“今天新建的任务”，而是查：

```text
status in (pending, accepted, in_progress, overdue)
按 severity -> due_at -> updated_at desc 排序
limit 5
```

解释：

- `severity`：先看紧急程度
- `due_at`：再看谁更临期
- `updated_at desc`：相同优先级下，最近追加新证据的更靠前

这样今天没排进前 5 的任务：

- 不会丢
- 明天仍在池子里
- 如果又新增了证据，反而会更容易被顶上来

## 5. 分阶段实施建议

## 5.1 第一阶段：只做后端合并入口

目标：先把“天天推不推”的核心逻辑控住。

改动：

1. `V125` migration

   - `findOpenTaskByStudentAndRule`
   - `findLatestClosedTaskByStudentAndRule`
3. 新增 `CareTaskRuleMatchService`
4. 新增单元测试 / 集成测试

这一步还不用动前端。

## 5.2 第二阶段：让扫描器改走 upsert

目标：让规则命中真正接入新逻辑。

改动：

1. `AlertScanScheduler` 或新 `CareRuleScanScheduler`
2. 命中后不直接 `insert care_task`
3. 统一走 `CareTaskRuleMatchService.upsert`

## 5.3 第三阶段：补通知节流

目标：解决老师体感。

改动：

1. seed 4 条 `notification_template`
2. 新建时按严重度发
3. 升级时发
4. 普通 evidence append 不发

## 5.4 第四阶段：前端切成“任务池视角”

目标：让“首页 5 条”和“全部待处理”是同一个池子的不同视图。

建议接口：

- `GET /api/v1/care/tasks/today`
- `GET /api/v1/care/tasks?statuses=...`

其中：

- `/today` = top 5
- `/tasks` = 全部待处理 / 超期 / 已关闭

## 5.5 第五阶段：并行观察 + 旧视图回退

目标：保守切换。

按 PRD 保留：

- 旧 `student_alert` 列表 toggle
- 新 `care_task` 工作台主入口

UAT 期如果老师反馈噪音或排序不稳，可以先切回旧视图，不影响底层任务池数据继续积累。

## 7. 为什么这套改法影响最小

我倾向这套方案，主要因为它只改“任务生成入口”，不去动已经稳定的大面：

1. **不改工作流主链路**  
   `care_task` 本身不是 workflow instance，不会牵连审批 DSL

2. **不重写状态机**  
   `CareTaskService` 现有接单 / 改期 / 转介 / 完成逻辑保留

3. **不新加基础设施**  
   继续用 PostgreSQL、Flyway、定时任务、JSONB

4. **不推翻旧规则扫描器**  
   可先桥接旧 `student_alert` 引擎，等新工作台稳定后再收口

5. **不牺牲可回退性**  
   保留旧 alert 列表 toggle，符合 PRD 已定的降级路线

6. **能直接解决老师最痛的体感问题**  
   不是“再做一个页面”，而是真正防止天天重复推送

---

## 8. 我建议的实际实施顺序

如果按“收益最大 / 风险最小”排，我建议：

1. 先上 `V125` + `CareTaskRuleMatchService`
2. 再让规则扫描改走 upsert
3. 再补通知节流
4. 最后切首页 `/today` 和旧视图 toggle

一句话总结：

> 先把“同一学生同一规则只维护一条 open 任务”这条铁律打进数据库和服务层，后面的首页展示、第二天是否还看得到、是否重复推送，都会自然顺起来。
