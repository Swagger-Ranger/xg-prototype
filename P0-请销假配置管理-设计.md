## 0. 背景与目标

### 0.1 现状问题
- 竞争对手金智的请销假配置项 90+ 条 × N 学院 ≈ 200-400 实际管理面，学工部老师初次配置 1-2 周
- 根因：**金智按"页签"切配置 + 学院差异化只能"复制一份"**——不是业务复杂导致，是数据模型抽象不足
- 我们 P0 的请销假**作为单一具体应用**，不做通用配置平台，但要**比金智少一个数量级的配置面**

### 0.2 设计原则

> **总则：功能设计可以复杂，用户使用一定要简单。**
>
> 内部数据模型 / patch 引擎 / 优先级合并 / 健康度巡检允许做得复杂，但**任何复杂都必须挡在用户视线之外**。老师在 UI 上看到的、要做的决定、要点的按钮——能少则少，能默认则默认。如果某项设计在内部清爽但要求老师理解新概念，**优先调整 UI 而不是要求老师学**。

1. **领域专用 UI**：老师面对的是"假别 / 证明 / 时长 / 审批角色 / 特殊时期"，不是抽象 workflow 节点 / patch / DSL
2. **底层不动**：底层仍由现有 workflow DSL 状态机执行，配置层产出 DSL 由翻译层完成
3. **Patch 而非复制**：学院差异 / 特殊时期作为**叠加在基线上的 patch**，不复制整份配置——但**老师不需要知道"patch"这个词**，UI 上叫"学院特殊规则"和"特殊时期"
4. **Dry-run 兜底**：保存前必有学生样本预演，AI 改的配置必须能被验证。预演面板用自然语言描述结果，**不展示 DSL / JSON**
5. **可视化前置**：配置概览 ≠ 编辑视图，老师"看现在是什么"和"改什么"是两类需求。摘要视图一屏读完，深度细节按需下钻
6. **AI 受限生成**：AI 只输出受限 schema 内的 patch，不允许自由 DSL 生成。**老师感知不到"受限 op 集合"的存在**——AI 自己处理无法表达的请求时礼貌回退到"请人工配置"

### 0.2.1 "用户使用要简单" 的具体落地清单

| 复杂留在内部 | 用户面只看到 |
|---|---|
| OrgPatch / TimePatch 数据结构、JSON Patch 风格的 diff[] | "学院特殊规则"和"特殊时期"两个 tab，每条 = 一个表单卡片 |
| 4 op 受限集合（replace/enable/disable/elevate） | UI 不暴露 op 概念；用户填"把 X 改成 Y" / "在 X 期间禁用 Y" / "在 X 后追加审批人" |
| 优先级合并算法 `Time > Org > Base` + 同级 createdAt 倒序 | dry-run 直接展示"这个学生最终会经历什么"，不解释"哪条规则赢了" |
| Patch path 表达式 `leaveTypes[code=personal].maxDays` | UI 用下拉选假别 + 字段，路径由前端拼出 |
| sys_role.code 引用 / org_unit.id 引用 | UI 选角色用"辅导员/班主任..."、选学院用树形选择器 |
| 健康度巡检 3 条程序化规则 + AI 翻译层 | 概览页只显示自然语言提醒 + "一键处置"按钮 |
| AI 受限 schema 的 prompt / 输出校验 | 用户只看到"我的请求 + AI 生成的草稿 + dry-run 结果"，听不到"schema"这个词 |
| changelog 表 + diff_before/after | 时间线一行一句话，"X 老师改了 Y，备注：Z" |

### 0.3 P0 交付物
| # | 模块 | 形态 |
|---|---|---|
| 1 | 请销假配置管理（编辑页） | 领域专用 UI，~20 项配置 |
| 2 | Patch 引擎（OrgPatch + TimePatch） | 数据结构 + 优先级合并 + 翻译层 |
| 3 | Dry-run 预演器 | 基于现有 workflow 引擎的只读模式 |
| 4 | 配置概览页 | 摘要视图 + 时间线（视图 1+4 必做，2+3 后置） |

### 1.1 LeaveConfigBase（基线配置）

#### 字段说明
| 字段 | 说明 | P0 必填 |
|---|---|---|
| `leaveTypes[].code` | 内部假别 ID，全租户唯一 | ✅ |
| `leaveTypes[].name` | 显示名 | ✅ |
| `leaveTypes[].enabled` | 是否启用此假别 | ✅ |
| `leaveTypes[].maxDays` | 单次最大天数；`null` 表示不限 | ✅ |
| `leaveTypes[].annualCapDays` | 学年累计上限；`null` 表示不限 | 可选 |
| `leaveTypes[].proof.required` | `none` / `optional` / `required` | ✅ |
| `leaveTypes[].proof.fileTypes` | 允许的附件类型 | ✅ |
| `leaveTypes[].approvalChain[]` | 时长档位 → 审批角色链 | ✅ |
| `leaveTypes[].parentCode` | 父类 code（如 `sick_on_campus.parentCode = sick`），用于分组展示和聚合统计 | 可选 |
| `leaveTypes[].sortOrder` | 显示顺序，整数升序 | ✅ |
| `leaveTypes[].extraFields` | **每假别的动态表单扩展字段**，迁移自 V010 `leave_type_config.extra_fields`；form 渲染和校验依赖此字段。**v0.5 明确选 map 形态**：`{ "<field_key>": { "type", "label", "required", "options"? } }`——比现有 V010 的数组形态紧凑，按 key 索引方便。迁移脚本负责形态转换 | 可选 |
| `notifications[]` | 通知规则数组，详见 §1.1.1 | ✅ |
| `lastModified.note` | **强制要求填写改动原因**，未来时间线靠它 | ✅ |

#### approvalChain 语义
- 数组按 `maxDays` 升序排列
- 取**第一个 `天数 ≤ maxDays`** 的档位
- `roles` 是顺序串行审批链
- 例：`maxDays=3, roles=[counselor]` 匹配 1-3 天；`maxDays=7, roles=[counselor, college_secretary]` 匹配 4-7 天
- `roles[]` 元素必须是 `sys_role.code`（参考 §6 决议 #8，需先做角色补齐迁移）
- **天数匹配是闭区间**：精确 `天数 == maxDays` 走该档；`leave_request.duration_days` 是 DECIMAL(5,1)，**向上取整后参与匹配**（7.5 天 → 8 天，按 ">7 天" 档位）

#### leaveTypes.code 不变性约束
- 全租户内 `code` 必须唯一
- 一旦使用过（产生过 leave_request 数据），**禁止删除，只能 disable**——避免历史外键失效
- 重命名（修改 `name`）允许，但 `code` 不可改
- 系统内置 6 类（事 / 病 / 公 / 丧 / 婚 / 产）默认 enabled，租户可自定义新增

### 1.1.1 通知规则（notifications[]）

P0 在请销假配置内**领域内配置**，不抽全局 NotificationRule。每条规则由以下字段组成：

| 字段 | 类型 | 说明 |
|---|---|---|
| `event` | enum | 触发事件（见下表） |
| `when` | string (可选) | 时间条件，仅 `deadline_warning` / `overdue_unreturned` 等延时事件用 |
| `to[]` | enum array | 接收人角色 |
| `channels[]` | enum array | 通道（wecom / sms / miniapp） |
| `templateCode` | string | 模板逻辑 key（对应代码端 i18n 资源文件，**P0 不引入数据库模板表**，详见 §8.4） |

**P0 支持的 event 列表**：

| event | 触发时机 |
|---|---|
| `submit` | 学生提交申请 |
| `approved_node` | 单节点审批通过（推进到下一节点） |
| `approved_final` | 全链审批完成 |
| `rejected` | 任意节点驳回 |
| `recalled` | 学生主动撤回 |
| `deadline_warning` | 到期前提醒（when: `Nd before scope.to`） |
| `overdue_unreturned` | 逾期未销假（when: `Nd after scope.to`） |

**P0 支持的 to 角色**（从 `sys_role.code` 选）：

`student / counselor / class_master / college_secretary / student_affairs_director / parent`

> ⚠ `parent` 是虚拟角色，对应学生主数据中的 `emergency_contact`。

#### Patch 对通知的支持（P0 限制）
- **P0 不允许 OrgPatch / TimePatch 修改 notifications**
- 学院通知差异 / 免打扰时段 / 频次控制 → **P1 引入 NotificationRule 抽象后再支持**
- P0 内"通知规则"只能在基线配，所有学院共享

> 这是有意收窄。先确保基线 + leaveTypes patch 跑通再扩展，避免 op 集合膨胀。

### 1.2 OrgPatch（学院差异）

> 单条 patch 影响 1 个或多个学院。

```json
{
  "patchId": "uuid-1",
  "type": "org",
  "tenantId": "demo_univ",
  "scope": {
    "orgIds": [10023],
    "orgNamesSnapshot": ["艺术学院"]
  },
  "diff": [
    {
      "path": "leaveTypes[code=personal].maxDays",
      "op": "replace",
      "value": 14
    }
  ],
  "enabled": true,
  "createdAt": "2026-03-20T09:00:00Z",
  "createdBy": "user_id_456",
  "createdByName": "王老师",
  "note": "演出季需要"
}
```

#### 字段说明
| 字段 | 说明 |
|---|---|
| `scope.orgIds` | 适用学院列表，元素为 `org_unit.id` (BIGINT)，参考 V002 已有组织架构表 |
| `scope.orgNamesSnapshot` | 冗余存名字便于历史展示（学院改名后仍可读） |
| `diff[]` | 一组差异修改，每条独立 |
| `enabled` | 软启停，便于学期切换 |
| `note` | **强制必填**——可视化层时间线展示用 |

### 1.3 TimePatch（特殊时期）

> 单条 patch 在指定时间段内生效。可作用于全校或部分学院。

```json
{
  "patchId": "uuid-2",
  "type": "time",
  "tenantId": "demo_univ",
  "name": "考试周",
  "scope": {
    "from": "2026-05-15",
    "to": "2026-06-15",
    "orgIds": null
  },
  "diff": [
    {
      "path": "leaveTypes[code=personal].approvalChain",
      "op": "elevate",
      "value": { "addRoles": ["student_affairs_director"] }
    }
  ],
  "enabled": true,
  "createdAt": "2026-04-12T10:30:00Z",
  "createdBy": "user_id_123",
  "createdByName": "李老师",
  "note": "防止考试周请假冲击教学秩序"
}
```

#### 字段说明
| 字段 | 说明 |
|---|---|
| `name` | 时期名（"考试周"、"迎评期"），便于可视化展示 |
| `scope.from` / `scope.to` | 生效区间，闭区间，按学生**请假发起日**判定 |
| `scope.orgIds` | `null` = 全校；非空 = 仅这些学院（元素为 `org_unit.id`） |
| `diff[].op` | 见下方 op 集合 |

### 1.4 受限 Op 集合（P0）

只允许以下 4 种，**AI 与 UI 均按此约束**：

| Op | 适用类型 | 语义 | 示例 |
|---|---|---|---|
| `replace` | 标量字段 / 数组 | 直接替换 | 事假最大天数 7 → 14；`proof.required` `optional → required`（替代原 `tighten`） |
| `enable` | leaveType 节点 | 启用某假别 | path 指向 `leaveTypes[code=xxx]`（**不含 `.enabled`**），op 自动设 enabled=true |
| `disable` | leaveType 节点 | 禁用某假别 | 同上，op 自动设 enabled=false（用于"考试周禁事假"） |
| `elevate` | approvalChain | **在每档 tier 的 roles 末端追加角色**（语义 A） | `addRoles: ["student_affairs_director"]` |

> ⚠ **不引入 add/remove 数组项 / 复杂 jq 表达式 / 任意脚本**——这些放到 P1 再评估。

> ⚠ **op 仅适用于 `leaveTypes` 结构**。`notifications` 数组在 P0 不接受 patch（见 §1.1.1）。

> ⚠ **v0.5 去 `tighten`**：原 `tighten` 的"单向加严"语义改用 UI 校验+`replace` 实现（保存时禁止从 `required` 降到 `optional`），减少 op 类型，降低 AI prompt 与测试矩阵复杂度。

### 1.5 Patch 优先级与合并规则

#### 合并语义：顺序叠加，不是覆盖整份

```
EffectiveConfig(student, today)
  = Base
  ▷ apply 所有适用 OrgPatch（按 createdAt 升序，逐条叠加在前一步结果上）
  ▷ apply 所有适用 TimePatch（按 createdAt 升序，逐条叠加）
```

**关键澄清**：高优先级 patch **不是替换前面的整份配置**，而是**在前面叠加结果上继续修改**。Org 改了 X 路径，Time 改了 Y 路径，两条都生效；只有当 Time 也改了 X 路径时，Time 在 X 上的值赢，但 Y 上 Org 的修改仍保留。

#### 优先级规则

| 层级 | 含义 | 同 path 冲突时 |
|---|---|---|
| **TimePatch** | 最强例外（考试周/迎评期） | 赢 |
| **OrgPatch** | 常态学院差异 | 赢过 Base |
| **Base** | 校级默认 | 兜底 |

#### 同 path 冲突裁决

- **同层多条 patch 同 path 同 op**：按 `createdAt` 升序应用，最新的覆盖前面（last-write-wins）
- **同层多条 patch 同 path 不同 op**：按 op 类型组合：
  - `replace` 后接 `replace` → 后者赢
  - `enable` 后接 `disable` → 后者赢（任何后续状态变更覆盖前面）
  - `replace` + `elevate` → 各自独立路径处理：replace 改的是值，elevate 改的是数组结构
- **跨层冲突**：高层 patch（Time）在 X path 上的值赢，但其他 path 上低层 patch（Org）依然生效

#### `elevate` 的特殊语义（语义 A：每档末端追加）

`elevate` 作用于 `approvalChain` 数组，在**每个 tier 的 roles[] 末端**追加角色——含义是"无论请求多少天都加这一级审批"。

**例 1**（典型场景：考试周升一级）：
```
Base approvalChain:
  [{ maxDays: 3, roles: [counselor] },
   { maxDays: 7, roles: [counselor, college_secretary] }]

TimePatch elevate +student_affairs_director（考试周升一级）:
  [{ maxDays: 3, roles: [counselor, student_affairs_director] },
   { maxDays: 7, roles: [counselor, college_secretary, student_affairs_director] }]
```
即考试周内即便是 1 天事假也要走到学工部部长——这就是"加严"的本意。

**例 2**（多 patch 叠加 + 去重）：
```
OrgPatch elevate +college_secretary（艺术学院加一级）：
  → 每档 tier.roles 都已 append college_secretary

TimePatch 又 elevate +student_affairs_director：
  → 每档 tier.roles 再 append student_affairs_director
```

**例 3**（replace 后接 elevate）：
```
Base: [{maxDays:7, roles:[counselor]}]
OrgPatch replace 整个 chain 为 [{maxDays:7, roles:[counselor, college_admin]}]
TimePatch elevate +student_affairs_director:
  → [{maxDays:7, roles:[counselor, college_admin, student_affairs_director]}]
```

#### `elevate` 去重规则（修订）

如果 `addRoles` 中的角色已经在 tier.roles 里，**跳过不重复添加**：
```
已有 roles: [counselor, student_affairs_director]
elevate addRoles: [student_affairs_director]
结果：[counselor, student_affairs_director]  // 不变
```

实现：合并阶段每次 append 前检查 `role not in tier.roles`。

#### 多条 patch 同时作用于一个学院

- 多个 OrgPatch `scope.orgIds` 包含同一学院时全部生效，按 createdAt 升序叠加
- 这是允许的（一个学院可能有"演出季加长 + 实习学分豁免"两条独立 patch）
- 健康度巡检会标记"同 path 冲突"为 warn（详见 §3.3）

#### 多条 patch 同时作用于一个时段

- 多个 TimePatch 区间重叠且适用学院重叠时全部生效，按 createdAt 升序叠加
- 例：考试周（全校禁事假）和迎评期（全校升一级审批）区间重叠 5/15-5/20，重叠期内两条都生效

#### 路径表达式与失效处理

- patch 的 `path` 形如 `leaveTypes[code=personal].maxDays`
- 当被引用的 leaveType `code` **被 disable**：相关 patch 自动失效（合并时跳过），但 patch 记录保留
- 当被引用的 leaveType `code` **不存在**（被错误删除前的残留 patch）：合并器记录 warning，跳过该 diff（仅 log，不再触发独立健康度告警——v0.5 已删 `patch_path_invalid` 规则）

#### v0.5 简化语义：编辑期间 patch 暂停生效

> ⚠ **重要 UX 含义**：编辑某条 published patch 时，该 patch 的 status 翻为 draft，**编辑期间不参与学生申请合并**（暂停生效）；发布后翻回 published 重新生效。

理由（vs. v0.4 的 parent_patch_id 双行方案）：
- 1-2 个老师场景下，编辑期短（分钟级）暂停影响极小
- 数据模型简化一半（删 parent_patch_id + FK + 双行同步逻辑）
- UI 透明告知："编辑此规则时该规则暂时不生效"，老师能理解

| 老师操作 | 数据库行为 |
|---|---|
| 新建 patch | 直接以 `status=draft` 落库 |
| 编辑 published patch | UPDATE status='draft'，老师改字段，期间该 patch 不生效 |
| 发布 draft | UPDATE status='published'，重新生效 |
| 放弃修改 | 若是新建未发布 → 删除行；若是编辑 published → 用 changelog 上一发布版本恢复字段 + status='published' |

> **base 配置例外**：base 必须始终对学生申请有效，因此**保留 (tenant_id, status) 双行 PK**——base 永远有 published 行，draft 行仅用于编辑缓冲。

#### 变更日志表（v0.5 修订：仅记状态变迁）

> ⚠ **v0.5 修订（v0.4 Bug 1）**：原 v0.4 设计 changelog 同时记录 draft 保存和发布动作，导致 `target='base'` 不能区分草稿态/发布态变更。现仅对**真正影响学生申请的变迁**记录日志，draft 保存不入此表。

```sql

CREATE TABLE leave_config_changelog (
  log_id       BIGSERIAL PRIMARY KEY,
  tenant_id    VARCHAR(32) NOT NULL,
  target       VARCHAR(64) NOT NULL,              -- 'base' | patch_id
  action       VARCHAR(16) NOT NULL,              -- 'publish' | 'enable' | 'disable' | 'delete'
  diff_before  JSONB,                              -- 变迁前 published 的快照
  diff_after   JSONB,                              -- 变迁后 published 的快照

  changed_by   VARCHAR(64) NOT NULL,
  changed_by_name VARCHAR(128) NOT NULL,
  note         TEXT NOT NULL
);

CREATE INDEX idx_changelog_tenant_time ON leave_config_changelog(tenant_id, changed_at DESC);
CREATE INDEX idx_changelog_target ON leave_config_changelog(target);
```

> **draft 保存不记 changelog**：保存到 `leave_config_base.config`（draft 行）或 `leave_config_patch`（status=draft），由 `version + updated_at + updated_by` 兜底；老师隔天接着改可以看到"上次编辑于 X"。

#### `diff_before` / `diff_after` 字段内容约定（v0.5 补充）

| target | 字段含义 |
|---|---|
| `target = 'base'` | 存 `leave_config_base.config` JSONB（含 `leaveTypes[]` + `notifications[]`），即整份基线配置 |
| `target = patch_id` | 存该 patch 行的**完整可变字段快照**：`{ "diff": [...], "scope": {...}, "name": "...", "enabled": true, "note": "..." }`——**用于"放弃修改"时恢复字段** |

> **"放弃修改"恢复路径**：编辑某条 published patch 后想放弃 → 查询 `leave_config_changelog WHERE target = patch_id AND action = 'publish' ORDER BY changed_at DESC LIMIT 1` → 用其 `diff_after` 的 5 个字段重写 patch 行 + status='published'。每条已发布过的 patch 必有至少一条 publish changelog 记录，恢复路径必然存在。

### 1.7 翻译层：两步处理（Patch 合并 + DSL 编译）

> ⚠ **C1 / M2 修订**：原伪代码混淆了 patch 合并和 DSL 编译。这两步必须显式分开，因为 `elevate` op 是合并阶段的事，不是编译阶段。

#### 第一步：Patch 合并器（PatchMerger）

**输入**：Base 配置 + 适用 patch 列表（按 §1.5 规则筛出）
**输出**：合并后的 EffectiveConfig（数据结构同 Base，但已应用所有 diff）

```python
def merge_patches(base, applicable_patches, today, student_org_id):
    """
    顺序叠加：先 OrgPatch 按 createdAt 升序，再 TimePatch 按 createdAt 升序
    """
    effective = deep_copy(base)

    org_patches = filter_by_type(applicable_patches, "org")
    org_patches.sort(key=lambda p: p.created_at)
    for patch in org_patches:
        if student_org_id in patch.scope.orgIds and patch.enabled and patch.status == "published":
            effective = apply_diff(effective, patch.diff)

    time_patches = filter_by_type(applicable_patches, "time")
    time_patches.sort(key=lambda p: p.created_at)
    for patch in time_patches:
        if patch.scope.from <= today <= patch.scope.to:
            if patch.scope.orgIds is None or student_org_id in patch.scope.orgIds:
                if patch.enabled and patch.status == "published":
                    effective = apply_diff(effective, patch.diff)

    return effective


def apply_diff(config, diff_list):
    """
    按 op 类型逐条应用。op 集合见 §1.4。
    elevate 在每个 tier 末端去重追加（语义 A）。
    enable/disable: diff.path 指向 leaveType 节点（不含 .enabled 后缀），op 自己拼。
    """
    for diff in diff_list:
        target = resolve_path(config, diff.path)  # 路径不存在则跳过 + log warning
        if target is None:
            log_warning("patch_path_invalid", diff.path)
            continue

        if diff.op == "replace":
            set_value(config, diff.path, diff.value)

        elif diff.op == "enable":
            # path = "leaveTypes[code=xxx]"，自动拼 .enabled
            set_value(config, diff.path + ".enabled", True)

        elif diff.op == "disable":
            set_value(config, diff.path + ".enabled", False)

        elif diff.op == "elevate":
            # diff.value = {"addRoles": [...]}
            # 语义 A：每个 tier 末端去重追加
            chain = get_value(config, diff.path)  # approvalChain 数组
            for tier in chain:
                for role in diff.value.addRoles:
                    if role not in tier.roles:    # 去重
                        tier.roles.append(role)

    return config
```

> v0.5 已删除 `tighten` op；`proof.required` 调整改用 `replace`，UI 层做"不允许从 required 降到 optional"的单向校验。

#### 第二步：DSL 编译器（DslCompiler）

**输入**：EffectiveConfig（合并后）+ 学生申请上下文（leaveType + days）
**输出**：现有 workflow DSL 实例（节点序列 + 条件 + SLA）

```python
def compile_to_workflow(effective_config, leave_type, days):
    type_def = effective_config.find_type(leave_type)
    if not type_def.enabled:
        raise BlockedByConfig("此假别在当前时期不可申请")
    if type_def.maxDays and days > type_def.maxDays:
        raise BlockedByConfig(f"超过最大天数 {type_def.maxDays}")

    days_rounded = ceil(days)  # 7.5 → 8
    chain = pick_chain(type_def.approvalChain, days_rounded)
    # chain = ["counselor", "college_secretary", "student_affairs_director"]
    if chain is None:
        # 边界：天数超过最大 tier maxDays（如 sick_off_campus.maxDays=30 但 chain 只到 7 天）
        # 直接 block 提交，提示老师补全 chain
        raise BlockedByConfig(
            f"未为 {leave_type} 配置 {days_rounded} 天档位的审批链——请到配置管理补全"
        )

    # 角色 → DSL 节点的 (role, scope) 映射（每角色 scope 由代码端常量决定）:
    # counselor                  → scope = same_class
    # class_master               → scope = same_class
    # college_secretary          → scope = same_college
    # college_admin              → scope = same_college
    # dean                       → scope = same_college
    # student_affairs_officer    → scope = global
    # student_affairs_director   → scope = global
    # school_admin               → scope = global
    nodes = [ApprovalNode(role=r, scope=ROLE_SCOPE_MAP[r], sla=default_sla(r)) for r in chain]

    return WorkflowDSL(
        nodes=nodes,
        proof_required=(type_def.proof.required == "required"),
        proof_optional=(type_def.proof.required == "optional"),
        proof_file_types=type_def.proof.fileTypes,
        proof_max_size_mb=type_def.proof.maxSizeMB,
    )
```

> 两步均为**纯函数**，便于单元测试。`apply_diff` 路径解析 / `pick_chain` 档位匹配 / `ROLE_SCOPE_MAP` 常量映射各自独立。

#### 角色 → 解析器的对应关系（执行时）

DSL 节点的 `(role, scope)` 由 workflow 引擎内的 `AssigneeResolver` 解析为具体审批人：

| role | scope | 解析路径 |
|---|---|---|
| `counselor` | `same_class` | `counselor_org_mapping` 反查（学生 → 班级 → 辅导员） |
| `class_master` | `same_class` | `org_unit.leader_id`（class 类型节点的 leader） |
| `college_secretary` | `same_college` | `sys_user_role` JOIN `org_unit`（type=college） |
| `college_admin` | `same_college` | 同上 |
| `dean` | `same_college` | 同上 |
| `student_affairs_director` / `_officer` | `global` | `sys_user_role` 租户级查 |
| `school_admin` / `super_admin` | `global` | 同上 |

### 1.8 配置生命周期：草稿 / 发布

> ⚠ **M6 修订**：避免老师改一半就生效造成不一致。

#### 状态机

```
[新建/编辑]
   │
   ▼
draft ──── 发布 ────► published
  ▲                       │
  │                       │ 老师再次编辑
  └───────────────────────┘
```

#### 状态语义

| 状态 | 行为 |
|---|---|
| `draft` | **不参与** 学生申请的合并计算；**参与** dry-run 预演（让老师能看到改完后效果）；可反复编辑 |
| `published` | 参与学生申请的合并计算（即真实生效）；编辑时自动派生新 draft 副本 |

#### 编辑 / 发布流程（v0.5 简化）

**Base 配置**（双行 PK 模型）：

1. 老师打开编辑页 → 系统检查是否已有 draft 行，无则从 published 行复制
2. 老师修改 → 每次保存只更新 draft 行的 `config`
3. 老师点"预览效果" → dry-run 用 draft 配置跑
4. 老师点"发布" → 校验通过后原子事务：删除原 published 行 + draft 行 status 改为 published

**Patch（OrgPatch / TimePatch）**（单行可变 status 模型）：

1. 新建：以 `status=draft` 落库，编辑期间不生效
2. 编辑已发布 patch：UPDATE status='draft'，**期间该 patch 暂停**；老师改字段
3. 发布：UPDATE status='published'，重新生效
4. 放弃：新建未发布 → 删除行；编辑已发布 → 用 changelog 上一发布版本恢复 + status='published'

#### 并发处理（v0.5 简化）

> ⚠ **v0.5 删除 `editing_session_id` / 接管机制 / 4 并发场景表**——P0 租户量个位数、配置员 1-2 人，并发概率近零。

**仅保留乐观锁兜底**：
- 所有 update 用 `WHERE version = ?`
- 冲突返回 HTTP 409，前端提示"配置已被其他人修改，请刷新后重试"
- 这是 1 处事件，不是常态——没必要为常态化设计

#### 发布范围（v0.5 简化）

**点"发布"时，发布该租户当前**所有 status='draft' 的行**（base draft + 所有 draft patches）一起**——不区分谁在哪个 session 改的。

理由：
- 1-2 个老师，编辑互相不干扰的概率极高
- 万一干扰，发布前老师在概览页能看到"还有 3 条草稿待发布"，可主动确认或放弃
- 不上 editing_session_id 复杂度

事务步骤：
1. BEGIN
2. 查询该租户所有 `status='draft'` 的 base + patch 行
3. 对每行做健康度校验（详见 §3.3）
4. 全部校验通过 → 一次性翻 status='published'，记 changelog（action=publish）
5. base 删除原 published 行（如有），draft 行升 published
6. 触发即时健康度巡检
7. COMMIT；失败 ROLLBACK

#### 撤销 / 回滚（v0.5 简化）

> ⚠ **v0.5 删除"回滚到上一个发布版本"专门按钮**——老师手动操作即可。

- **放弃当前修改**：编辑过程中点"放弃" → base 删 draft 行；patch 删行（新建未发布）或恢复字段（编辑发布过的，从 changelog 取 diff_after）
- **想回退已发布的修改**：老师重新打开编辑、修改回去、再发布——和首次配置同一流程，无新概念
- changelog 仍可看历史 diff 帮助老师"想起"之前的值，但不提供一键恢复

#### UI 上的呈现（"使用要简单"原则）

- 老师面对的不是"draft / published"两个词，而是：
  - 进入编辑页顶部 banner："您正在编辑草稿，未保存修改不会影响学生申请"
  - 改完后底部按钮："保存草稿" / "预览效果" / "发布生效"
  - 概览页有"上次发布于 X 时间，当前有未发布修改"提示
  - 编辑某条 published patch 时，**额外显著提示**："正在编辑此规则，编辑期间该规则暂时不生效。完成编辑并发布后将重新生效。"

### 2.4 实现路径

> ⚠ **M1 修订**：现有 workflow 引擎**没有任何 dry-run 设施**——`WorkflowEngine.startWorkflow()` 直写 `workflow_instance` / `form_data`，`ApprovalExecutor` 直写 `task_instance`，`AssigneeResolver` 也是直读 DB。"加 `dry_run=true` 参数"严重低估工作量。

#### 工程量评估：独立 3-5 天工作项

不是"加个参数"。需要：

1. **新建 `DryRunEngine`**（独立执行器，不复用 `WorkflowEngine`）
   - 复用节点定义和条件表达式求值
   - **不**调用任何 `*Mapper.insert/update/delete`
   - 节点推进时收集 trace event 而非写库
2. **改造 `AssigneeResolver` 双模运行**
   - 真实模式：解析为具体 `user_id`
   - dry-run 模式：解析为"角色 + 候选人数 + 第一候选人 displayName"
3. **`ConditionExecutor` 的 dry-run 兼容**
   - 当前用真实表单数据求值；dry-run 需用样本学生属性求值
4. **Trace 输出格式映射**
   - 把 trace event list 转换为 §2.2 的 JSON 输出格式
5. **事务隔离保证**
   - 即便实现层用了事务回滚兜底（防止误写），也要在请求维度强制 read-only 连接

#### P0 实现策略选择

**推荐方案 B（独立 DryRunEngine）**：
- 优点：与生产引擎完全隔离，不存在"忘记加 dry-run 分支"的隐患
- 缺点：节点定义升级时要双边同步（约束：通过抽象 `NodeBehavior` 接口共享）

**方案 A（WorkflowEngine 加 dry_run 模式）**：
- 优点：代码复用度高
- 缺点：每个 mapper 调用前都要 `if (!dryRun)` 判断，**漏一处就炸**

#### 验收标准（dry-run 单独）

- [ ] 5 个样本学生 × 3 种典型配置 = 15 组测试用例全部命中预期路径
- [ ] dry-run 跑完**全程零 SQL INSERT/UPDATE/DELETE**（用 PG 级别审计验证）
- [ ] 同一份配置 + 同一份学生数据，dry-run 输出与真实提交后 `task_instance` 链一致

### 3.3 健康度巡检规则（P0）

#### 为什么必要

老师配 patch 多了**一定会忘**。学校政策一变旧 patch 不会自动消失，时间一长配置"腐化"——能跑但越来越乱。健康度巡检由系统**主动**找配置问题告诉老师，类比体检：医生发现你血压高告诉你。**这是金智完全没做的能力**，是我们差异化的一部分。

巡检规则**纯程序化**（非 LLM），零幻觉风险。AI 仅负责把 issue code 翻译成自然语言显示。

#### 触发时机
- **每天凌晨 02:00 全租户跑一次**（定时任务）
- **保存配置后即时跑一次**（针对刚改的租户，让老师立刻看到新隐患）

#### P0 3 条规则（v0.5 砍小）

> ⚠ **v0.5 删 3 条**：`dormant_patch`（驱动 applied_patch_ids 字段，工程性价比低）、`redundant_org_patch`（清洁度规则，无业务影响）、`patch_path_invalid`（合并器已 log+skip 处置，UI 卡片冗余）。
>
> 保留 3 条核心规则，**P1 视情况补充**。

| 规则 code | 触发条件 | 严重度 | 实际场景 |
|---|---|---|---|
| `expired_time_patch` | `scope.to < today` AND `enabled = true` AND `status = 'published'` | warn | "迎评期 patch 已过期但还启用，下次同名时期复活会造成混乱" |
| `conflicting_patches` | 两个 published patch 修改同一 path 且值不同（同 op type） | warn | "OrgPatch 计科院说 7 天，TimePatch 考试周说 5 天——优先级裁决后老师可能不自知"（提示需说明**最终生效结果**） |
| `chain_role_undefined` | approvalChain 引用的 role 在对应解析路径上**对当前学生 / 学院无候选人**（按角色路由查表，见下） | error | "审批链引用 `class_master`，但艺术学院某些班级没人是班主任——流程会卡死" |

#### `chain_role_undefined` 按角色路由查表

不同角色的"有无候选人"判定逻辑不同，**不能统一查 sys_user_role**：

| role | 判定逻辑 |
|---|---|
| `counselor` | `counselor_org_mapping` JOIN 学生班级链——若该学生所在班级无映射记录 → 触发 |
| `class_master` | `org_unit.leader_id` 在 type=class 的节点上为 null → 触发 |
| `college_secretary` / `college_admin` / `dean` | `sys_user_role` JOIN org_unit type=college，对应学院无该角色记录 → 触发 |
| `student_affairs_director` / `_officer` | `sys_user_role` 全租户级查，无任何用户挂该角色 → 触发 |
| `school_admin` / `super_admin` | 同上 |

> 巡检时按学院 × 角色组合扫描。具体学生提交时如果该学生路径上无人，提交即拦截（兜底，不归健康度管）。

#### `conflicting_patches` 提示要求（critic 反馈改进）

提示**必须包含最终生效结果**，不只是"有冲突"：

❌ 反例："OrgPatch 计科院 vs TimePatch 考试周 在 personal.maxDays 上有冲突"
✅ 正例："考试周期间，计科院学生事假上限是 5 天（不是 7 天），因为特殊时期规则优先级最高。如这是预期，可忽略；如非预期，请编辑相应规则。"

#### 展示与交互（v0.5 简化）

> ⚠ **v0.5 删除 `silenced_until` 字段 + "我知道（静默 30 天）"按钮**——保留 3 条规则的 info 噪音可控，不需要静默机制。

- 概览页"健康度提醒"卡片，按 error / warn 两档（已无 info 级）
- 每条点开看是哪条 patch / 是什么问题（带最终生效结果）
- 提供 3 种处置：**一键禁用 / 一键删除 / 跳到编辑页**

## 4. AI 在 P0 的接入点

### 4.1 NL → TimePatch（首要切入）

**最低风险、最高感知**的切入点。

```
输入：自然语言
   "考试周 5/15-6/15 禁事假升一级审批"

处理：
   1. LLM 解析意图，输出受限 schema 的 JSON：
      { "type": "time", "name": "考试周", "scope": {...}, "diff": [...] }
   2. 后端 schema 校验：op 必须在受限集合内、path 必须存在于 base
   3. 自动 dry-run（用全部 5 个样本学生跑一遍）
   4. 返回前端：patch 草稿 + dry-run 结果 + AI 生成的"为什么"解释

输出：
   老师在预演面板看到"如果保存会发生什么"，点击确认 → 落库
```

**LLM 提示词要点**：
- 明确告知可用 op 列表
- 给出 base 当前 schema 摘要作为上下文
- 强制 JSON 输出
- 任何无法落入受限 op 的请求 → 返回"无法处理，请人工配置"，**不允许 fallback 到自由生成**

### 4.2 NL → OrgPatch（次要切入）

同上，输出 OrgPatch 草稿。

### 4.3 摘要视图渲染（极低风险）

输入摘要 JSON，按上面 handlebars 模板渲染。**LLM 仅做"自然化"**——比如把 `事假最长 14 天（演出需要）` 中的 note 整合到自然语言描述里。

### 4.4 解释"为什么这样配"

老师在概览页点击任意一条规则 → AI 拼合该规则的修改历史 + note → 给一段解释。

```
输入：
  规则：事假审批 4-7 天 = 辅导员 + 院系副书记
  历史：
    - 2026-02-10 创建 by 张老师 (note: 学期初例行)
    - 2026-03-15 修改 4-7 天加院系副书记 by 王老师 (note: 上学期院系反馈，长事假需院系把关)

输出：
  这条规则在 3 月 15 日由王老师调整，原因是上学期院系反馈"长事假需要院系把关"。
  目前 4-7 天事假需要辅导员 + 院系副书记串行审批。
```

### 4.5 健康度巡检的"AI 翻译层"

程序规则跑出 `healthIssues`，AI 把 code 翻译成自然语言（已在 3.2 模板里完成，不需要额外调用）。

## 8. 已知风险与 P1 缺口

P0 不解决但需要在文档中明示，避免后续踩坑：

### 8.0 v0.5 减法清单（推迟到 P1 的设施）

| 设施 | 原文档位置 | P0 替代方案 | P1 何时考虑引入 |
|---|---|---|---|
| `editing_session_id` + 接管机制 + 4 并发场景表 | §1.8 | 乐观锁兜底（version + 409） | 配置员 ≥ 3 人时 |
| `leave_config_template` 系统模板表 + 创建租户 hook | §1.6.1 | Java 常量 / classpath JSON 资源 + 手工触发 | 需要支持多模板（高校/中学/高职）选择 |
| `leave_request.applied_patch_ids` + GIN 索引 | §1.6 | 无（dormant_patch 规则同步删） | 需要细粒度配置追溯/审计时 |
| `parent_patch_id` FK 链 双行模型 | §1.6 | 单行可变 status，编辑期间 patch 暂停 | 用户反馈"编辑期间不能暂停"是问题时 |
| `tighten` op | §1.4 | `replace` + UI 单向校验 | AI 大量自然语言模式需要"加严"语义时 |
| `dormant_patch` / `redundant_org_patch` / `patch_path_invalid` 规则 | §3.3 | 无 | 配置规模 ≥ 10 patch 后清洁度成为问题时 |
| "回滚到上一个发布版本"按钮 | §1.8 | 老师手动编辑+发布 | 用户反馈手动回滚频率高时 |
| `silenced_until` 字段 + 静默按钮 | §3.3 | 无 | 健康度告警噪音超过老师容忍度时 |

### 8.1 双工作流定义层的发散风险

- 现有 workflow 管理 UI（如有）可以直接编辑 `workflow_definition` 表
- 本设计引入 `leave_config_base` → 翻译层运行时生成 DSL
- **必须在请销假流程定义上加只读锁**：上线后任何 workflow 管理 UI 不允许编辑请销假相关 `workflow_definition` 记录；只能通过本系统的配置管理页修改
- **实施动作**：在 `workflow_definition` 表加 `managed_by` 字段，请销假流程标记为 `managed_by = 'leave_config_v3'`，UI 据此禁用编辑入口

### 8.2 时区处理（P1 解决）

- TimePatch 的 `scope.from` / `scope.to` 是日期字符串（`"2026-05-15"`）
- P0 默认按服务器本地时区（北京时间）解释
- 学生在不同时区提交可能产生边界争议
- **P1 处置**：增加 `timezone` 字段或统一按租户配置的时区

### 8.3 学年累计上限（annualCapDays）执行点

- 设计中定义了 `annualCapDays` 字段，dry-run 也检查了
- 但实际提交时**当前 `LeaveService` 没有累计校验逻辑**
- **P0 处置**：dry-run 中作为 warn 提示；实际提交校验放 P1
- **P1 处置**：在 `LeaveService.submit()` 中增加学年累计查询 + 拦截

### 8.4 通知模板表

- §1.1.1 引用的 `templateCode` 当前对应**代码端的 i18n key**（资源文件管理）
- 现有 V006 创建的 `notification` / `notification_recipient` 是**实例化记录**，不是模板表
- P0 不引入新的模板表，模板内容由代码资源文件管理
- **P1 处置**：如需运营自助修改模板文案，再引入 `notification_template` 表

### 8.5 多租户并发发布（已通过 §1.8 处置）

- §1.8 原子发布是**单租户内**原子的，发布范围 = 该租户当前所有 `status='draft'` 的行
- 跨租户场景目前无冲突可能（每租户独立 base + 独立 patch）
- **不构成 P0 风险**

### 8.6 `parent` 虚拟角色的真实数据源

- 通知规则中的 `to: parent` 引用学生紧急联系人
- 当前 V065/V066 添加的 `emergency_contact*` 字段在 `leave_form` 上而非 `student_profile`
- **P0 处置**：确认从 `student_profile` 还是 `leave_form` 取（建议 `student_profile` 加冗余字段，由学生在首次请假时回填）
- **待研发评估具体字段路径**

### 8.7 AI NL → TimePatch 失败回退

- DeepSeek 输出无法落到受限 schema 时（如老师说"考试周遇到雨天怎么办"这类无法表达的请求）
- **P0 处置**：返回"无法处理，请人工配置"，**不允许 fallback 到自由生成**
- AI 端需明确兜底：`function calling`（结构化输出）失败 → JSON 校验失败 → 返回错误提示
- **不引入"创意性 fallback"**，宁可少做不可错做
