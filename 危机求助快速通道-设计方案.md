# 危机求助快速通道 设计方案 v1 最小版（P1 例外）

> **状态**：提案 / 已收敛为最小可救命闭环 / 待产品 + 心理中心 + 法务评审
> **日期**：2026-05-18
> **关联**：`PRD-主动关怀工作台-重写版.md`、`CLAUDE.md §6 通知铁律`、`项目开发约定.md §2 通知发送铁律`
> **定位**：涉及学生生命安全 / 法律 duty of care / 心理伦理，工程不单方面拍板。
> **修订说明（v4）**：上一版把一个小核心包进了一圈第二系统（AI 疑似层、分流、升级引擎、A/B 架构选型、18 字段表、9 决策点）。本版收敛为**最小可救命闭环**：钦定一种链路、单层判定、一次超时再通知；被砍的全部降级进 §附录 backlog（缓做，非删除）。目标：救命的那一条更早能上。

---

## 0. 一句话

学生在对话里**主动显式**说「活着没意思 / 没钱吃饭」这类求助信号时，做两件**互不依赖**的事：

1. **当场**：学生侧立即收到一张**固定支持卡**（心理中心写死的话术 + 热线/辅导员/值班联系方式）。
2. **旁路**：落一条 `crisis_signal`，由 `NotificationOrchestrator` **同时**给责任辅导员 + 危机值班发一条**不可静默**的紧急通知。

不变量：**判定确定、跑在 LLM 调用之前、不依赖 LLM 结果**——大模型挂了也照常触发。判定不由 LLM 产生。

## 1. 为什么这是 P1 例外，不是 P2

PRD 现状（已核实证据）：

| 处 | 现状 | 文件 / 行 |
|---|---|---|
| 检测 | 13 条规则无一监听 chat / 心理 / 经济信号 | `CareRuleCatalog.java` |
| 事件 | `AI_CHAT_SENSITIVE` 是死枚举，全仓库 publish = 0 | `StudentEventType.java` |
| 生成 | care_brief sanitize 强制抹除「自杀/自残/轻生/抑郁」并禁「联系家长/转介/上报」 | `sanitize.py:20-27`、`graph.py:55/58` |
| PRD | 心理/危机线索 → P2；P1 不接心理数据；不让 AI 判断心理风险 | PRD §85/§87/§101/§139/§163 |

**关键区分**：PRD 排除的是「AI **推断**学生有没有心理风险」（防误判、防贴标签、防舆情）。但**学生显式求助是学生自己说出来的事实信号，不是 AI 推断**。把「显式求助」和「AI 心理诊断」一起埋进 P2，是这版 PRD 的盲区——v1 只接前者，绝不做后者。

## 2. v1 范围（明确边界，反过度设计）

**做**（不可再砍）：

- 单层 L2：确定性词表命中**显式求助/危机**词 → 触发。无分级、无 AI 疑似层。
- 学生侧固定支持卡（确定性、非 LLM 生成）。
- 一条紧急通知，走 `NotificationOrchestrator`，不可静默。
- 首次通知**同时**扇出责任辅导员 + 危机值班（无 timer、无延时窗口、无定时 job）。

**v1 不做**（进 §附录 backlog，缓做非删除）：L1 AI 疑似层、L2-S/L2-B 收件人分流、首通知后延时/timer 升级窗口、多级升级 + `crisis_signal_escalation` 子表、`hit_count` 提级、ack 状态机/运营队列、Java 网关前置（旧方案 A）、AI 助手改危机词表、DB 故障应急外发、冻结语料回归门禁、误报率/ack 时长度量闭环。

> 砍它们不是因为没价值，是因为**它们解决的是"已经在运营一个危机队列"之后的问题**，而 v1 还没救过一个学生。先把核心跑起来。

## 3. 触发判定（就一层）

确定性词表命中「显式生命安全 / 自伤自杀 / 立即危险 / 没钱吃饭 / 断粮 / 无处可住」等显式求助词 → 触发 L2。

- **只由确定性规则或人工确认产生，LLM 不产生正式信号。**
- 词表由心理中心提供并定期复核，工程不自拟。v1 = 心理中心经受控路径维护的配置（seed / 配置表），**不接「AI 助手能改的业务配置」机制**——那是 backlog（危机词表交给 AI 改风险高）。
- 误报处理原则：**宁可误报、不可漏报**；命中即人工核实。误报多了怎么办由 D2 定，v1 不做自动分级消解。

> **决策点 D2（心理中心）**：**L2 词表初稿是 go/no-go**——没有它本功能是空管子，一行都不触发（见 §7）。**附录 B 是工程草稿起点，红笔改它即可，不必从零写。** 另含复核周期 + **误报处理标准**（尤其：人工收到通知时若无对话原文上下文，见 §5，强制全量联系？还是最小受控摘要？）。

## 4. 链路（钦定一种，不再 A/B）

```text
学生消息 → xg-ai 入口 → L2 确定性规则段（跑在 LLM 调用之前）
  ├ 命中：回调 Java 内部 crisis endpoint（Java 落 crisis_signal + 同时通知辅导员&危机值班）
  │        + 学生侧直接返回固定支持卡（不经 LLM）
  └ 未命中：照常走 LLM
```

**为什么不要旧方案 A（Java 网关前置）**：当前 web/小程序均直连 xg-ai，前置 Java 网关 = 给整个 AI 对话产品新插强制跳（SSE 中继 / 鉴权 / 全产品改道）。它唯一独有收益是「xg-ai 进程宕也能判」，但进程宕学生根本发不进消息，收益为空（见 §6）。规则段跑在 LLM 之前，对「LLM 挂」已经免疫。**A 是过度设计，本版直接不选。**

### 4.1 服务间回调铁律（v1 必须，非选配——这是正确性底线不是镀金）

- xg-ai → Java 只调内部 crisis endpoint，用 `X-Internal-Token` / 服务间签名；**不得信任前端透传的 `X-User-Id` / `X-Tenant-Id` 作安全边界**。
- Java 负责终态落库 + 通知；xg-ai 只交判定结果、消息标识、命中词表信息，不决定收件人或渠道。
- Java 校验 `student_id`/`tenant_id`/会话归属一致，**并在落库前从校验后的 `tenant_id` 显式建立 TenantContext**（调用方是服务账号、无 Sa-Token 会话，schema-per-tenant 不会自带，错了就写错 schema）。
- **`message_id` 来源**：xg-ai 对每条入站消息生成稳定 id 随回调传递，**不要求该消息被持久化**（chat 不落库时也有幂等 key，见 §5）。
- 按 `tenant_id + message_id + rule_version` 幂等：重复回调只更新命中时间 / 返回已有 `crisis_signal`。

### 4.2 `crisis_signal`（最小表）

| 字段 | 说明 |
|---|---|
| `id` / `tenant_id` / `student_id` | 基础标识 |
| `message_id` | §4.1 的稳定消息 id；本表不存原文 |
| `rule_version` | 命中时词表版本，复核 / 回溯用 |
| `status` | `pending` / `closed`（v1 只要这两态） |
| `created_at` / `first_notified_at` / `notify_status` | 触发与首次通知结果 |
| `handled_at` / `handled_by` | 人工处置闭环 |

### 4.3 通知（复用现成 + 一个 boolean）

- **收件人同时扇出**：责任辅导员 + 危机值班（D3 定义）一并收，**不设延时窗口**——生命安全信号不留死窗口，值班第一秒在场。
- 走 `NotificationOrchestrator.send(code, sourceType="crisis_signal", sourceId=id, RecipientContext, vars)`，**禁止业务侧直接调短信/企微/`NotificationService`**（通知铁律）。
- 一个模板 seed（零诊断词）：标题「有学生需要尽快人工关心」，正文「{班级} 一名同学在沟通中主动表达需要帮助，请尽快当面或电话联系核实」。**不写**「自杀/心理危机/抑郁」。是否带姓名见 D5。
- 不可静默：给 `notification_template` 加 `must_deliver BOOLEAN DEFAULT FALSE`，危机模板置 true → orchestrator 至少强制保留 `in_app`，后台不允许禁用，角色 `muted` 不能覆盖。
- **失败响亮报错**：通知返回 `null` 或收件人为空 → 写 `notify_status=failed` + 运维告警。v1 不建通用补偿扫描器；最多一次有界重试。

## 5. 留痕与隐私（最小）

- `crisis_signal` **不存学生原话**，只存 `message_id` + `rule_version`。
- **chat 消息是否落库 = 待核实前提**：本项目有「谈话/心理原文不落库」纪律，chat 消息很可能不持久化。若不落库，人工收到通知时零上下文 → 这正是 D2「误报处理标准」必须正面回答的输入，不是脚注。
- 学生侧**不显示**「被标记 / 危机信号」，只看到支持与资源。
- 数据保留期限 / 导出 / 审计查询权限由法务定（D5）。

## 6. 降级（诚实，不假装）

| 故障 | v1 行为 |
|---|---|
| **大模型 API 挂 / 超时，xg-ai 在**（真正要兜的） | 规则段在 LLM 之前，照常落 `crisis_signal` + 发通知；仅损失 LLM 生成式回复；学生侧返回固定支持卡 |
| **xg-ai 进程整体宕** | 学生无对话入口（web/小程序直连 xg-ai），本通道无从触发——属 SRE 项（高可用/探针/拉起），**须在评审显式认领风险归属**，不靠"判定下沉"假装解决 |
| 通知渠道部分不可用 | `NotificationOrchestrator` 仍写站内通知；外部投递失败进重试/失败态 |
| 模板被禁/偏好静默 | `must_deliver` 保护，禁不掉、静默不掉 |
| 收件人解析为空 / DB 写失败 | 响亮报错（运维告警 + 应用错误日志），明确人工通知未完成；不静默当成功 |

## 附录 A. 后续增强 Backlog（v1 砍掉，降优先级，非删除）

跑通 v1、确有运营需求后再评估。**不要在 v1 提前实现任何一项。**

| 项 | v1 为什么不做 | 解锁条件 |
|---|---|---|
| L2-S / L2-B 收件人分流 | v1 统一通知辅导员 + 一个危机联系人，让人工分诊；分流是在有量之后的优化 | 真实数据显示心理 vs 经济求助需走不同人 |
| L1 AI 弱信号识别 | 重新引入「AI 推断」，正是 PRD 排除项；v1 守住「只接显式求助」 | 须重过 PRD 心理边界 + 法务，单独立项 |
| 首通知后延时/timer 升级窗口 | v1 改用「首通知同时扇出辅导员+危机值班」——生命安全不留死窗口；timer 仅是告警疲劳优化（运营成熟度），且会带回定时 job + 异步租户坑 | 确证值班告警疲劳且具备分级响应能力 |
| 多级升级引擎 + `crisis_signal_escalation` 子表 | 通用升级引擎是早熟优化，且自带唯一索引坑 | 出现「需要 N 级、分时段、分对象升级」的真实流程 |
| `hit_count` 同会话提级 | v1 同会话只发一条；提级逻辑在有运营盯队列后才有意义 | 有人值守危机队列 + 有提级响应能力 |
| ack 状态机 + 运营队列视图 | v1 `pending/closed` 够用 | 危机量到需要队列化运营 |
| 误报率 / ack 时长 / 漏报复盘度量闭环 | v1 先救人；度量是 v1.1 硬化 | v1 上线后接入 |
| 冻结语料回归门禁 | 好实践但不挡第一版救命 | 词表进入频繁变更期 |
| AI 助手改危机词表 | v1 词表只走心理中心受控配置；交给 AI 改风险高 | 词表稳定 + 审批留痕成熟 + 心理中心认可 |

## 附录 B. L2 词表草稿（工程 strawman · 待心理中心定稿 · 可改可删）

> 不是定稿，是 D2 的**起点**：让心理中心在已有东西上红笔增删，不必从零写。工程负责「四类结构 + 匹配机制」，**词条内容以心理中心定稿为准**。

**① L2-S 安全危机 · 正向短语（命中即触发）**
不想活了 / 活不下去了 / 活着没意思 / 不想活下去 / 想死 / 想自杀 / 想结束这一切 / 想消失 / 一了百了 / 解脱 / 自残 / 割腕 / 撑不下去了 / 熬不下去了
立即危险（critical）：我站在天台 / 我手里有刀 / 我已经吃了药（轻生语境）

**② L2-B 基本生存求助 · 正向短语**
没钱吃饭 / 吃不上饭 / 好几天没吃饭 / 断粮了 / 交不起学费连饭都没了 / 没地方住 / 无处可去 / 被赶出来没地方住
（单纯「经济困难」不收；只收已影响基本生存的显式表达）

**③ 习语排除（命中即便含敏感字也不触发）**
难死了 / 累死了 / 烦死了 / 笑死了 / 热死了 / 困死了 / 饿死了 / 气死了 / 急死了 / 无聊死了 / 笑死 / 这题难死了 / 卷死了 / 期末要我命

**④ 人称指向排除（非本人求助）**
你去死 / 你怎么不去死 / 叫他去死 / 让他死

**匹配机制（工程负责，确定性、无 AI）**
- 先过 ③④ 排除，再过 ①②；**绝不对「死 / 药 / 跳」单字匹配**（单字必废）。
- 命中只记 `rule_version` + 命中 pattern，**不记原文**。

**两个草稿解决不了、必须 D2/D3 拍的真坑（别让工程拍）**
1. **同伴转介**「我室友说他不想活了」——本人无危机但是真线索，风险在被转述者。v1 二选一：转述也触发但标「转述」、人工去核被转述者 / 或 v1 只接第一人称、转述进 backlog。
2. **强歧义词**「吃药」「跳」——孤立词必误判，只能收共现短语（如「吃了药不想醒」）。这说明表里是**短语 / 共现，不是危险词清单**。
