> 说明：这个文件不是重写整份 PRD，而是把需要修订的段落单独整理出来，方便直接替换原文。每一段都写清楚了“改了什么”和“为什么这么改”。

## 1. 修订总览

| 位置 | 修改内容 | 为什么这么改 |
|---|---|---|
| 9.4.1 / 11.4 | 学生侧查询从“近 6 个月次数 + 闭环状态”改成“当前未关闭类型摘要” | 和 v1.2 的合规收口一致，避免把数字重新暴露给学生 |

## 2. 可直接替换的段落

### 2.1 学生侧查询口径

**替换原 9.4.1 / 11.4：**

```markdown
#### 9.4.1 学生侧“我的关怀记录”（v1.2 修订）

| 项 | 设计 |
|---|---|
| 入口 | 学生小程序“我的” → “个人信息保护” → “我的关怀记录” |
| API | `GET /api/v1/care/me/active-summary` |
| 展示 | 仅展示当前未关闭关怀的类型摘要（学业 / 行为 / 生活 / 勤工助学）；全部关闭时显示“目前无主动关心记录” |
| 不展示 | 历史总次数、闭环总数、规则名称、严重度、AI 解释、辅导员姓名、被下钻信息 |
| 数据范围 | 仅自己，后端强制 `student_id = current_user.id` 过滤 |

> 说明：学生侧保留知情权，但只给“当前状态 + 类型”，不回传可用于比较或焦虑放大的数字。
```

**为什么这么改：**

- 原文在 9.4.1 里已经决定不展示历史数字，但 11.4 还写“关怀次数 + 闭环状态”，前后打架。
- 改成 active-summary 后，接口语义更直白，也更容易和 UI 文案对齐。
- 这样能保留合规知情，同时避免把“次数”重新做成评估信号。

### 2.2 领导下钻额度

**替换原 9.5 / 11.5：**

```markdown
### 9.5 领导下钻个体（v1.2 修订：按角色分级）

| 角色 | 默认下钻额度 | 备注 |
|---|---|---|
| 辅导员 | - | 本就管自己班学生档案，不属于下钻 |
| 院系学工领导 | 20 / 天 | 覆盖值班、家长来访、月度抽查 |
| 学校学工管理者 | 50 / 天 | 跨学院协调、突发事件研判 |
| 校领导 | 不限 | 全部留痕，不限频但接受全量审计 |

### 11.5 领导下钻

| Method | Path | 说明 |
|---|---|---|
| POST | `/api/v1/care/dean/drill-down/{studentId}` | 下钻个体档案，body 必填 `reason`（≥30 字） |
| GET | `/api/v1/care/dean/drill-down/log` | 本院 / 全校下钻日志（学工部部长可看） |

后端强制：
1. 角色额度按上表校验
2. 留痕到 `task_audit`，`action = drilled_down`
3. 超额返回明确错误，不做静默降级
```

**为什么这么改：**

- 原文把额度写成统一 10 次/天，但 v1.2 已经改成角色分级，不能再退回一刀切。
- 20 / 50 / 不限更符合真实工作负载，也能减少“明面上不让看、私下截图看”的反作用。

### 2.3 通知扇出和模板

**替换原 6.5 / 11.3 / 附录 B：**

```markdown
### 6.5 通知扇出（分层）

新模块发通知必须先写 `notification_template` seed，再调用
`NotificationOrchestrator.send(code, sourceType, sourceId, recipients, vars)`。

本模块统一：
- `sourceType = care_task`
- `sourceId = task_id`

模板建议最少 4 条：
- `care_task_high_immediate`
- `care_task_daily_digest`
- `care_task_critical_dashboard`
- `care_task_urge`

禁止：
- 业务侧直接调用 `NotificationService.send`
- 业务侧自己拼 title/body 字符串发消息

### 7.4 通知策略

- 日聚合默认在 09:00 推送
- 节假日不推 medium 日聚合
- critical 只站内 + 看板，不自动外发
- 领导点击“督办”时，系统私下通知责任辅导员，但领导界面不展示姓名
```

**为什么这么改：**

- 这是项目级硬约束，必须统一走 Orchestrator，不然通知中心 UI、去重、偏好开关都接不住。
- `care_task_urge` 是“督办”这个动作缺失的模板，不补就会让场景 B 落空。

### 2.4 反馈回流

**替换原 6.2.5：**

```markdown
#### 6.2.5 反馈飞轮回流机制（v1.2 修订）

P1 不引入 ClickHouse。

回流方式改为：
- 学校侧每周批量导出脱敏反馈汇总
- 产品方先落 PostgreSQL 汇总库
- 只保留规则效果统计所需字段
- 若后续跨校分析量级明显上升，再单独评估数仓组件
```

**为什么这么改：**

- P0/P1 当前栈强调轻量优先，没必要在这个阶段引入新的基础设施。
- 先用 PostgreSQL 能把飞轮跑起来，避免把项目复杂度提前抬高。

### 2.5 审计和 brief 留存

**替换原 9.4 / 9.6 / 10.5：**

```markdown
### 9.4 审计与学生只读查询（v1.1 修订）

- 每条任务的全生命周期写入 `task_audit`，保留 5 年
- 学生侧只读查询保留“当前未关闭”的类型摘要，不返回历史总次数

### 10.5 AI brief 留存

- `task_ai_brief_history` 改为 append-only
- 保留期不再是 1 年硬删，至少与任务审计保持一致
- 任务关闭后仅进入归档态，不允许把历史 brief 直接清掉
```

**为什么这么改：**

- 原文一边说 brief 用来解决审计断链，一边又写 1 年后删除，逻辑上冲突。
- 统一留存期后，辅导员才能回看“昨天 AI 为什么这么建议”，审计链条也不断。

```markdown
### 10.1 核心表（修订建议）

- 产品方内置规则作为只读清单管理，不向学校暴露编辑器
- 学校侧仅保存：
  - `tenant_alert_rule_setting`
  - `rule_id`
  - `enabled`
  - `severity_offset`
  - `updated_at`
  - `updated_by`
- 规则版本、shadow、canary、回滚都不放进 P1 学校侧数据模型

### 13.1 P1 上线门槛（必须全过）

| 维度 | 标准 |
|---|---|
| 功能完整 | 13 个 rule_id 全部跑通；任务全生命周期闭环 |
```

**为什么这么改：**

- 原表实际列了 13 条，但验收写 12 条，数量不一致。
- P1 规则治理本来就是“产品方内置、学校只开关/偏移”，没必要把 shadow/canary/version 一起塞进学校侧模型里。

## 3. 建议保留的原文方向

- “不做评分排名、不贴标签”
- “AI 不参与判定”
- “学生侧只看当前状态，不看历史总数”
- “critical 不自动外发”
- “规则归产品方维护，学校只做启停和全局偏移”

这些方向不用改，修订稿只是把它们和 API、数据模型、验收口径对齐。
