# 项目开发约定

> 这个文件记录"做这件事**只能**这么做"的硬约束 — 跟 CLAUDE.md 第 6 节「关键设计约束」呼应,
> 区别是这里更细、按业务面分组、随项目演进追加。
>
> 加新约定的判断标准:**绕开它会留下脏数据 / 失去横向能力 / 让用户看到英文 code**。

## 1. AI 助手能改的"业务配置"清单

小夕(AI 助手)暴露给业务管理员的"对话改配置"能力。**所有 AI 改配置走同一套机制**:

```
管理员对小夕说话
  → sidecar Agent 翻译成 propose API
  → 前端弹"AI 推荐"确认 Drawer / Modal(用户能编辑、能取消)
  → 调原生 admin API 落库(走同样的权限码、审计)
```

**绝不允许** sidecar 直接写库或绕过原生 API。理由:
- 原生 API 的权限校验(`@SaCheckPermission`)是唯一安全边界
- 操作日志、ai_message 字段、回滚靠原生 API 路径自然实现
- AI 失败时 fallback 到手动 UI 操作语义一致

### 已接入(P0 可工作)

| 配置面 | 触发方式 | propose 端点(sidecar) | 落库 admin API | 前端确认 UI |
|--------|---------|----------------------|---------------|------------|
| 请假规则 | 小夕对话 / 配置页内"AI 改" | `/ai/api/v1/workflow/propose` | `/api/v1/workflow/definitions/{id}` | `WorkflowAuthorDrawer` |
| 勤工助学业务配置 | 同上 + 深链(?workflow=&prompt=) | 同上 | 同上 | `BusinessConfigTab` → `WorkflowAuthorDrawer` |
| 通知规则(双轨 / 渠道) | 通知中心内"AI 改" | `/ai/api/v1/notification/propose` | `/api/v1/notifications/rules/{id}` | `NotificationCenterPanel` |
| 自定义角色(权限码组合) | 角色权限页"新建自定义角色" → AI 推荐 | `/ai/api/v1/role-config/propose` | `/api/v1/system/roles` | `CreateRoleModal` |

### 计划接入(P1)

- 团队管理(评审委员会等业务编组)
- 字段字典 / 表单 Schema(field_catalog)
- 学年配置(work-study year_setting)
- 工资 / 薪资规则
- 数据导入字段映射

### 加新业务到这套机制的 4 步

1. **后端 sidecar**(`xg-ai/app/api/`):加一个 `xxx_config.py`,实现 `propose` 端点。复用 `chat.py` 里 Agent 的 prompt 模板风格。
2. **前端 AI Panel**(`AIPanel.tsx`):`navigate` tool schema 加该业务的 deep-link 参数(`workflow=...&prompt=...`)。
3. **业务配置页**:`useEffect` 监听 URL 参数,自动打开 author Drawer + 自动提交首句指令(一次性 ref 防重提)。
4. **chat.py system prompt**:补充该业务的"用户怎么说话 → 用什么 deep-link"对照例子。

## 2. 通知发送铁律(扩展 CLAUDE.md 第 6 节)

新模块发通知 = 严格 2 步,**绝不允许业务侧绕开**:

1. **INSERT 一行 notification_template seed**(Flyway migration 里):code / title_template / body_template / 关联的 sourceType
2. **调 `NotificationOrchestrator.send(code, sourceType, sourceId, recipients, vars)`**

**禁止**:
- ❌ 业务侧直接调 `NotificationService.send` / `SmsClient.send` / `WechatClient.push`
- ❌ 业务侧拼 title/body 字符串发出去(template 必须落库,才能被通知中心 UI 编辑)

**这套强制的横向能力**:
- 通知中心 UI 列表自动出现该模板(管理员可禁用 / 改文案 / 改触发条件)
- 角色偏好生效(用户能在个人设置关该类通知,业务零代码改动)
- 双轨去重(同 source_id 同 user_id 同 channel 一小时内只 1 条)
- 三渠道扇出(站内 / 微信 / 短信)按用户偏好 + 规则自动分流
- 小夕能识别这类通知(统一 code → 模板 → 可解释)

## 3. 角色 / 团队 / 权限模型

### 数据层
`sys_role.kind` 字段区分两种语义,**前端 UI 也分两个 tab**:

| kind | 语义 | UI 入口 | 关注点 |
|------|------|--------|--------|
| `'role'` | 长期岗位定义 | 系统管理 → 角色权限 | 权限码矩阵 |
| `'team'` | 业务编组(临时 / 跨部门) | 系统管理 → 团队管理 | 成员 + 时间窗 |

底层都是 `sys_role` 一行,挂在 `sys_user_role` 表上等价。从"给用户挂角色码"角度,内置角色 / 自定义角色 / 团队**完全等价** — 所以用户管理下拉**三类都列**(团队加前缀「团队 · 」视觉区分)。

### 权限码新增
不在 UI 里加 — 只能改 `RolePermissionDefaults.java` 重新发版。UI 误操作让全平台学生失去 `leave:submit` 比"加错权限"危险大得多。

### 虚拟角色(employer_leader / position_owner)
**不进 `sys_role`** — 通过业务表 FK 解析(`employer.leader_user_id` 等)。工作流节点写
`assignee: { role: employer_leader, ref: 'employer.{employer_id}' }`,引擎在派任务时反查业务表。

理由:每个用人单位都建一行 sys_role 会导致数据爆炸 + 角色权限 UI 无法管理。

## 4. 初始密码与登录

### P0 现状
所有教职工初始密码统一 **`xg@123456`**,单一来源:`SystemUserService.DEFAULT_PASSWORD`。

3 个创建路径全部走默认:
- 管理员手建用户(用户管理 / 新建)— 表单不暴露密码字段
- 教师批量导入(数据导入)— Executor 写 sys_user 时 setPasswordHash
- 重置密码(用户列表)— 同样默认值

### P1 接入项
- **首登强制改密**:加 `sys_user.must_change_password` 字段(默认 true)+ 登录后拦截器 + 改密页
- **SSO 单点登录**:CAS / OAuth2 / 企业微信扫码,优先级看学校已有 IDP

P1 上线前不要硬编码"用户首次必须用 xg@123456 登录"提示给学生 — P0 只面向内部教职工,学生走 SSO 后这条不存在。

## 5. 数据导入 vs UI 创建的边界

| 实体 | 创建路径 | 理由 |
|------|---------|------|
| 学生 | 数据导入 | 数量大、字段多、教务系统对接 |
| 班级 | 数据导入 | 同上 + 关系复杂(parent_id / closure) |
| 教师 / 辅导员 / 行政 | 数据导入 | 批量;UI 个别补救可用 |
| **学院** | UI 新建(组织派班 → [+ 新增院系]) | 数量少(< 30),UI 创建合理;支持随时增加新院 |
| 用人单位 | UI 新建(勤工助学 → 用人单位) | 数量中等,业务管理员频繁建 |
| 自定义角色 | UI 新建(角色权限) | 同上 |
| 团队 | UI 新建(团队管理) | 高频(每学期评审 / 临时编组) |

**不要在 UI 上加"批量导入学生"以外的逐条创建入口** — 否则会绕过教务对接,造出脏数据。

## 6. 错误提示统一

所有 mutation `onError` **必须用 `describeApiError(e, fallback)`** 透传后端 `BizException.message`,
不要自己拼"创建失败,请重试"覆盖掉具体原因。

```ts
onError: (e) => message.error(describeApiError(e, '创建失败')),  // ✅
onError: (e) => message.error('创建失败,请重试'),               // ❌ 用户不知道为啥失败
```

后端校验失败时(name 重复、角色被绑、parent 不存在等)用 `throw new BizException(GlobalErrorCode.BAD_REQUEST.getCode(), "具体原因")`,前端会原样展示给老师看。

## 8. 工作流引擎约束

- DSL = YAML/JSON,5 种节点类型:`form_submit / approval / notification / condition / finish`
- 表达式受限语法,`computed` 只允许内置函数(`duration_days` / `date_diff` / `if_then` / `coalesce` / `compare`)
- **不暴露通用脚本引擎**(JS / Groovy / SpEL 都不行)
- assignee 解析:
  - `role` → `sys_user_role` 联表(配合 `scope: global / same_class / same_college`)
  - 虚拟角色 → 业务表 FK 解析(`ref: 'employer.{employer_id}'`)
- 工作流终态:`WorkflowFinishedEvent` 广播,业务侧 `@EventListener` 回写业务表(如 `leave_request.status`)

## 9. RBAC / Workflow 补充约定

来源 `RBAC改造落地方案-xg1.md` §12,经 Sprint 1-4 落地后定为硬约束:

1. 租户上下文必须来自登录 session;`X-Tenant-Id` 只能与 session tenant 一致,不能单独决定 schema。
2. `kind='team'` 是业务编组,不授予功能权限;功能权限只来自 `kind='role'`。
3. workflow DSL 的 `assignee.role + scope` 必须通过 `AssigneeCatalog` 校验后才能发布。
4. 虚拟角色不进 `sys_role`,但必须在 `AssigneeCatalog` 登记 label、scope、bizType 和解析策略。
5. 审批节点解析不到受理人必须失败或告警,不允许静默产生无任务的 running workflow。
6. 外部系统角色不直接映射为 xg1 权限;先同步身份、组织和师生关系,权限由 xg1 本地角色控制(接入边界见 `金智对接边界-RBAC.md`)。
